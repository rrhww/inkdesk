# Inkdesk Chat-First Research Shell Design

## 摘要

本次改造把 Inkdesk 的主前端从“研究后台”收束为一个更直接的 **对话型研究工作台**。`/app` 不再是 dashboard 聚合页，而是直接成为 `问答` 主入口；用户进入系统后，第一感知应该是“我可以立刻提问、继续追问、查看证据、发现知识缺口，并决定是否把本轮结果沉淀到知识库”。

这次不是单纯给 Ask 页面换皮，而是重写主路径的信息架构与交互重心：左侧改成历史优先，中间是连续对话流，顶部是中文直白页签，右侧是当前研究上下文与动作。整体气质参考 ChatGPT、Gemini、豆包、DeepSeek 这类成熟对话产品，但必须保留 Inkdesk 的 `raw / ingest / wiki / ask` 研究工作流特征。

## 已确认决策

- `/app` 直接变成 `问答` 首页，不再先落 dashboard
- 用户可见命名全部改成中文，但本轮 **不修改 URL**
- 导航语义采用对象式命名，而不是动词式命名
- 当前主路径命名固定为：
  - `问答`
  - `资料`
  - `审阅`
  - `知识库`
- 侧栏优先级采用“历史优先”，而不是传统后台导航优先
- 顶部保留显眼的中文页签，用于在 `问答 / 资料 / 审阅 / 知识库` 之间切换
- `问答` 页面采用“对话主入口 + 研究侧栏”结构
- 本轮只重构主路径，不做 URL 迁移，不改后端 API 契约，不恢复旧的 plans / publish / public 流程

## 目标

- 让用户打开 `/app` 后立刻理解产品用途，而不是先学习信息架构
- 让 `问答` 成为产品第一感知入口，并且具备成熟聊天产品的使用习惯
- 让 `资料 / 审阅 / 知识库` 继续服务 vault-first 研究工作流，而不是被聊天壳吞掉
- 让页面文案、命名和布局都显著降低理解门槛
- 为后续 agent 层增强保留稳定的前端壳层与组件边界

## 非目标

- 不在本轮更换前端框架，继续使用现有 Next.js + React + Tailwind
- 不在本轮引入第二套 UI 系统或复杂设计 token 平台
- 不在本轮改写现有后端接口协议
- 不在本轮实现多会话树、团队协作、公开阅读面或 plugin 市场
- 不把 `资料 / 审阅 / 知识库` 做成与 `问答` 等量的复杂新产品线

## Product Narrative

新的主叙事固定为：

`围绕私人研究记忆提问 -> 基于 wiki/raw 得到可追溯回答 -> 识别缺口 -> 决定是否沉淀到知识库`

用户不应该先看到“工作台概览”“模块卡片”或“管理指标”，而应该先进入一个明确的研究对话环境。系统的“Agent 感”不来自夸张的品牌包装，而来自三个事实：

- 用户可以从一个问题直接启动研究
- 回答始终带来源、缺口和后续动作
- 问答结果可以继续流入 `审阅 -> 知识库`

## Information Architecture

### 主导航

主导航固定为：

- `问答`
- `资料`
- `审阅`
- `知识库`

### 信息架构职责

- `问答`
  - 主入口
  - 承载提问、追问、引用来源、知识缺口、联网补料与“沉淀到知识库”动作
- `资料`
  - 承载原始材料导入、浏览与来源追踪
  - 让用户理解“原始材料在这里，尚未变成知识页”
- `审阅`
  - 承载 ingest proposal 的 accept / reject 流程
  - 让用户理解“AI 只提出变更，不会静默写 wiki”
- `知识库`
  - 承载已经沉淀的 wiki 页面
  - 让用户理解“这里是已确认的研究记忆”

### 兼容性

- 兼容期内，旧的 `Raw / Ingest / Wiki / Ask` 英文路由继续保留
- 页面内的任何一级标题、导航标签、操作文案都切换为中文
- 旧兼容路由如果还存在，只承担跳转或别名职责，不再作为新的产品命名依据

## Shell Layout

整体采用三段式壳层：

### 左侧栏

左侧改成历史优先的对话侧栏，而不是传统模块导航栏。它包含：

- 品牌区
- 新建对话入口
- 最近问答历史列表
- 可折叠的次级入口或快捷过滤

用户进入系统时，最先看到的应该是“最近都问了什么”，而不是“系统里有哪些模块”。

### 顶部页签

顶部提供显眼的中文页签：

- `问答`
- `资料`
- `审阅`
- `知识库`

这些页签承担产品理解职责，所以需要比当前后台式 header 更醒目、更直白。它们不是辅助 breadcrumb，而是主要导航。

### 主画布

中间区域按页面承担主任务：

- `问答` 页显示完整对话流与输入区
- `资料` 页显示来源导入与 raw 列表
- `审阅` 页显示待确认提案
- `知识库` 页显示 wiki 页面列表和详情入口

### 右侧上下文面板

右侧统一保留研究侧栏，用于显示当前页相关上下文。对 `问答` 页尤其重要，至少包含：

- 本轮来源
- 知识缺口
- 当前 topic / wiki 上下文
- `沉淀到知识库`
- 待审阅项摘要

右侧栏的作用不是塞信息，而是把“研究”和“知识沉淀”这两件事可视化。

## Ask Page Design

`问答` 页是本次改造的核心，必须从“左右两张卡片的信息页”升级为真正的聊天式研究界面。

### 核心结构

- 顶部保留轻量欢迎和能力提示，但不做大段 marketing
- 中间是连续消息流，支持：
  - 用户问题
  - 系统回答
  - follow-up 推荐
  - 引用来源
  - 知识缺口
  - 外部补料提示
- 底部是固定输入区，具备：
  - 明显的主输入框
  - topic 选择
  - `仅基于知识库` / `显式联网补料` 模式切换
  - 提交动作

### 问答交互模型

- 用户应当从输入框发起问答，而不是先阅读一页说明卡片
- 推荐问题作为“快速开始”存在，但优先级低于输入区
- 继续追问必须看起来像同一条对话，而不是刷新成一页新的结果
- 当证据不足时，系统优先展示 `知识缺口`，而不是用高置信语气硬答
- “沉淀到知识库”必须在问答结果附近显眼出现，但语义明确为“生成提案”，不是“直接写入”

### 问答页右侧面板

问答页右侧栏固定展示：

- 本轮使用到的 wiki / raw / web 来源
- 本轮尚未覆盖的知识缺口
- 是否触发了联网补料
- 当前可执行动作：
  - `沉淀到知识库`
  - `查看待审阅提案`
  - `切换研究范围`

## Other Page Direction

### `资料`

- 首页目标是“导入和管理原始材料”，不是表格式后台
- 第一屏就要说明：这里保存网页、PDF、旧笔记等原始材料
- 导入入口应明显大于筛选器和列表控制项

### `审阅`

- 首页目标是“处理 AI 提案”，不是泛管理页
- 第一屏应突出：待处理数量、最近提案、accept / reject 动作
- 每条提案要让用户一眼理解它影响的是哪一个知识主题

### `知识库`

- 首页目标是“浏览已确认知识页”
- 第一屏应突出主题页本身，而不是表格字段
- 需要强化“这是已沉淀内容”的感知，和 `资料` 区形成鲜明区分

## Visual Direction

视觉方向采用“高可信、轻编辑器感、弱后台感”的研究产品语言。

### 风格原则

- 避免后台式 dense cards、指标条和表格优先布局
- 避免英文缩写在首屏主叙事里喧宾夺主
- 保留研究产品应有的秩序感，但让对话成为视觉主轴
- 让“来源”“缺口”“沉淀”这些动作通过色彩层次和空间关系被看懂

### 字体与界面表面

- 继续使用现有字体体系的基础上，强化对话区与知识区的层次差异
- 中间消息流应拥有更宽松的留白和更稳定的阅读节奏
- 右侧研究栏使用轻量分组面板，不重复主对话内容
- 顶部页签与左侧历史需要更清楚的当前态，而不是仅靠细微背景变化

### 动效

- 页面首屏进入时使用轻量分层出现
- 对话消息新增时使用克制的渐显或位移动画
- 不做炫技型动效，不模仿社交产品

## Implementation Boundary

### 推荐基础

- `assistant-ui`
  - 作为 `问答` 页交互壳层的直接参考和可选底座
  - 主要吸收消息列表、输入区、流式态、历史感和聊天骨架
  - 不接管我们现有 API、鉴权或 vault 业务协议

### 仅作参考

- `lobe-chat`
  - 借鉴布局节奏与成熟聊天产品的壳层组织
  - 不引入其 provider、插件、市场和平台能力
- `ai-website-cloner-template`
  - 借鉴“先抽结构与视觉 token，再还原”的页面重构方法
  - 不做表面克隆
- `Onlook`
  - 作为后续 Next.js + Tailwind 视觉联调工具保留
  - 不作为产品运行时依赖
- `OpenGenerativeUI`
  - 只作为后期 generative UI 参考
  - 不进入本轮主实现范围

### 本地组件策略

优先在现有前端内重建以下边界：

- `ChatShell`
- `ConversationHistoryRail`
- `TopSectionTabs`
- `AskComposer`
- `AskMessageList`
- `ResearchContextPanel`
- `RawWorkspace`
- `ReviewWorkspace`
- `WikiWorkspace`

目标是先把壳层和页面职责拉直，再决定是否局部吸收外部组件模式。

## 错误处理与空状态

- `/app` 初次进入无历史时，应展示可直接发问的空白对话态，而不是 dashboard 空壳
- `问答` 无答案时，不显示大量解释性卡片；只保留轻量引导与建议问题
- `资料` 空状态要强调“先导入原始材料”
- `审阅` 空状态要强调“当前没有待确认提案”
- `知识库` 空状态要强调“还没有确认沉淀的知识页”
- 如果 Ask 返回错误，错误反馈应就地出现在对话流或输入区附近，而不是跳转到后台式错误页

## Testing And Acceptance

### 验收标准

- 打开 `/app` 时，首屏感知应是对话型研究入口，而不是卡片式后台
- 用户 5 秒内能看懂：
  - 这里可以提问
  - 回答会带来源
  - 证据不足时会暴露缺口
  - 可以决定是否沉淀到知识库
- 所有一级可见命名都切换为中文：
  - `问答`
  - `资料`
  - `审阅`
  - `知识库`
- `问答` 页必须呈现连续对话，而不是单次问答结果卡
- 右侧研究栏必须强化来源、缺口和沉淀动作
- 不在主路径中重新出现 plans / publish / settings / tags 等旧产品残留

### 验证 Areas

- 页面结构：
  - `/app` 是否直接进入 `问答`
  - 左侧是否为历史优先
  - 顶部是否为中文页签
- 文案：
  - 用户可见命名是否全部切成中文
  - 页面作用是否一眼可懂
- 交互：
  - `问答` 是否支持发问、追问、查看引用、查看知识缺口
  - `沉淀到知识库` 是否语义清晰
- 兼容：
  - 旧路由是否仍能进入兼容页面
  - 但用户不会在主路径中被旧命名带偏
