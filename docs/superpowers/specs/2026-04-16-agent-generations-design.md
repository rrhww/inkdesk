# Inkdesk Agent 三代演进设计

## 1. 目标

为 Inkdesk 的 Agent 层固定一条清晰的三代演进路线，回答三个问题：

- 第一代、第二代、第三代 Agent 分别做什么
- 每一代在主系统里以什么形态出现
- 每一代依赖什么技术形态，以及明确不做什么

这份设计默认服务当前 Inkdesk 的主系统定位：

- Agent 不是独立产品
- Agent 的主交互入口是 `/app`
- Agent 必须服务笔记、计划、检索和发布，而不是脱离主系统单独生长

## 2. 总定义

Inkdesk 的 Agent 演进路线，不是把聊天框越做越强，而是让主系统中的 Agent 从：

1. `判断中枢`
2. `受控执行层`
3. `有限自治协作者`

逐步成长。

三代划分采用双层标准：

- 表层按 `能力成熟度` 划分：判断、执行、自治
- 底层按 `技术架构升级` 划分：聚合与建议、工具调用、自动化与调度

这意味着三代 Agent 的升级不是“多接几个模型”这么简单，而是主系统角色的升级。

## 3. 第一代 Agent：判断中枢

### 3.1 定位

第一代 Agent 的目标，是让 `/app` 真正成为主人进入系统后的第一屏，而不是一个普通导航页。

它首先解决的问题不是自动执行，而是：

- 今天该推进什么
- 当前最重要的阻塞是什么
- 哪些历史知识应该重新回到眼前
- 下一步动作应该怎么拆

### 3.2 用户可见能力

- 首页上下文摘要
  - 汇总最近笔记、焦点计划、待发布内容和最近检索线索
- 阻塞识别
  - 指出当前最卡住的项目、原因和建议动作
- 下一步建议
  - 将模糊工作压缩成 1 到 3 个明确动作
- 知识召回
  - 在计划、笔记或首页场景中带回相关历史内容
- 主题归纳
  - 从多篇笔记里提炼当前正在反复出现的主题

### 3.3 主页面形态

- `/app` 仍然是 Agent 控制台
- 页面以工作台卡片为主，而不是完整对话线程
- Agent 输出以四类模块组织：
  - 今日判断
  - 焦点阻塞
  - 相关知识
  - 推荐动作
- 可以保留简短输入框，但它是“补充上下文或追问”的入口，不是主产品形态

### 3.4 底层架构

- 数据来源以现有主系统读模型为主：
  - 笔记
  - 计划
  - 检索结果
  - 发布状态
- 后端先通过聚合接口组织 Agent 首页上下文
- Agent 建议可以由规则模板、提示词模板和有限 LLM 总结共同生成
- 本阶段不要求完整多轮 Agent 生命周期

### 3.5 边界与非目标

- 不直接修改系统状态
- 不自动创建计划、更新笔记或发布内容
- 不把 `/app` 做成一个通用聊天产品
- 不引入后台持续运行的 Agent

### 3.6 成功标志

- 主人进入 `/app` 后，首先获得的是“今天该怎么推进”的判断感
- Agent 首页不再只是模块入口聚合，而是形成真实上下文压缩
- 笔记、计划、检索开始围绕 Agent 协同

## 4. 第二代 Agent：受控执行层

### 4.1 定位

第二代 Agent 的目标，是让 Agent 从“只会判断”升级为“能帮你代办一部分工作”，但所有关键写操作仍然由你确认。

这一代的核心不是完全自动化，而是：

- 先生成可执行草稿
- 再由主人确认
- 最后由系统落库执行

### 4.2 用户可见能力

- 计划草拟
  - 根据一篇笔记、一个主题或当前上下文生成计划草稿
- 笔记整理
  - 生成标题候选、摘要、标签建议、结构重组建议
- 发布转译
  - 将内部知识压缩成公开标题、摘要、slug 建议和发布说明
- 跨模块串联
  - 从一条建议直接生成“候选动作包”，例如：
    - 创建计划草稿
    - 关联笔记
    - 准备发布草稿
- 会话工作记录
  - 将一次 Agent 交互沉淀为可回看的工作记录，而不是一次性聊天

### 4.3 主页面形态

- `/app` 仍然是总入口
- 页面开始出现“Agent 行动提案”区，而不只是建议卡片
- 建议项升级为带确认按钮的候选动作：
  - 接受并创建计划
  - 接受并更新笔记草稿
  - 接受并生成发布草稿
- 可以引入独立的 Agent 工作面板，但它从属于主系统中枢，不与 `/app` 平级争夺入口

### 4.4 底层架构

- 在第一代读聚合基础上，增加受控工具调用能力
- Agent 输出不再只是文本建议，而是结构化动作提案
- 系统需要明确的工具边界，例如：
  - 创建计划草稿
  - 更新笔记元信息草稿
  - 生成发布草稿
- 所有写操作必须经过确认门
- 需要保留最小执行记录，能够回答：
  - Agent 建议了什么
  - 你确认了什么
  - 系统实际执行了什么

### 4.5 边界与非目标

- 不允许无确认地执行高风险写操作
- 不允许直接重写正文事实来源而不保留审阅态
- 不实现完整自治调度系统
- 不把 Agent 权限扩成“能改一切”的后台管理员

### 4.6 成功标志

- Agent 不再只是解释系统，而是能明显减少手工整理与录入成本
- 主人愿意使用 Agent 来生成计划草稿、整理候选结构和准备发布草稿
- Agent 的工具执行仍然保持可信和可控

## 5. 第三代 Agent：有限自治协同

### 5.1 定位

第三代 Agent 的目标，是让 Agent 在你不持续盯着它的时候，仍能帮助 Inkdesk 维持秩序和节奏。

但这一代的上限不是“强自治”，而是 `有限自治`：

- 低风险动作可自动执行
- 中高风险动作进入确认队列
- 所有后台行为都必须可审计、可追溯、可回退

### 5.2 用户可见能力

- 定时回顾
  - 自动生成每日 / 每周回顾草稿
- 计划巡检
  - 检测长期未推进的计划并生成提醒或候选重排建议
- 知识维护
  - 自动识别值得整理的笔记、待补标签内容、待发布候选
- 发布节奏辅助
  - 自动收集近期成熟内容并形成发布候选清单
- 低风险自动动作
  - 例如更新回顾草稿、生成提醒、整理候选队列
- 高风险确认队列
  - 例如大规模计划重排、正文覆盖、公开发布

### 5.3 主页面形态

- `/app` 依然是主入口
- Agent 控制台新增“后台协作状态”视图：
  - 今日自动回顾
  - 待确认动作
  - 自动执行记录
  - 异常与失败任务
- 主系统里需要一个 Agent 队列 / 收件箱区域，用来承接“等待你确认”的内容
- 仍然不把 Agent 做成脱离主系统的自治黑箱

### 5.4 底层架构

- 具备基础调度与事件触发能力
- 能区分动作风险等级：
  - 低风险可自动执行
  - 中高风险需确认
- 需要稳定的执行日志与审计记录
- 需要最小策略层，决定：
  - 什么时间触发
  - 什么动作可自动做
  - 什么动作必须等你批准
- 与后续自动化 / 工作流能力对接，但不等于完整通用编排平台

### 5.5 边界与非目标

- 不做完全自主 Agent 执行系统
- 不允许无痕修改主系统关键事实
- 不允许自动公开发布内容
- 不把第三代定义成“替代主人决策”

### 5.6 成功标志

- Agent 已经开始帮助系统持续运转，而不只在你打开页面时才工作
- 自动动作让系统更稳，而不是更乱
- 主人对 Agent 的默认感受是“可信的协作者”，而不是“不可控的黑箱”

## 6. 三代与现有路线图的映射

- `第一代 Agent`
  - 对应当前路线中的 Agent 协同增强阶段
  - 重点是上下文汇总、知识回顾、下一步建议
- `第二代 Agent`
  - 对应 Agent 深度参与计划推进和发布准备的阶段
  - 重点是受控工具执行
- `第三代 Agent`
  - 对应自动化与工作流阶段
  - 重点是有限自治与后台协同

这条路线与当前产品边界保持一致：

- 先把主系统做顺
- 再增强执行层
- 最后进入有限自动化

## 7. 统一原则

- 主交互入口始终是 `/app`
- Agent 永远服务主系统，而不是反过来让主系统服务 Agent
- 笔记始终是事实来源
- 计划始终是推进抓手
- 发布始终是次级模块，不重新变成产品中心
- 所有自动增强能力都必须保留确认、记录和审计能力

## 8. 当前建议的开发顺序

如果按实际落地顺序推进，建议先做：

1. 第一代 Agent 的首页上下文压缩和建议框架
2. 第二代 Agent 的结构化动作提案和确认执行
3. 第三代 Agent 的调度、队列和有限自动执行

当前最适合优先落地的，不是自治，而是第一代：

- Agent 首页真实聚合
- 上下文压缩
- 阻塞识别
- 下一步建议
- 相关知识召回

只有这一层成立，后面的执行与自治才不会失去基础。
