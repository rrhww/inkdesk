# 知识库初始化设计

> 对应文章《AI研发自动化：Wiki知识库+技能包》4.1.1 最初建设。本文是 Inkdesk 开发计划的第一步——定义一套 vault 的"诞生规范"：目录结构、页面模板、Schema 文件。

## 文章做法

mkt-link 团队的初始步骤：

1. 用 `/superpowers-brainstorming` 设计 KB 结构
2. 确定三层架构：Raw sources（只读）→ Wiki（LLM 维护）→ Schema（CLAUDE.md）
3. 设计目录结构（8 个子目录，各有职责）
4. 定义页面格式规范（frontmatter + 每类页面的章节模板）
5. 定义核心工作流（ingest / code-kb / query / lint）
6. 按 5 个 Phase、16 个 Task 分批实施

核心原则是 **Schema 先于内容**——CLAUDE.md 在第一批文档摄入之前就写好了。

## Inkdesk 的 vault 结构设计

Inkdesk 需要支持两类场景：
- **代码项目型**（类似 mkt-link）：有代码仓库 + 文档
- **通用知识型**（个人学习、读书笔记、研究专题）：以文档和网页为主

两类场景共享同一套骨架，按需裁剪。

### 完整结构（代码项目型）

```text
vault/
  raw/                    原始材料（只读，不可修改）
  wiki/                   LLM 生成和维护的知识页
    entities/             实体页（领域对象、数据模型、接口定义）
    concepts/             概念页（核心概念、业务流程、架构思想）
    index.md              内容目录
    log.md                操作日志
  schema/                 面向 Agent 的维护规则
  skills/                 可复用工作流
  evals/                  golden tasks + rubrics
  runs/                   Agent 运行记录
```

### 简化结构（通用知识型）

```text
vault/
  raw/
  wiki/
    topics/               主题页
    sources/              来源摘要
    queries/              问答存档
    index.md
    log.md
  schema/
  skills/
  evals/
  runs/
```

### 与文章 mkt-link-kb 结构的对比

| mkt-link-kb | Inkdesk | 说明 |
|-------------|---------|------|
| `kb-source/`（外部） | `raw/` | 同样只读 |
| `mkt-link/`（外部代码仓） | 通过 KB-META 关联 | Inkdesk 不把代码仓放在 vault 内 |
| `entities/` | `wiki/entities/` | 归入 wiki 子目录 |
| `concepts/` | `wiki/concepts/` | 同上 |
| `sources/` | `wiki/sources/` | 同上 |
| `code/modules/`、`code/classes/` | `wiki/entities/` | Inkdesk 不区分 code 和 doc 实体 |
| `queries/` | `wiki/queries/` | 同上 |
| `CLAUDE.md` | `schema/` 下的多个文件 | Inkdesk 拆分为多个专题 schema |
| 无 | `skills/`、`evals/`、`runs/` | Inkdesk 的阶段二、三、五的预留 |

## 页面模板

文章定义了 entity / concept / source / module / class / query 六类页面，每类有 frontmatter 规范 + 章节模板。Inkdesk 需要通用化这些模板。

### Frontmatter 规范

```yaml
---
type: concept | entity | source | query
created: 2026-06-05
updated: 2026-06-05
tags: []
related: []
source_file: ""
status: stub | draft | stable | outdated
---
```

比文章多了一个 `status` 字段——文章在评测阶段才发现需要追踪页面成熟度，Inkdesk 从初始化就内置。

### 概念页模板

```markdown
---
type: concept
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: []
related: []
status: stub
---
# 概念名称

## 概述
（概念的核心定义，2-3 句话）

## 关键要点
- [[要点1]]：描述
- [[要点2]]：描述

## 相关实体
- [[实体名]] — 关系说明

## 相关来源
- [[来源摘要名]]

## 待澄清
- 尚不明确或需要进一步研究的问题
```

## Schema 文件设计

文章把 Schema 写在一个 CLAUDE.md 里。Inkdesk 拆成多个专题文件，每个文件管一件事。Agent 按需读取，不需要一次性加载全部。

### schema/vault-layout.md

告诉 Agent 目录怎么组织、文件放哪里：

```markdown
# Vault 目录约定

## 目录职责
- `raw/`：原始材料。Agent 只读，不可修改。
- `wiki/`：已接受的长期知识页。Agent 可以提案修改，但必须经 owner 审阅。
- `schema/`：本目录。Agent 的维护规则。
- `skills/`：可复用工作流定义。
- `evals/`：评测任务和评分标准。
- `runs/`：Agent 运行记录。

## 文件命名
- 知识页：`{主题关键词}.md`，使用连字符分隔
- 来源摘要：`{来源标题}.md`
- 问答存档：`YYYY-MM-DD-{问题关键词}.md`

## 新增页面的规则
- 先检查是否已有同主题页面（搜索 wiki/ 目录）
- 新页面必须至少有一个 inbound 链接
- 创建后更新 index.md
```

### schema/wiki-page-template.md

告诉 Agent 每种页面类型应包含哪些章节：

```markdown
# Wiki 页面模板

## 所有页面必须
- 包含完整的 frontmatter（type / created / updated / tags / related / source_file / status）
- 至少有一个 inbound 链接（从其他页面指向它）
- 引用来源时使用 `[[来源页名]]` 格式

## 概念页（type: concept）
（章节模板引用 templates/concept-page.md）

## 实体页（type: entity）
（章节模板引用 templates/entity-page.md）

## 来源摘要页（type: source）
（章节模板引用 templates/source-summary.md）

## 状态流转
stub → draft → stable → outdated
- stub：刚创建，内容不完整
- draft：主要内容已填充，待补充细节
- stable：内容完整，来源清晰
- outdated：内容可能过时，待更新
```

### schema/source-citation-rules.md

告诉 Agent 如何引用来源：

```markdown
# 来源引用规则

## 强制要求
- 每个知识页的 claims 必须能追溯到 raw 来源或已接受的 wiki 页面
- 来源引用格式：`[[来源页名]]` 或 `[标题](raw/路径)`
- 从外部网页或 PDF 提取的信息，必须先创建来源摘要页再引用

## 禁止行为
- 不得编造不存在的来源
- 不得引用已被拒绝的提案
- 不得引用不存在或已删除的 raw 文件
```

### schema/ingest-proposal-rules.md

告诉 Agent 如何创建摄入提案：

```markdown
# 摄入提案规则

## 提案类型
- new-topic：创建新知识页
- patch-topic：修改已有知识页
- open-question：标记待澄清问题
- conflict-resolution：解决知识冲突

## 提案必须包含
- 提案类型
- 目标页面路径
- 变更说明（新增/修改/删除的内容）
- 来源依据
- 影响的关联页面列表

## 审阅规则
- 提案未经 owner 接受前不得写入 wiki/
- 被拒绝的提案记录原因后归档
- 冲突提案需列出冲突双方并建议裁决方向
```

### schema/ask-answer-rules.md

告诉 Agent 如何回答查询以及何时建议沉淀：

```markdown
# 查询回答规则

## 查询优先级
1. 先搜索 wiki/ 中的已接受知识
2. 再搜索 raw/ 中的原始材料
3. 最后使用模型自身知识（需标注"模型知识，未经本知识库验证"）

## 回答格式
- 引用 wiki 页面时使用 `[[页面名]]` 格式
- 引用 raw 来源时附路径
- 不确定的内容标注"待验证"

## 沉淀建议
- 回答涉及新发现且不在 wiki 中 → 建议创建新页面
- 回答补充或修正了已有 wiki 内容 → 建议 patch
- 回答发现了知识矛盾 → 建议创建冲突裁决提案
```

### schema/wiki-health-rules.md

告诉 Agent 如何自检知识库质量：

```markdown
# Wiki 健康检查规则

## 结构检查
- 缺 frontmatter：页面无 type / created / tags 字段
- 缺来源引用：stable 状态页面无 source_file 或内联引用
- 断链：`[[xxx]]` 指向不存在的页面
- 孤页：无任何 inbound 链接的页面
- 短页：内容少于 100 字符的页面（可能是 stub 但未标记）

## 内容检查
- 重复标题：两个页面标题相同或高度相似
- 过期标记：引用的 raw 文件已删除但页面未标记 outdated
- 矛盾检测：不同页面对同一事实的描述冲突

## 健康分计算
- 每项检查通过得 1 分，失败得 0 分
- 总分 / 检查项数 = 健康分（0-100）
- 低于 60 分触发维护提醒
```

## KB-META.md：知识库元信息

文章的经验教训之一是把硬编码路径集中管理。Inkdesk 从初始化就生成 `KB-META.md`，放在 vault 根目录：

```markdown
# KB-META

## 基本信息
- name: {知识库名称}
- created: 2026-06-05
- schema_version: 1.0

## 目录结构
- raw: raw/
- wiki: wiki/
- schema: schema/
- skills: skills/
- evals: evals/
- runs: runs/

## 关联资源
- code_repo: {代码仓库本地路径，可选}
- related_kbs: {关联的其他知识库路径，可选}

## 当前统计
- 概念页：
- 实体页：
- 来源摘要：
- 总页数：
```

## 与文章做法的差异总结

| 设计决策 | 文章 | Inkdesk | 原因 |
|---------|------|---------|------|
| Schema 组织 | 单文件 CLAUDE.md | 多文件（6 个专题） | 按需加载，Agent 不需要一次读全部 |
| 页面类型 | 6 类（含 module/class） | 4 类（concept/entity/source/query） | 通用化，代码特有类型由模板扩展 |
| 代码仓位置 | vault 外部同级目录 | KB-META 关联路径 | 不把代码仓放在 vault 内 |
| 页面状态 | 无 | stub/draft/stable/outdated | 文章评测阶段才发现需要追踪成熟度 |
| 元信息管理 | 后来才抽取出 KB-META.md | 初始化就生成 | 吸取文章教训，避免路径硬编码 |
| 页面模板 | 内嵌在 CLAUDE.md | 独立 `templates/` 目录 | 模板可独立编辑和版本化 |
