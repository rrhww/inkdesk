# 问答沉淀主流程实施计划

> **给 agentic worker：** 必须先使用 `omni-superdev` 路由；涉及行为变更时使用 `superpowers:test-driven-development`，本地界面验证使用 `webapp-testing`。步骤使用 checkbox（`- [ ]`）语法跟踪。

**目标：** 在构建 Skill Workbench、Wiki Health、Evaluation Harness 或外部 Agent 连接器之前，先把 `Ask -> 沉淀` 做成 Inkdesk 的主产品路径。

**架构：** 复用现有 `AskTurn -> writeback -> ReviewItem -> ingest -> wiki` 路径，不另建一套并行沉淀系统。在当前 writeback 内部机制之上引入更清晰的 `deposit` 契约，增加整段回答和选中片段两类沉淀输入，所有正式 wiki 修改仍必须经过审阅，并记录足够元数据，供后续 Skill Workbench、评测和外部 Agent 沉淀使用。

**技术栈：** Next.js 16, React 19, TypeScript, FastAPI, SQLAlchemy, LangGraph, PostgreSQL, Vault Markdown, pytest, Vitest, node:test, Playwright

---

## 产品决策

下一个开发切片暂时不是 Skill Workbench。

文章里的 `Claude Code + Skill repo + Obsidian` 形态很有参考价值，但 Inkdesk 应该先把记忆写入口做稳定。如果 `Ask -> 沉淀` 不清晰，Skill Workbench 和外部 Agent 沉淀只会在模糊写入路径上增加更多入口。

当前实现已经具备：

- `AskTurn` snapshots in `server/inkdesk_server/models.py`.
- Ask writeback through `POST /api/ask/{ask_turn_id}/writeback`.
- Review proposals through `ReviewItem`.
- Vault-first accepted wiki writes.
- Frontend `AskAnswerCard` with a writeback button.

下一目标：

```text
Ask answer -> deposit request -> review proposal -> owner accepts -> wiki patch
```

对应两个用户可见入口：

- 沉淀整段回答。
- 沉淀选中片段。

## 交付顺序

1. 在现有 writeback 之上稳定后端沉淀契约。
2. 增加整段回答沉淀反馈和幂等性。
3. 增加选中片段沉淀。
4. 把 Ask UI 升级成明确的沉淀入口。
5. 增加待沉淀内容的健康度和就绪信号。
6. 更新文档，然后进入 Skill Workbench。

## 验收标准

- 当 `canWriteback` 为 true 时，每条 Ask 回答都有固定的 `沉淀这次回答` 动作。
- 用户可以沉淀整段回答。
- 用户可以沉淀回答中的选中文本。
- 两条沉淀路径都会创建 `ReviewItem` 提案，不直接编辑 wiki 文件。
- 重复沉淀同一内容不会创建重复的待审提案。
- 如果存在外部网页证据，必须先通过 raw 物化，再进入审阅。
- 现有 `/api/ask/{id}/writeback` 保持兼容。
- `/app/ingest` 仍是审阅门禁。
- 后端和前端验证命令通过。
