# 按文章流程开发实施计划

> **给执行型 Agent：** 先使用 `omni-superdev` 做任务路由。涉及功能实现时使用 `superpowers:test-driven-development`；涉及界面改动时使用 `ui-ux-pro-max`；涉及本地页面验收时使用 `webapp-testing`。步骤使用复选框语法，便于逐项追踪。

**目标：** Inkdesk 的开发顺序严格按文章启发的流程推进：`Sources -> LLM-Wiki -> Schema -> Skills -> Agent Runtime -> Evaluation -> Harness`。

**架构：** Inkdesk 是控制面。Vault Markdown 保存长期知识和工作流文件；PostgreSQL 保存索引、队列、运行状态和缓存读模型；LangGraph 或外部 Runtime 负责执行；Inkdesk 负责审阅、健康检查、评测和晋级门禁。

**技术栈：** Next.js 16、React 19、TypeScript、FastAPI、SQLAlchemy、LangGraph、PostgreSQL + pgvector、Vault Markdown、pytest、Vitest、node:test、Playwright。

---

## 文章流程

开发主线固定为：

```text
Sources -> LLM-Wiki -> Schema -> Skills -> Agent Runtime -> Evaluation -> Harness
```

Inkdesk 当前已经具备 `Sources -> LLM-Wiki` 的一部分：

```text
raw -> ingest -> wiki -> ask
```

下一步不是直接做自治 Agent，而是把这条现有闭环扩展成文章里的长期可演化系统。

## 执行原则

- 先建设持久层，再提升自治度。
- AI 产生的知识变更默认进入审阅，除非未来有评测支撑的门禁允许自动通过。
- 已接受知识和可复用工作流必须以文件形式保存。
- Claude Code、Codex、Cursor、Obsidian、LangGraph 都是执行器或编辑器，不是长期真相来源。
- 只有 Wiki Health 与 Evaluation Run 证明流程可靠后，才提升 Harness 自治度。

## 阶段一：Sources 与 LLM-Wiki 对齐

**目标：** 让当前 `raw -> ingest -> wiki -> ask` 闭环符合长期 LLM-Wiki 的维护方式。

**当前基线：**

- `raw/` 已保存文本、网页、PDF 和迁移笔记。
- `ingest` 已能创建可审阅提案。
- 已接受提案会写入 `wiki/*.md`。
- Ask 已能引用 wiki/raw，并创建 writeback 提案。

**下一步建设：**

- [ ] 新增显式 `/api/ask/{id}/deposit`，同时保留 `/api/ask/{id}/writeback` 兼容。
- [ ] 支持整条回答沉淀。
- [ ] 支持选中片段沉淀。
- [ ] 所有 Ask 沉淀都进入 ingest 审阅。
- [ ] Ask UI 固定展示 `沉淀这次回答` 主操作。
- [ ] 增加第一批 Wiki Health 检查：缺 frontmatter、缺来源引用、内部链接断裂、孤立页面。

**具体实施计划：** [问答沉淀主流程实施计划](2026-06-04-问答沉淀主流程实施计划.md)

**验收标准：**

- 用户可以导入来源、审阅提案、接受 wiki 页面、提问，并把有价值回答送回审阅。
- Wiki 文件可以在 Inkdesk 外部直接阅读。
- Ask 沉淀不会静默修改正式 wiki 文件。
- Wiki Health 至少能报告上述四类结构问题。

## 阶段二：Vault Schema 层

**目标：** 增加面向 Agent 的 schema 文件，说明 LLM-Wiki 应该如何维护。

**建设内容：**

- [ ] 懒创建 vault 目录：`schema/`、`skills/`、`evals/`、`runs/`。
- [ ] 添加初始 schema 文件：
- [ ] `schema/vault-layout.md`
- [ ] `schema/wiki-page-template.md`
- [ ] `schema/source-citation-rules.md`
- [ ] `schema/ingest-proposal-rules.md`
- [ ] `schema/ask-answer-rules.md`
- [ ] `schema/wiki-health-rules.md`
- [ ] 增加后端初始化助手，确保目录和默认 schema 文件存在。
- [ ] 在 UI 中提供只读 schema 浏览入口，或先放入早期 Workbench 区域。

**验收标准：**

- 新本地 vault 初始化后自动具备 schema 文件。
- schema 文件可以在 Obsidian 或任意 Markdown 编辑器中打开。
- 后端测试证明初始化是幂等的。

## 阶段三：Skill 仓库与 Skill Workbench

**目标：** 把 skills 作为一等工作流文件管理，而不是藏在应用代码里的 prompt。

**建设内容：**

- [ ] 定义 Inkdesk skill 文件格式。
- [ ] 决定 v1 使用 `skills/<skill-name>/SKILL.md`，还是更简单的 Markdown frontmatter 格式。
- [ ] 初始化第一批 skills：
- [ ] `ingest-source`
- [ ] `patch-wiki-page`
- [ ] `answer-from-wiki`
- [ ] `create-research-brief`
- [ ] `run-wiki-health-check`
- [ ] `extract-insight-from-chat`
- [ ] 增加 Skill Workbench 列表页。
- [ ] 增加 skill 详情页，展示用途、输入、上下文、输出契约、安全规则和验证要求。
- [ ] 支持导出 Codex / Claude 兼容的 skill 文件。

**验收标准：**

- skill 既能作为文件检查，也能在 UI 中看到。
- 用户不读应用代码也能理解一个 skill 的用途。
- skill 不能直接改写 wiki；输出必须成为 proposal 或 run record。

## 阶段四：Agent Runtime Run

**目标：** 为受控 skill run 增加可追踪的输入、输出与审阅状态。

**建设内容：**

- [ ] 增加数据库 run 记录。
- [ ] 增加 `runs/` vault 输出约定。
- [ ] 增加启动 skill run 的 API。
- [ ] 记录 resolved context、runtime、output、error 与 proposed wiki changes。
- [ ] 将所有 wiki-changing output 转成 ingest proposal。
- [ ] 增加 Agent Run Console，展示近期运行。

**验收标准：**

- 一个 skill run 可以根据记录的输入、上下文和输出复现。
- Runtime 错误不会污染 wiki。
- 所有 wiki 变更仍经过 review。

## 阶段五：Evaluation 层

**目标：** 在提高自治度前加入 golden tasks 与 rubrics。

**建设内容：**

- [ ] 增加 `evals/golden-tasks/` 存储。
- [ ] 定义 rubric 文件格式。
- [ ] 增加 evaluation run 记录。
- [ ] 支持隔离评测运行，不修改 canonical wiki。
- [ ] 支持比较 skill / wiki / schema 版本。
- [ ] 为 skill version、schema version、runtime configuration 增加 promotion gate 字段。

**验收标准：**

- Evaluation run 可见、可重复。
- 失败评测不能修改 accepted wiki。
- skill 或 schema 变更可以在 promotion 前比较。

## 阶段六：Harness 与自治门禁

**目标：** 只有在前面几层稳定后，增加多步骤 Harness。

**建设内容：**

- [ ] 增加 staged run orchestration。
- [ ] 增加阶段间 gates。
- [ ] 增加 retry 与 rollback 记录。
- [ ] 增加人工审批 checkpoint。
- [ ] 只有具备通过评测历史的 workflow 才允许更高自治度。

**验收标准：**

- Harness 记录每个阶段的决策。
- Owner 可以暂停、拒绝、重试或回滚一个 run。
- 高自治度必须 opt-in，并且有 evaluation history 支撑。

## 当前优先级

从阶段一开始。

当前实现目标仍然是 Ask deposit，但原因必须按文章流程理解：

```text
Ask deposit 是 LLM-Wiki 写入通道。只有这条通道稳定，后续 Schema、Skills、Evaluation 和 Harness 才能安全叠加。
```

在阶段一完成以下能力前，不开始 Skill Workbench 实现：

- 整条回答沉淀。
- 选中片段沉淀。
- Review-first proposal 流程。
- 基础 Wiki Health 检查。

## 提交策略

建议提交顺序：

```bash
git commit -m "feat: add ask deposit endpoint"
git commit -m "feat: support selected ask deposit"
git commit -m "feat: make ask deposit the primary action"
git commit -m "feat: add basic wiki health checks"
git commit -m "feat: initialize vault schema directories"
git commit -m "feat: add initial skill repository"
```

## 后续阶段前暂不做

- Skill Workbench 未完成前，不做外部 MCP server。
- Evaluation 未完成前，不做自治多步骤运行。
- 不允许静默写 wiki。
- 不做团队或多用户知识库。
- 不把 Obsidian 作为硬依赖。
