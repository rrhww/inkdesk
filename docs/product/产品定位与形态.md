# 产品定位与形态

> 基于文章《AI研发自动化：Wiki知识库+技能包》的思考和对 Inkdesk 产品方向的重新确认。

## 产品定位

```text
Inkdesk 是一个 AI 研发自动化控制台。

它把 LLM-Wiki 知识库、研发 Skill 技能包、Evaluation 评测系统和 Harness 编排层整合成一个完整应用。

你用 Inkdesk 来：
  - 导入 PRD、bug、改造目标或研发任务
  - 为任务生成 context pack、技术方案、评审意见、coding 输入和测试计划
  - 让研发过程中的判断、方案、修复经验沉淀进 LLM-Wiki
  - 通过 Skills、Evaluation 和 Harness 把重复研发步骤逐步自动化

如果你也用 Claude Code 或 Codex，
Inkdesk 通过 MCP 接入它们，作为统一的研发上下文和沉淀底座。
但 Inkdesk 本身是一个独立、完整的产品。
```

## 产品形态

### 界面结构

```text
/app
├── runs/          研发任务台 —— PRD / bug / 改造任务的自动化运行记录
├── ask/           Context Ask —— 为当前任务查询长期上下文
├── review/        审阅台 —— 待处理的 AI 提案、方案、wiki patch、沉淀裁决
├── wiki/          知识库浏览 —— 页面、claim、来源追溯
├── skills/        技能工作台 —— 技术方案、评审、coding、测试、排查等技能
├── health/        健康仪表盘 —— 知识、Skill、评测与运行质量趋势
├── evals/         评测中心 —— golden tasks、版本对比、晋级门禁
└── settings/      基础设置
```

### 核心闭环

```text
PRD / 研发任务
  → Context Pack
  → 技术方案
  → 技术评审
  → Coding
  → 测试 / 修复
  → Review / Deposit
  → Wiki / Runs / Evals
```

Ask 不是产品主驾驶舱，而是研发任务里的上下文查询能力：

```text
研发任务进行中
  → 询问相关历史方案 / 领域模型 / 风险 / open questions
  → 基于 Wiki 和 raw 返回带引用回答
  → 关键判断可沉淀为 proposal
  → 进入 review-first 流程
```

### 三层扩展

```text
schema → skills → evals → runs → harness
```

- `schema/`：面向 Agent 的知识维护规则
- `skills/`：技术方案、评审、coding、测试、排查等可复用研发流程
- `evals/`：golden tasks、rubrics 和评测运行记录
- `runs/`：研发任务运行记录、阶段输出和审阅状态
- `harness`：PRD 到交付链路的门禁、重试和回滚编排

### MCP Server 是产品的延伸

Inkdesk Web 是用户唯一的交互界面。外部 Agent 在 Harness 路径中是后端执行服务：

```text
用户 ←→ Inkdesk Web（唯一交互界面）
           │
           ├─ 创建任务、审阅阶段输出、管理知识与评测
           │
           ├─ Inkdesk LangGraph（知识层编排）
           │
           └─ Harness（桥接层）
                │
                └─ 调用外部 Agent（后端执行服务）
                     └─ Claude Code / Codex：coding、test、debugging
```

用户在 Web 端创建 Dev Run、审阅阶段输出、裁决关键门禁，Harness 在背后调度外部 Agent 执行 coding 和测试。用户不需要打开 Claude Code。

同时 MCP 保留手动路径：用户在 Claude Code 里做探索性工作时，可以手动调用 `/context-pack` 获取知识底座，用 `/deposit` 把有价值的东西沉淀回来。两条路径共用同一套 MCP 工具（详见 [工具链与模型上下文协议](../architecture/工具链与模型上下文协议.md)）。

## 与文章方案的对比

| | 文章方案（工具拼装） | Inkdesk（产品） |
|------|------|------|
| 知识摄入 | Claude Code 对话中执行 skill | Web 引导式上传 → 后台编译 → 审阅队列 |
| 审阅提案 | 对话中逐条确认 | 独立审阅台，diff 预览，批量操作 |
| 知识浏览 | Obsidian | Web 内浏览 + 关联图 + 来源追溯 |
| 任务入口 | 用户在 Agent 里手动触发 skill | Web 研发任务台：PRD / bug / 改造任务进入 Dev Run |
| 搜索 | qmd CLI | Context Ask，服务当前研发任务 |
| 健康检查 | 手动跑 lint | 自动定期扫描 + 仪表盘 + 趋势图 |
| 评测 | `claude -p` 子进程 | 评测中心：配置 golden tasks → 跑分 → 对比版本 |
| 多平台 | 手动配置各平台 | 开箱即用，MCP 作为可选扩展 |
| 上手 | 需学会 Obsidian + qmd + Git + Claude Code | 打开浏览器，直接开始 |

## 从文章吸收的结构改进

| 改进点 | 说明 |
|--------|------|
| Schema 提前 | Schema 在内容灌入前就位，不做事后补救 |
| index.md + log.md | 初始化就生成全局导航和操作日志 |
| KB-META.md | 集中管理元信息，避免路径硬编码 |
| status 字段 | 页面 frontmatter 从初始化就包含 stub/draft/stable/outdated |
| 评测分层 | 结构层 → 实用层 → 信号层 → 治理层 |
| 盲评机制 | 评测时分离生成和评分，避免自我偏好 |

## 当前开发策略

**先修正产品主入口，再打造 MCP Server 与研发自动化底座。**

Inkdesk 不应长期停留在 Ask-first。Ask 是研发自动化里的上下文查询能力，首页应该表达“研发任务正在被自动化推进”。MCP Server 是连接外部 Agent（Claude Code / Codex）与 Inkdesk 研发上下文的核心通道，但 Web 产品首先要提供 Dev Run Console，让用户从 PRD、bug 或改造任务进入自动化流程。

### 开发主线

```text
Dev Run Console（PRD / bug / 改造任务入口）
  → Context Ask + MCP Server（context_pack / deposit / search）
    → 后台编译流水线（知识沉淀 + 阶段输出审阅）
      → Skills（方案 / 评审 / coding / 测试 / 排查）
        → Evaluation + Harness（门禁、重试、回滚、晋级）
```
