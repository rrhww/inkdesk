# Inkdesk MVP 产品需求文档

## 1. 产品背景

Inkdesk 当前不是一个公开发布平台，也不是一个带完整计划系统的超级工作台。

它已经从“私有 LLM Wiki / Ask-first 研究工作台”继续收敛成一个更明确的产品：

- 单人私有
- vault-first
- Dev Run-first
- PRD / bug / 改造任务驱动
- Ask 作为 Context Ask 服务研发任务
- 可作为外部 Agent 的研发上下文和沉淀底座

当前最重要的问题不是“展示给访客看什么”，也不是“让用户在首页聊天”，而是：

**如何让用户从 PRD / bug / 改造任务启动一条可追踪 Dev Run，并让知识、Skill、评测和外部 Agent 都围绕这条研发任务服务。**

## 2. 产品定位

Inkdesk 当前 MVP 的正式定位是：

- 一个以 `PRD / 研发任务 -> Dev Run -> 阶段输出 -> Review / Deposit` 为用户主路径、以 `raw -> ingest -> wiki -> ask -> health -> ingest` 为内部闭环的 AI R&D Automation Runtime

它的核心不是“直接让 AI 自动上线代码”，而是：

1. 用户提交 PRD、bug 或改造目标
2. 系统生成任务 context pack，召回 wiki、raw、历史判断和 open questions
3. 系统用 Skills 逐步生成技术方案、评审意见、coding 输入和测试建议
4. 每个阶段输出进入 review-first 流程，必要时由 owner 确认
5. 阶段成果和关键判断 deposit 回长期知识，继续服务后续 Dev Run 和外部 Agent

## 3. 目标用户

### 主要用户

- owner，也就是你自己
- 使用目标：把 PRD、bug、改造任务、网页、PDF、旧笔记和外部 Agent 产出的高价值方案/修复经验沉淀进一个长期可追溯的研发自动化系统

### 扩展使用者

- Claude Code、Codex、Cursor 等外部 Agent
- 使用目标：在研发任务前获取 Inkdesk 的长期上下文，在任务后把方案、代码决策、测试修复和复盘交给 Inkdesk 沉淀

### 当前非目标用户

- 公开访客
- 协作者
- 团队成员

当前产品没有公开阅读面，也没有多人协作角色。

## 4. MVP 目标

MVP 需要跑通三条链路。

### 链路一：研发任务入口闭环

1. owner 通过 `/login` 登录
2. 进入 `/app` 研发任务台
3. 新建 PRD / bug / 改造任务
4. 系统展示任务阶段：Context Pack、方案、评审、coding、测试、沉淀
5. 当前阶段可以先用 mock / fixture 表达 Dev Run 结构，不要求第一版真正自动编码

### 链路二：知识沉淀闭环

1. 原始材料进入 `raw/`
2. 系统生成 `ingest` 审阅提案
3. owner 接受后沉淀到 `wiki/`
4. Dev Run 或 Context Ask 中的高价值判断可以 deposit
5. 沉淀动作生成 topic create / patch / open question / 冲突裁决

### 链路三：Context Ask 闭环

1. owner 在 Dev Run 中发起上下文查询
2. 系统基于 wiki 和 raw 返回带引用回答
3. 回答可以继续追问，也可以 deposit 回审阅队列

知识沉淀闭环已经有基础实现；下一阶段 MVP 的主目标是把 `/app` 从 Ask-first 改成 Dev Run-first，并把 Ask 降级为 Context Ask。

## 5. 核心价值

- 对 owner：把“任务输入、上下文准备、阶段产出、审阅沉淀”放进同一条研发闭环
- 对系统：让 AI 参与研究，但不允许它静默污染长期知识
- 对用户体验：把复杂的 claim / ingest / health 逻辑藏到后台，让用户看到研发任务进度和关键裁决点
- 对外部 Agent：提供统一的研发上下文、任务阶段输出与沉淀接口

## 6. MVP 范围内

### 私有入口

- `/login`
- 单 owner 登录
- `/app` 研发任务台首页
- `/` 根据登录态重定向到 `/login` 或 `/app`

### 研发任务主路径

- `runs`：研发任务、阶段状态、待确认输出
- `context`：为当前任务生成 context pack
- `skills`：技术方案、评审、coding、测试、排查等技能入口
- `review`：阶段输出和知识提案审阅
- `deposit`：任务成果、回答片段、方案判断进入长期知识

### 知识支撑路径

- `raw`：原始材料导入与索引
- `ingest`：AI 提案审阅
- `wiki`：知识页列表与详情
- `ask`：Context Ask，服务当前研发任务的上下文查询、追问、writeback 提案

### 支撑能力

- Vault Markdown 持久化
- PostgreSQL 索引与队列
- owner session
- 本地全栈验收与自动化测试
- MCP / CLI 接入设计文档

## 7. MVP 范围外

- 公开文章与公开首页
- plans 主路径
- search 主路径
- settings 主路径
- 多人协作
- 自治 Agent 执行系统
- 面向公众的营销站
- 让用户手动管理完整 claim 生命周期
- 在第一阶段实现完整 MCP server 生产化
- 真正自动修改代码并提交上线
- 完整 Harness 编排和自动回滚

## 8. 核心流程

### 流程一：导入原始材料

1. owner 在 `/app/raw` 提交网页、PDF 或文本
2. 后端写入 `raw/` vault 文件并建立索引
3. 材料进入待编译状态

### 流程二：审阅 AI 提案

1. owner 打开 `/app/ingest`
2. 查看 AI 对 source 的编译结果或补丁建议
3. 决定接受或拒绝

### 流程三：沉淀到 wiki

1. 被接受的提案写入或更新 `wiki/`
2. 页面保留 current understanding、claims、questions、sources
3. 数据可从 vault 恢复，不依赖数据库独占保存

### 流程四：创建研发任务

1. owner 在 `/app` 新建 PRD / bug / 改造任务
2. 系统生成 Dev Run，展示阶段轨道和当前状态
3. 第一版可以先把阶段输出作为结构化占位或人工触发结果
4. 后续逐步接入 Skills、外部 Agent 和 Harness

### 流程五：Context Ask

1. owner 在 Dev Run 或 `/app/ask` 发起上下文问题
2. 系统优先读取 wiki，再补充 raw
3. 如 owner 显式允许，可短暂联网补料
4. 如答案值得沉淀，owner 点击“沉淀这次回答”

### 流程六：后台沉淀

1. 主 Agent 接收当前问题、回答、引用、上下文和用户沉淀意图
2. Insight Extractor 提取可沉淀判断
3. Evidence Binder 绑定 raw / wiki / web source 证据
4. Topic Router 决定新建 topic、patch 旧 topic 或生成 open question
5. Conflict Checker 检查和旧 claim 的冲突
6. Patch Writer 生成 vault/wiki patch
7. Quality Gate 判断是否自动写入，或生成轻量用户裁决卡片

### 流程七：外部 Agent 接入

1. Claude Code、Codex 或 Cursor 通过 MCP / CLI 请求 Inkdesk context pack
2. 外部 Agent 使用相关长期知识完成方案、coding、测试或排查任务
3. 用户要求沉淀时，外部 Agent 调用 Inkdesk deposit
4. Inkdesk 复用后台沉淀流水线，不要求外部 Agent 理解内部知识协议

## 9. 产品原则

- Vault 为底：长期真相必须能回到 Markdown 文件
- 审阅优先：AI 只能提议，不能静默写正式知识
- 私有优先：先把 owner 的闭环做稳，再考虑对外表达
- Run-first：`/app` 首先表达研发任务，而不是一个大问答框
- Context Ask：Ask 的关键不是聊天体验，而是为当前 Dev Run 提供可引用上下文
- 低打扰优先：用户只看到“沉淀”和必要冲突裁决，后台治理不暴露成日常负担
- 低复杂度优先：MVP 只做当前主路径，不恢复旧模块叙事

## 10. 成功标准

满足以下条件即可认为 MVP 可用：

- `/login` 可登录
- `/app` 首屏展示研发任务台，而不是 Ask-first 问答页
- owner 可以创建或查看一个 Dev Run 结构
- `/app/raw` 可导入并读取来源
- `/app/ingest` 可接受 / 拒绝提案
- `/app/wiki` 与 `/app/wiki/[id]` 可稳定展示知识页和来源
- `/app/ask` 作为 Context Ask 可返回带引用的回答
- Ask writeback 只生成提案，不会直接改写 wiki
- 每个回答都能触发沉淀动作
- 沉淀动作能生成可审阅的 topic create / topic patch / open question
- 外部 Agent 接入方案在文档中明确，后续可落地为 MCP / CLI

## 11. 阶段边界

### 当前阶段进度

- 本地全栈闭环已成立
- owner 登录、raw、ingest、wiki、ask 已有实现
- Ask writeback 已有后端和前端入口雏形
- claim 证据状态、使用次数、过期和冲突信号已有实现
- 本地 Docker 栈、后端 pytest、前端 typecheck 已验证
- 旧 public / plans / note editor 方向已退出主规范

### 下一阶段优先项

- `/app` 从 Ask-first 改成 Dev Run Console
- 增加研发任务数据模型或前端结构化 fixture，用于表达 PRD -> 方案 -> 评审 -> coding -> test -> deposit 阶段
- Ask 改名和定位为 Context Ask
- 回答卡片上的“沉淀这次回答”服务 Dev Run 和知识沉淀
- 后台专项子 Agent 沉淀流水线
- MCP / CLI 接入外部 Agent
- 更稳定的 retrieval 与证据链
- 更清晰的 source/topic/claim 演化流程
