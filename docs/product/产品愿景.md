# Inkdesk 产品愿景

## 当前定论

Inkdesk 的核心不是问答，也不是 wiki，而是一个面向 AI 研发流程自动化的控制台。

用户最终只需要负责两类高价值判断：

1. 提供 PRD、bug、改造目标或研发任务。
2. 在关键门禁处确认方案、风险、代码变更、测试结果和知识沉淀。

系统负责把任务拆成需求理解、上下文检索、技术方案、技术评审、coding、测试准备、自动化测试修复、问题排查和复盘沉淀等阶段。LLM 是执行者，Harness 是导演；Inkdesk 负责上下文、审阅、评测、门禁和长期记忆。

这意味着 Inkdesk 的产品主形态不是：

```text
Ask -> 沉淀
```

而是用户可感知的 Dev Run：

```text
PRD / 研发任务
  -> Context Pack
  -> 技术方案
  -> 技术评审
  -> Coding
  -> 测试 / 修复
  -> Review / Deposit
```

内部仍然保留 `raw / ingest / wiki / ask / health` 的工程闭环，但它们是 Dev Run 的基础设施，不是首页主叙事。

## 一句话定义

Inkdesk 是一个 AI R&D Automation Runtime：以 LLM-Wiki 作为长期上下文，以 Skills 作为研发动作协议，以 Evaluation 和 Harness 作为质量门禁，把 PRD 到交付的研发流程逐步自动化。

它既可以作为独立的研发任务控制台使用，也可以作为 Claude Code、Codex、Cursor 等外部 Agent 的上下文、沉淀和门禁底座。

## 核心洞察

AI 研发自动化最大的难点，不是让模型写出一段代码，而是让模型稳定地完成一条可交付、可验证、可回滚的研发链路。

普通 AI 编码工具解决的是：

- 怎么补全代码
- 怎么解释代码
- 怎么临时生成一个方案
- 怎么回答当前上下文里的问题

AI 研发自动化真正要解决的是：

- PRD 怎么被稳定理解
- 历史业务和代码上下文怎么复用
- 技术方案怎么被评审和修正
- coding 输出怎么和方案、规范、测试对齐
- 测试数据和自动化测试怎么保证真实可靠
- 每一步失败后怎么重试、回滚和沉淀经验
- 哪些阶段可以自动通过，哪些阶段必须人工确认

所以，Inkdesk 不应该只做“基于资料回答”，而应该做：

**研发任务从输入到交付的受控自动化。**

用户体验必须足够清楚：用户从 PRD、bug 或改造任务进入 Dev Run，看到当前阶段、阶段输出、阻塞点和下一步动作。复杂的 claim、health、lint、ingest 队列仍然存在，但它们服务于任务推进和门禁判断。

## 新产品命题

每一个研发任务，都应该生成一条可追踪 Dev Run：

- 用户提交 PRD / bug / 改造目标。
- 系统生成 context pack，召回相关 wiki、raw、历史方案、open questions。
- 技术方案 Skill 生成方案草案。
- 技术评审 Skill 检查边界、风险、依赖、测试范围。
- Coding / Test Skill 在外部 Agent 或本地 runtime 中执行。
- Evaluation 和 Harness 判断能否进入下一阶段。
- 任务结果、关键判断和修复经验 deposit 回长期知识。

这意味着 Ask 不是首页主功能，而是 Dev Run 的 Context Ask：当任务需要解释历史判断、查相关模块、补足证据或复盘结论时，Ask 提供带引用的上下文回答，并把高价值回答送回沉淀流程。

## 外部 Agent 底座

Inkdesk 的长期产品目标，是成为外部 Agent 的研发自动化底座。

Claude Code、Codex、Cursor 等 Agent 可以通过 MCP、CLI 或项目级 Skill 接入 Inkdesk：

- 任务开始前：调用 Inkdesk 获取 PRD 相关 context pack、历史方案、领域约束和 open questions。
- 任务进行中：查询某个 topic、decision、接口、风险或测试要求。
- 阶段结束后：把方案、评审意见、代码决策、测试修复和复盘提交给 Inkdesk 沉淀。

外部 Agent 不需要理解 Inkdesk 内部的 claim、review、health、vault patch。它们只需要两类能力：

```text
context_pack(task) -> 返回短上下文包
deposit(answer/context) -> 后台沉淀长期知识
```

这让 Inkdesk 从“自己的知识 App”升级为“所有 Agent 都能调用的研发上下文、评测和沉淀层”。

## 新范式

### 1. 用户动作不是提问，而是启动研发任务

用户不应该被迫理解复杂的知识治理流程。

主路径必须保持：

```text
提交任务 -> 查看阶段输出 -> 关键节点确认 -> 交付 / 沉淀
```

`raw / ingest / wiki / ask / claim / health / lint` 都是后台协议。只有当系统无法安全自动推进某个研发阶段时，才用轻量确认卡片请用户裁决。

### 2. 知识不是笔记，而是有状态的 claim

文档仍然重要，但文档不是最小知识单位。

Inkdesk 真正关心的是 claim：

- 它说了什么
- 它来自哪里
- 它被哪些 Ask 调用过
- 它有多新
- 它有没有冲突
- 它是否仍然值得相信

wiki 页面只是 claim 的阅读界面，不是知识系统的终点。

### 3. Ask 不是首页，而是 Context Ask

一次 Ask 应该输出两种结果：

- 给用户看的回答
- 给沉淀系统看的上下文包

上下文包至少包括：

- 本次回答依赖了哪些 wiki / raw
- 证据是否足够
- 新增了哪些候选 claim
- 触发了哪些 knowledge gap
- 是否建议生成 ingest 提案
- 是否发现旧理解需要重审

### 4. Ingest 不是用户主路径，而是认知变更管理

Ingest 不应该成为用户每天必须处理的沉重队列。

它是 Dev Run 和 deposit 背后的后台管理层，负责判断：

- 这是不是新知识
- 这是不是旧知识补丁
- 这是不是冲突
- 这是不是低证据推测
- 这是不是重复 topic
- 这是不是应该暂存为 open question

### 5. Wiki 不是仓库，而是当前理解的可读投影

wiki 保存的是用户确认过的当前理解，但它必须暴露自己的状态。

一个 wiki 页面应该能回答：

- 当前我们相信什么
- 依据是什么
- 哪些地方不确定
- 最近一次被验证是什么时候
- 哪些 Ask 最近调用过它
- 哪些 claim 需要重审

### 6. Knowledge Health 应该后台运行，必要时轻量打断

知识健康不能消失，但也不能压到用户主路径上。

它应该后台运行，并在以下情况才打断用户：

- 新沉淀结论和旧判断冲突。
- 新沉淀内容证据不足但会影响核心判断。
- 系统需要用户在“覆盖旧判断 / 保留旧判断 / 合并表达”之间做选择。

## 产品定位

Inkdesk 不是：

- AI 聊天工具
- 普通 RAG 知识库
- Obsidian 替代品
- Notion 式工作台
- llm-wiki 的 UI 外壳
- 需要用户手工维护 claim 的知识治理后台
- 一个已经放开自治权限的自动写代码黑盒

Inkdesk 是：

- 面向 AI 研发自动化的 Dev Run Console
- 一个把 PRD、上下文、Skills、评测和沉淀串起来的 runtime
- 一个让外部 Agent 能取用研发上下文并回写阶段成果的系统
- 一个把复杂知识治理和研发门禁藏在后台、只把必要裁决交给用户的控制台

## 核心闭环

用户感知闭环：

```text
PRD / 任务 -> Dev Run -> 阶段输出 -> 审阅 / 交付 / 沉淀
```

内部工程闭环：

```text
raw -> ingest -> wiki -> ask -> health -> ingest
```

- `raw` 提供证据
- `ingest` 管理认知变更
- `wiki` 展示已确认理解
- `ask` 为当前任务提供上下文查询
- `health` 暴露缺口、老化、冲突和未沉淀价值
- 新的 health 信号再进入 ingest

用户不需要直接操作这条内部闭环，但系统必须保留它，确保自动化研发不是把模型输出直接推向交付，而是进入可追溯、可审阅、可评测、可修正的流程体系。

## 创新重点

### Dev Run Console

`/app` 应该首先展示研发任务、阶段状态、待确认输出、可用 Skills 和知识健康，而不是一个大问答框。

### Context Ask

Ask 作为 Dev Run 的辅助能力存在，用于查询当前任务相关的长期知识、历史判断、风险和 open questions。

### Answer-level Deposit

每个 Context Ask 回答都可以沉淀。沉淀按钮是 Dev Run 内的关键动作，不是隐藏在二级页面里的 writeback 功能，也不是把 Ask 重新抬回产品首页。

### Specialized Sub-agent Pipeline

沉淀由主 Agent 编排专项子 Agent 完成：Insight Extractor、Evidence Binder、Topic Router、Conflict Checker、Patch Writer、Quality Gate。

用户不看这条流水线，但它决定沉淀质量。

### Claim-first Knowledge

知识原子从 page 下沉到 claim。系统关心每条 claim 的来源、状态、使用记录和健康度。

### Inkdesk as Agent R&D Context Runtime

Inkdesk 通过 MCP / CLI / Skill 接入外部 Agent，为 Claude Code、Codex、Cursor 等工具提供研发上下文、Dev Run 事件记录和沉淀接口。

### Inkdesk as AI R&D Automation Runtime

Inkdesk 通过 Skills、Evaluation 和 Harness，把技术方案、评审、coding、测试准备、自动化测试修复和问题排查逐步编排成可控研发流程。

### Ingest as Cognitive Change Control

Ingest 是后台认知变更管理层，负责处理新增、补丁、冲突、弱证据和开放问题。

### Knowledge Health as Background Engine

知识健康不是复杂报表，而是后台质量引擎。它告诉系统哪些内容可以自动沉淀，哪些需要用户裁决。

## 第一批用户

第一批用户不是泛知识管理用户，而是有高频专业判断需求的人。

典型用户是：

- 正在学习和研究前沿技术的人
- 长期分析某个行业或产品方向的人
- 有一批专业资料，希望 AI 真正围绕它们工作的个人
- 高频使用 Claude Code、Codex、Cursor 等 Agent，希望 Agent 有长期上下文的人
- 想展示 agent 开发能力和 AI 产品理解的 builder

他们的共同点不是“爱记笔记”，而是：

**他们需要让自己的专业判断和 Agent 工作上下文越来越可靠。**

## 成功标准

Inkdesk 的成功不看“能不能问”，而看：

- 用户是否能从 PRD / bug / 改造任务启动 Dev Run
- 每个阶段是否有清晰输入、输出、状态和下一步动作
- Ask 是否能服务当前任务，而不是抢占首页主心智
- 外部 Agent 是否能获取研发上下文并回写阶段成果
- 高价值方案、评审、修复经验是否能进入长期知识体系
- wiki 是否能暴露 claim、证据和未解问题
- 系统是否越用越会研发，而不是越用越像聊天记录

## 产品原则

- 先做 Dev Run 主入口，再做功能广度
- 先把用户主路径从“问答”改成“研发任务推进”，再扩展后台治理
- 先把 claim 管好，再扩展页面和自动化
- AI 可以提出认知变更，但不能直接污染正式知识
- 外部 Agent 接入时只暴露简单协议：取上下文、沉淀结果
- LLM-Wiki 是底层机制，不是产品终点
