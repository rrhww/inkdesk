# Inkdesk，正在把 AI 研发任务变成可控自动化流程

> 不是又一个 AI 问答软件，也不是单纯的知识库，而是一套从 PRD 到交付逐步自动化的私有研发运行时。

很多 AI 产品已经很会回答问题，也很会生成代码了。

但真正麻烦的，从来不是让 AI 回一句话，或者写出一段看起来能跑的代码。

麻烦的是：一个研发任务从 PRD、上下文整理、技术方案、技术评审、编码、测试、修复，到最后复盘沉淀，通常散落在聊天窗口、代码仓库、临时文档和人的脑子里。今天 Agent 做过的判断，明天另一个 Agent 未必知道；这次踩过的坑，下次还可能重新踩；这轮需求背后的产品约束，到了测试修复阶段又被遗忘。

Inkdesk 想解决的，就是这个问题。

它当前不是一个公开发布平台，也不是一个什么都想装进去的超级工作台。它更像一个为单个 owner 服务的 AI 研发自动化控制台，把原始材料、长期知识、上下文包、技能链路、阶段产物和外部 Agent 工作过程放进同一条可审阅、可追溯、可沉淀的闭环里。

它的最终目标，不是让 AI 替你完成一次孤立的问答。

而是让你拥有一套可以长期复用上下文、编排研发流程、审阅关键产物、沉淀团队记忆的 AI R&D Automation Runtime。

## 01 Inkdesk 的目标，不是替你记笔记

传统笔记软件解决的是记录问题。

但 AI 研发自动化真正难的，不是记录，而是把研发过程变成可持续推进、可复盘、可复用的流程。

资料越来越多，结论越来越碎，聊天记录越来越长，代码改动越来越分散，最后真正能留下来的东西却很少。很多系统都在强调生成、强调自动化、强调更强的模型，但真正缺少的是一套稳定的研发任务进入机制和流程控制台。

Inkdesk 要做的，不是把所有输入都直接变成最终结论，也不是让用户手工维护一套复杂的知识治理后台。

它更关心的是：当用户丢进一个 PRD、bug、重构任务或技术问题时，系统能不能自动准备上下文、选择技能链路、生成阶段产物、暴露关键审阅点，并在任务结束后把有价值的判断沉淀回长期知识。

它解决的不是单次写作效率，而是 Agent 参与研发时的长期上下文能力、流程可控性和交付可靠性。

## 02 主形态：Dev Run，然后沉淀

Inkdesk 的用户主路径应该围绕研发任务展开：

`PRD / bug / 改造任务 -> Dev Run -> Context Pack / Context Ask -> 阶段输出 -> Review / Deposit`

用户进入 Inkdesk 后，看到的主入口不应该是一个巨大的 Ask 输入框，而应该是一个研发任务运行台。用户创建一个 Dev Run，系统围绕这个任务组织上下文、技能、阶段状态和审阅卡片。

Dev Run 负责把一次研发工作拆成可控阶段：

- 任务进入：记录目标、范围、约束、关联仓库和风险。
- 上下文准备：从 wiki、raw、历史 run 和外部 Agent 记录中生成 Context Pack。
- 技能编排：按需触发技术方案、技术评审、编码准备、测试设计、修复分析等技能。
- 质量门禁：在关键节点暴露审阅、冲突、证据不足和测试风险。
- 结果沉淀：把方案、判断、踩坑、测试结论和复盘写回长期知识。

Ask 仍然重要，但它不再是产品主形态。它应该被重新定位为 Context Ask：服务于某个 Dev Run 的上下文查询、追问和解释能力。

这才是产品体验上的关键：用户不是来“问 AI 一个问题”，而是在启动和推进一个 AI 参与的研发任务。

## 03 私有、可审阅、可追溯，不是附加项

对于 AI 研发流程来说，AI 能不能真正用起来，不只取决于它是否聪明，更取决于它是否可靠。

所以 Inkdesk 从一开始就把几件事放在核心位置。

第一，它是单人私有的。

当前产品默认服务对象只有 owner 自己，没有公开阅读面，也没有多人协作角色。系统首先要做稳的，是 owner 自己的研发自动化闭环，而不是对外展示。

第二，它是审阅优先的。

AI 可以提出技术方案，可以提出 wiki patch，可以整理测试风险，也可以把 Dev Run 阶段里有价值的发现转成 writeback 提案，但它不能静默改写正式知识或绕过关键质量门。进入 wiki 的内容，必须经过 owner 接受。

第三，它是来源可回溯的。

每一条被接受的知识、每一个阶段结论，都应该能回到原始材料、具体任务、相关代码上下文和当时的审阅记录。对研发自动化系统来说，这不是锦上添花，而是底线。

## 04 从 raw 到 Dev Run，Inkdesk 的内部路径已经很明确

Inkdesk 当前的内部闭环可以理解为：

`raw -> ingest -> wiki -> context -> dev run -> deposit`

`raw/` 保存网页、PDF、迁移进来的旧笔记、PRD、技术资料和其他原始材料。

`ingest` 是 AI 提案和人工审阅的中间层。系统先理解材料，生成待审阅提案，再由 owner 决定接受还是拒绝。

`wiki/` 保存被接受后的知识页。这里不是临时聊天记录，而是长期记忆区，承接 current understanding、claims、questions、sources 和研发判断。

`context` 负责把长期知识转换成任务可用的上下文包，让外部 Agent 和 Dev Run 不必从零开始。

`dev run` 是 AI 研发自动化的主入口，负责承载一次从任务输入到阶段产物、审阅门禁、沉淀复盘的完整研发流程。

`deposit` 则把 Dev Run 或外部 Agent 的高价值产物送回后台沉淀流程。

这条链路的意义，不在于把功能拆成几个名字。

而在于它把原始输入、长期知识、任务上下文、AI 阶段产物和研发复盘第一次真正串成了一条完整路径。

## 05 Vault-first，让长期判断真正留在你手里

Inkdesk 的另一个关键选择，是 vault-first。

系统可以用数据库做索引、做队列、做缓存、做工作流状态管理，但长期真相不能只存在数据库里，更不能只存在一段大模型对话里。

被接受的知识，最终要能回到 vault markdown。

这意味着，哪怕索引层丢了，哪怕读模型需要重建，长期知识本身依然是可恢复、可迁移、可备份的。对于一个希望长期使用的私人研发自动化系统来说，这种边界非常重要。

它形成的不是一个被产品界面绑住的内容池，而是一套 owner 真正持有的数据资产。

## 06 Inkdesk 不只服务自己的 App，也服务外部 Agent

Inkdesk 的长期目标，是成为外部 Agent 的研发上下文和流程底座。

用户使用 Claude Code、Codex、Cursor 或其他 Agent 时，也应该能接入同一个长期研发系统：

- 任务开始前，Agent 调用 Inkdesk 获取相关产品记忆、历史判断、技术约束和开放问题。
- 任务推进中，Agent 把阶段产物、关键决策、测试结果和阻塞点同步给 Inkdesk。
- 任务结束后，用户要求沉淀，Agent 把方案、回答、代码解释或选中片段交给 Inkdesk。
- Inkdesk 复用同一套后台沉淀和质量门禁，不要求外部 Agent 理解内部知识治理逻辑。

所以 Inkdesk 未来会提供几类接入：

- `context_pack`：任务前取上下文。
- `deposit`：任务后沉淀结果。
- `run_start`：创建或绑定一次研发任务。
- `run_event`：记录阶段产物、测试结果、风险和审阅事件。

这让 Inkdesk 从一个私有研究工作台，扩展成所有 Agent 都可以调用的 AI 研发自动化运行时。

## 07 Context Ask，不再是产品主入口

很多 AI 产品把问答当成终点。

但在 Inkdesk 里，Ask 更像 Dev Run 里的上下文工具。

用户不是来和一个泛化助手闲聊，而是在某个研发任务中围绕已有 wiki、raw、历史 run 和代码上下文继续追问，识别缺口，补足证据，形成下一轮判断。

系统优先读取 wiki 和 Context Pack，再按需回退 raw。必要时可以在 owner 明确允许下短暂联网补料，但新的外部材料也不能直接绕过系统边界，而是要重新回到 raw 和 ingest 体系里。

这样一来，Ask 不再只是一次性消费答案的界面。

它变成 Dev Run 的辅助能力：帮助用户解释上下文、追问方案、定位冲突，并把有价值回答送入长期沉淀。

## 08 Inkdesk 的最终形态，是 AI R&D Automation Runtime

如果只看当前阶段，Inkdesk 已经有一条收敛得很明确的基础闭环。

它正在稳定一套单人私有、vault-first、以 `raw -> ingest -> wiki -> context -> dev run -> deposit` 为内部路径的研发自动化底座。

但如果往最终形态去看，它想成为的并不是一个窄功能工具。

它更像一个 AI R&D Automation Runtime。

在这个系统里，原始材料有稳定入口，AI 负责提出理解和阶段产物，人负责审阅关键门禁，长期知识被持续整理进 wiki，而 Dev Run、Context Ask 和外部 Agent 负责把这些记忆重新调动起来，继续参与新的需求分析、方案设计、编码、测试、修复和复盘。

到那时，Inkdesk 承担的就不只是资料管理。

它会成为一个 owner 的长期研发底盘，一个可以持续积累判断、保留上下文、支持复盘、追问和外部 Agent 协作的私人研发中枢。

这也是项目最终目标最准确的表达。

不是再做一个会生成内容的 AI 工具。

而是做一套真正属于你自己的、可编排、可审阅、可追溯、可持续沉淀、可被 Agent 调用的 AI 研发自动化 Runtime。

## 当前阶段

- 已经明确收敛到单人私有研发自动化闭环。
- 产品主入口从 Ask-first 修正为 Dev Run-first。
- 当前代码已有 owner 登录、raw、ingest、wiki、ask 等基础实现。
- Ask writeback、claim 证据状态、usage、stale/conflict 信号已经证明“回答沉淀”链路可行。
- 下一阶段重点是把 `/app` 从 Ask 页面升级为 Dev Run Console，并把 Ask 迁移为 `/app/ask` 下的 Context Ask。
- MCP / CLI 接入的优先目标，是让外部 Agent 能取 Context Pack、提交 Deposit，并逐步参与 Dev Run 事件流。
- 后续自动化主线是技术方案、技术评审、编码准备、测试设计、修复分析、问题复盘和知识沉淀的技能链。

## 一句话总结

Inkdesk 的最终目标，是成为一套长期可用的 AI 研发自动化运行时。

Inkdesk 的最终形态，是一个单人私有、vault-first、可编排、可审阅、可追溯、可持续沉淀、可被 Agent 调用的 AI R&D Automation Runtime。
