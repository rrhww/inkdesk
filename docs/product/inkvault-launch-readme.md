# Inkvault，正在把 AI 回答变成长期可用的知识底座

> 不是又一个 AI 笔记软件，而是一套让用户“一键沉淀回答”、让 Agent 共享长期知识的私有知识运行时。

很多 AI 产品已经很会回答问题了。

但真正麻烦的，从来不是让 AI 回一句话。

而是你手上的网页、PDF、旧笔记、零散摘录、临时追问，怎么才能不在下一轮工作里重新散掉。你今天看过的材料，明天还能不能接着追问。你上个月做过的判断，三个月后还能不能回到原始来源重新核对。你积累下来的知识，到底是在增长，还是只是被一次次聊天窗口消耗掉。

Inkvault 想解决的，就是这个问题。

它当前不是一个公开发布平台，也不是一个什么都想装进去的超级工作台。它更像一个为单个 owner 服务的私有知识底座，把原始材料、AI 回答、长期知识和外部 Agent 工作过程放进同一条可沉淀闭环里。

它的最终目标，不是让 AI 替你写一个看起来很像答案的答案。

而是让你真正拥有一套可以长期沉淀、持续复用、反复追溯的个人知识运行时。

## 01 Inkvault 的目标，不是替你记笔记

传统笔记软件解决的是记录问题。

但研究工作的难点，往往不是记下来，而是沉淀不下来。

资料越来越多，结论越来越碎，聊天记录越来越长，最后真正能留下来的东西却很少。很多系统都在强调采集、强调生成、强调自动化，但真正缺少的是一套稳定的知识进入机制。

Inkvault 要做的，不是把所有输入都直接变成最终结论，也不是让用户手工维护一套复杂的知识治理后台。

它更关心的是：当 AI 给出一个有价值回答时，用户能不能只点一下“沉淀”，系统就自动判断留下什么、放到哪里、需要补哪条证据、是否和旧判断冲突。

它解决的不是单次写作效率，而是长期判断能力和 Agent 的长期上下文能力。

## 02 主形态：Ask，然后沉淀

Inkvault 的用户主路径应该非常短：

`Ask -> 沉淀`

用户在 Ask 中得到回答，每个回答都提供一个“沉淀这次回答”的入口。

按下沉淀按钮后，用户不需要理解 claim、topic、health、ingest、lint 这些内部概念。主 Agent 会接管当前问题、回答、引用和上下文，再触发专项子 Agent 完成后台沉淀：

- Insight Extractor 提取可沉淀判断
- Evidence Binder 绑定 raw / wiki / web source 证据
- Topic Router 判断新建 topic、patch 旧 topic 或生成 open question
- Conflict Checker 检查和旧 claim 是否冲突
- Patch Writer 生成 vault/wiki patch
- Quality Gate 判断自动写入、进入审阅还是要求用户裁决

只有当系统发现冲突、证据不足或高风险覆盖旧判断时，才用一张轻量卡片请用户确认。

这才是产品体验上的关键：复杂逻辑在后台，用户只负责判断“这个回答值得留下吗”。

## 03 私有、可审阅、可追溯，不是附加项

对于研究型工作和 Agent 工作来说，AI 能不能真正用起来，不只取决于它是否聪明，更取决于它是否可靠。

所以 Inkvault 从一开始就把几件事放在核心位置。

第一，它是单人私有的。

当前产品默认服务对象只有 owner 自己，没有公开阅读面，也没有多人协作角色。系统首先要做稳的，是你自己的研究闭环，而不是对外展示。

第二，它是审阅优先的。

AI 可以提出知识页草案，可以提出补丁建议，也可以把 Ask 阶段里有价值的发现转成 writeback 提案，但它不能静默改写正式知识。进入 wiki 的内容，必须经过 owner 接受。

第三，它是来源可回溯的。

每一条被接受的知识，都应该能回到原始材料，回到具体的网页、PDF、文本摘录和上下文。对研究系统来说，这不是锦上添花，而是底线。

## 04 从 raw 到 ask，Inkvault 的内部路径已经很明确

Inkvault 当前的内部闭环非常清楚。

`raw -> ingest -> wiki -> ask`

`raw/` 保存网页、PDF、迁移进来的旧笔记和其他原始材料。

`ingest` 是 AI 提案和人工审阅的中间层。系统先理解材料，生成待审阅提案，再由 owner 决定接受还是拒绝。

`wiki/` 保存被接受后的知识页。这里不是临时聊天记录，而是长期记忆区，承接 current understanding、claims、questions 和 sources。

`ask` 则是后续研究和沉淀的入口。系统会优先基于已有 wiki 回答，再按需补充 raw，并在用户点击沉淀后把新的高价值发现送入后台沉淀流程。

这条链路的意义，不在于把功能拆成四个名字。

而在于它把原始输入、AI 回答、用户沉淀意图和长期知识第一次真正串成了一条完整路径。

## 05 Vault-first，让知识真正留在你手里

Inkvault 的另一个关键选择，是 vault-first。

系统可以用数据库做索引、做队列、做缓存、做工作流状态管理，但长期真相不能只存在数据库里，更不能只存在一段大模型对话里。

被接受的知识，最终要能回到 vault markdown。

这意味着，哪怕索引层丢了，哪怕读模型需要重建，长期知识本身依然是可恢复、可迁移、可备份的。对于一个希望长期使用的私人研究系统来说，这种边界非常重要。

它形成的不是一个被产品界面绑住的内容池，而是一套 owner 真正持有的数据资产。

## 06 Inkvault 不只服务自己的 App，也服务外部 Agent

Inkvault 的长期目标，是成为外部 Agent 的知识底座。

用户使用 Claude Code、Codex、Cursor 或其他 Agent 时，也应该能接入同一个长期知识系统：

- 任务开始前，Agent 调用 Inkvault 获取相关产品记忆、历史判断和开放问题。
- 任务结束后，用户要求沉淀，Agent 把回答、方案或选中片段交给 Inkvault。
- Inkvault 复用同一套后台沉淀流水线，不要求外部 Agent 理解内部知识治理逻辑。

所以 Inkvault 未来会提供两类接入：

- `context_pack`：任务前取上下文
- `deposit`：任务后沉淀结果

这让 Inkvault 从一个私有研究工作台，扩展成所有 Agent 都可以调用的长期记忆层。

## 07 Ask，不只是聊天入口

很多 AI 产品把问答当成终点。

但在 Inkvault 里，Ask 更像研究工作的继续。

用户不是来和一个泛化助手闲聊，而是围绕已经沉淀下来的知识继续追问，识别缺口，补足证据，形成下一轮判断。

系统优先读取 wiki，再回退 raw。必要时可以在 owner 明确允许下短暂联网补料，但新的外部材料也不能直接绕过系统边界，而是要重新回到 raw 和 ingest 体系里。

这样一来，Ask 不再只是一次性消费答案的界面。

它变成了长期知识系统里的推进器，让研究可以在已有积累之上持续前进，也让有价值回答可以直接进入长期知识沉淀。

## 08 Inkvault 的最终形态，是一个 Agent Knowledge Runtime

如果只看当前阶段，Inkvault 已经是一套收敛得很明确的 MVP。

它正在稳定一条单人私有、vault-first、以 `raw -> ingest -> wiki -> ask` 为内部路径的研究闭环。

但如果往最终形态去看，它想成为的并不是一个窄功能工具。

它更像一个 Agent Knowledge Runtime。

在这个系统里，原始材料有稳定入口，AI 负责提出理解和提案，人负责判断是否沉淀，长期知识被持续整理进 wiki，而 Ask 和外部 Agent 负责把这些记忆重新调动起来，继续参与新的研究、写作、编码和决策。

到那时，Inkvault 承担的就不只是资料管理。

它会成为一个 owner 的长期知识底盘，一个可以持续积累判断、保留上下文、支持复盘、追问和外部 Agent 协作的私人思考中枢。

这也是项目最终目标最准确的表达。

不是再做一个会生成内容的 AI 工具。

而是做一套真正属于你自己的、可沉淀、可追溯、可持续演化、可被 Agent 调用的知识底座。
