# 产品路线图

## 目标

这份路线图描述 Inkdesk 如何按文章《AI研发自动化：Wiki知识库+技能包》的开发流程，从当前私有研究闭环演进成一个 LLM-Wiki + Skill Workbench + Agent Harness。

主开发顺序固定为：

```text
Sources -> LLM-Wiki -> Schema -> Skills -> Agent Runtime -> Evaluation -> Harness
```

完整执行计划见 [开发计划总指南](../development/plans/开发计划总指南.md)。
文章与项目的对应关系见 [项目映射](../references/AI研发自动化：Wiki知识库+技能包-项目映射.md)。

用户主路径保持极简，但入口应是研发任务，而不是 Ask：

```text
PRD / bug / 改造任务
  -> Dev Run
  -> 阶段输出
  -> Review / Deposit
```

当前系统内部已经具备前两层的雏形：

```text
raw -> ingest -> wiki -> ask -> health -> ingest
```

外部 Agent 通过 MCP / CLI / Skill 接入同一知识底座：

```text
context_pack -> agent work -> deposit
```

这里需要区分两层含义：

- 当前要做的是“薄 MCP 接入层”，把现有知识底座暴露给外部 Agent
- 阶段六要做的是“高自治 Harness”，包含 staged orchestration、gates、rollback 和更高自动化门禁

前者可以前置，后者仍然放在后期

## 阶段一：Sources 与 LLM-Wiki 对齐

### 阶段目标

- 把 `raw -> ingest -> wiki -> ask` 主路径做稳，并明确对应文章里的 `Sources -> LLM-Wiki`
- 让 Ask 回答能稳定回流成可审阅的 wiki 提案
- 增加第一批 Wiki Health 检查，确保 LLM-Wiki 不只是文档堆积
- 把 Web 首页从 Ask-first 修正为 Dev Run Console，让知识底座服务 AI 研发自动化主线

### 核心能力

- owner 登录
- raw 导入
- ingest 审阅
- wiki 页面沉淀
- Ask 引用与 writeback
- Ask deposit：整条回答与选中片段都能进入 ingest
- Wiki Health：frontmatter、来源引用、断链、孤页等基础检查
- Dev Run Console：PRD / bug / 改造任务入口、阶段轨道、待确认输出、Context Ask 入口

### 完成标志

- 闭环可本地跑通
- 代码、测试和文档不再混入旧 public / plans / note editor 语义
- 用户能从 Dev Run 或 Context Ask 触发沉淀，但 AI 仍不能静默改写 wiki
- vault 中的 wiki 文件能被普通 Markdown 工具理解
- `/app` 不再呈现为大问答框，而是呈现研发任务运行台

### 当前进度

截至 2026-05-22：

- [x] owner 登录、本地 `/app` 工作区、raw、ingest、wiki、ask
- [x] Ask writeback、claim 证据状态、usage、stale/conflict 信号
- [x] 本地 Docker 栈：Next.js、FastAPI、PostgreSQL + pgvector
- [x] 后端 `python -m pytest` 与前端 `npm run typecheck`
- [x] 阶段一主体成立，产品重点转向 Dev Run-first 与 Context Ask 沉淀体验
- [ ] 每个 Context Ask 回答固定提供“沉淀这次回答”
- [ ] 支持完整回答与选中片段两种沉淀入口
- [ ] 基础 Wiki Health：缺 frontmatter、缺来源、断链、孤页
- [ ] `/app` 改为 Dev Run Console；`/app/ask` 保留为 Context Ask

### 当前执行任务

先执行“产品主入口修正 + Context Ask 沉淀流程”：`/app` 改为 Dev Run Console，`/app/ask` 保留为 Context Ask，并继续执行 [Context Ask 沉淀流程实施计划](../development/plans/2026-06-04-问答沉淀主流程实施计划.md)，同时准备薄 MCP 接入层。

这不是偏离文章流程，而是在补齐文章流程里的 LLM-Wiki 写入通道：

```text
dev run / context ask -> deposit -> ingest proposal -> review -> wiki patch
```

阶段一内允许增加一层受控外部入口：

```text
external agent -> context_pack/search -> agent work -> deposit -> ingest review
```

它的边界是：

- 不绕过 ingest / review
- 不直接写 accepted wiki
- 不等同于完整 Harness

## 阶段二：Schema 层

### 阶段目标

- 在 vault 中加入 agent-facing schema 文件
- 明确 wiki、source、ingest、ask、health 的维护规则
- 让 Inkdesk、Codex、Claude Code 或其他 Agent 都能读取同一套维护协议

### 核心能力

- 懒创建 `schema/`、`skills/`、`evals/`、`runs/`
- `schema/vault-layout.md`
- `schema/wiki-page-template.md`
- `schema/source-citation-rules.md`
- `schema/ingest-proposal-rules.md`
- `schema/ask-answer-rules.md`
- `schema/wiki-health-rules.md`

### 完成标志

- 新 vault 初始化后自动具备 schema 文件
- schema 文件可在 Obsidian 或普通编辑器中阅读
- 后端测试证明 schema 初始化幂等

## 阶段三：Skills 与 Skill Workbench

### 阶段目标

- 把重复工作沉淀为显式 skill 文件
- 在 UI 中浏览、审阅、运行和导出 skills
- Skill 输出仍通过 proposal / review 进入 wiki
- 将 Skills 从知识管理类扩展到研发流程类：技术方案、技术评审、coding、测试准备、测试修复、问题排查

### 核心能力

- 初始知识 skills：Ingest Source、Patch Wiki Page、Answer From Wiki、Create Research Brief、Run Wiki Health Check、Extract Reusable Insight From Chat
- 初始研发 skills：Tech Solution、Tech Review、Coding、Test Prep、Test Fix、Problem Solve、Skill Router
- Skill 列表页
- Skill 详情页
- Skill 文件导出
- Skill 输出转 review proposal

### 完成标志

- skill 可作为文件检查，也可从 UI 启动
- skill 具备输入、上下文、输出契约、安全规则和验证要求
- skill 不能直接静默写 wiki

## 阶段四：Agent Runtime Runs

### 阶段目标

- 为 skill 执行建立可追踪 run 记录
- 支持 LangGraph、本地执行器或未来外部 Agent runtime
- 将运行输出、错误、提案和审阅状态记录下来

### 核心能力

- run records
- `runs/` vault 输出约定
- Agent Run Console
- 记录输入、resolved context、runtime、output、error、review state
- wiki-changing output 统一转 review proposal

### 完成标志

- 同一个 skill run 可以复现输入、上下文和输出
- runtime 失败不污染正式 wiki
- 所有 wiki 修改仍经过 review

## 阶段五：Evaluation

### 阶段目标

- 在提高自治度前加入 golden tasks 与 rubrics
- 让 schema、skills、wiki、runtime 变更可评测
- 阻止 blind prompt drift

### 核心能力

- `evals/golden-tasks/`
- rubric 文件
- evaluation run records
- isolated evaluation，不修改 canonical wiki
- skill/wiki/schema 版本对比
- promotion gate

### 完成标志

- evaluation run 可重复
- 失败评测不能改变 accepted wiki
- skill 或 schema 变更可在 promotion 前比较

## 阶段六：Harness 与外部 Agent 接入

### 阶段目标

- 在评测和 review 足够可靠后，增加多步骤 harness
- 接入 Claude Code、Codex、Cursor 等外部 Agent
- 为外部 Agent 提供任务前 context pack 与任务后 deposit
- 将 PRD -> 技术方案 -> 技术评审 -> coding -> test -> fix -> deposit 串成受控 Dev Run

### 核心能力

- staged run orchestration
- gates between stages
- retry / rollback records
- manual approval checkpoints
- `inkdesk context` CLI
- `inkdesk deposit` CLI
- `inkdesk-mcp`：`context_pack`、`deposit_answer`、`deposit_selection`、`search`、`open_questions`
- Dev Run orchestration：PRD intake、phase state、stage output、approval gate、rollback note
- Codex Skill / AGENTS.md 接入说明
- Claude Code MCP / slash command 接入说明

### 完成标志

- 外部 Agent 能任务前取上下文
- 外部 Agent 能任务后提交沉淀
- 高自治度必须 opt-in，并且有 evaluation history 支撑

## 长期方向

长期仍可以向更大的“个人主系统”扩展，但前提是：

- Dev Run Console 已经成为主入口
- 当前私有研究闭环已经长期稳定
- 回答沉淀能力已经足够顺滑并能服务 Dev Run
- 外部 Agent 接入不破坏用户体验和知识边界
- 文档、数据边界和 AI 审阅机制没有继续漂移
- 知识层与行为层已经清晰分离：`raw/wiki` 承担长期真相，`ingest/ask/lint/writeback` 承担可审计的操作协议
- Agent 的能力增长建立在可控自进化上，而不是不可追踪的自治行为

在那之前，不恢复旧的 public / plans 产品叙事。
