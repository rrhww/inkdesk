# 备份与恢复

## 目标

这份文档用于明确 Inkvault 当前私有 LLM Wiki 的数据保护策略，保证即使服务器故障或误操作，也能恢复到可继续使用的状态。

## 备份范围

MVP 需要保护的核心资产包括：

- PostgreSQL 数据库
- vault 目录中的 `raw/` 与 `wiki/` Markdown
- 部署配置与环境变量清单

## 一、数据库备份

### 备份方式

- 使用 `pg_dump` 进行逻辑备份
- 备份文件定时保存到对象存储或服务器安全目录

### 备份频率

- 每日一次全量备份

### 保留策略

- 保留 7 天日备份
- 保留 4 周周备份

## 二、Vault 目录备份

### 备份对象

- `raw/`
- `wiki/`
- vault 根目录中的辅助说明文件（如 `AGENTS.md`）

### 策略

- 使用文件级备份或卷级快照
- 备份时保留目录结构和文件修改时间
- 允许按天打包，也允许同步到对象存储或远端磁盘

## 三、恢复目标

MVP 的最小恢复目标为：

- 恢复到最新可用备份的数据库状态
- 恢复 vault 中的 `raw` 与 `wiki` 文件
- 恢复主系统可登录
- 恢复 `raw -> ingest -> wiki -> ask` 主路径可用

## 四、数据库恢复步骤

### 标准步骤

1. 准备新数据库实例或清空目标库
2. 导入最新可用备份
3. 校验表结构和数据完整性
4. 启动应用并验证登录、搜索、发布是否可用
4. 启动应用并验证登录、raw、ingest、wiki、ask 是否可用

## 五、Vault 恢复步骤

### 标准步骤

1. 准备新的 vault 挂载目录
2. 恢复 `raw/` 与 `wiki/` 文件
3. 确认后端的 `INKVAULT_VAULT_ROOT` 指向恢复后的目录
4. 启动后端，让数据库索引与 vault 内容重新对齐

## 六、恢复演练要求

- 首次正式上线前至少完成一次完整恢复演练
- 演练结果要记录恢复耗时和失败点
- 只有演练通过后，备份策略才算真正成立

## 七、风险边界

当前 MVP 备份策略仍有以下边界：

- 不保证秒级恢复
- 不保证跨地域容灾
- 不保证附件系统长期兼容，因为附件不在当前主路径内

对于当前单人私有部署，这个强度已经够用；等后续数据规模和自动化复杂度继续上来，再升级。
