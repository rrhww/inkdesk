## 四、Inkdesk 作为外部 Agent 的研发自动化底座

Inkdesk 的长期形态不是只在自己的 Web App 里回答问题，而是作为 Claude Code、Codex、Cursor 等外部 Agent 的研发上下文、阶段记录和沉淀底座。

外部 Agent 接入 Inkdesk 时，第一版只需要理解四个动作：

```text
context_pack -> 任务前获取相关长期知识、产品约束、历史方案和 open questions
run_start -> 创建或绑定一条 Dev Run
run_event -> 把阶段产物、测试结果、风险和阻塞点写回 Dev Run
deposit -> 任务后把高价值方案、判断或片段交给 Inkdesk 沉淀
```

外部 Agent 不应该直接写 `wiki/`，也不需要理解 `claim / ingest / health / lint` 的完整内部协议。它们只把上下文和沉淀请求交给 Inkdesk，由 Inkdesk 的主 Agent 和专项子 Agent 完成后续逻辑。

### MCP 的双重角色

MCP 工具同时服务于两条路径：

```text
路径一：自动（Harness 驱动）
  Inkdesk Harness
    → 调用外部 Agent MCP（传入 context_pack、任务 briefing）
    → 外部 Agent 执行
    → 外部 Agent 通过 MCP 调用 run_event / deposit 回写
  用户不需要打开外部 Agent

路径二：手动（用户驱动）
  人在外部 Agent 里做探索性工作，不在正式 Dev Run 里
    → 手动调用 context_pack 获取知识底座
    → 干完活手动调用 deposit 沉淀
  用户直接在外部 Agent 里操作
```

两条路径共用同一套 MCP 工具，区别只是调用方：

| 工具 | 自动路径（Harness → Agent） | 手动路径（用户 → Agent） |
|------|---------------------------|-------------------------|
| `context_pack` | Harness 组装 briefing 时调用 | 用户做探索前自己拉上下文 |
| `run_start` | Harness 创建 Dev Run | 用户在 Agent 里开始一个任务 |
| `run_event` | Agent 写完代码自动回写结果 | 一般不手动触发 |
| `deposit` | Harness 在任务结束时沉淀关键产物 | 用户干完活手动沉淀 |
| `search` | Harness 准备上下文 | 用户随手搜索 |
| `health_check` | Harness 在 gate 处自动检查 | 用户随手看知识库状态 |

对 Web 端用户而言，外部 Agent 在自动路径中是 Harness 调用的后端执行服务——用户在 Inkdesk Web 创建 Dev Run、审阅阶段输出，不需要打开 Claude Code。手动路径保留给探索性工作和未接入 Harness 的场景。

### 对外接口草案

第一版建议同时提供 CLI 与 MCP server。

CLI：

```bash
inkdesk context --task "<当前任务>" --repo "<仓库路径>" --max-tokens 3000
inkdesk run start --task "<PRD / bug / 改造任务>" --repo "<仓库路径>"
inkdesk run event --run-id "<run-id>" --type "tech_solution_created" --file solution.md
inkdesk deposit --question "<用户问题>" --answer-file answer.md --repo "<仓库路径>"
inkdesk deposit-selection --text-file selection.md --reason "<为什么值得沉淀>"
```

MCP tools：

- `inkdesk.context_pack(task, repo, files, max_tokens)`：返回外部 Agent 可直接阅读的短上下文包。
- `inkdesk.run_start(task, repo, metadata)`：创建或绑定一条 Dev Run。
- `inkdesk.run_event(run_id, type, payload)`：记录阶段产物、测试结果、风险、阻塞和审阅事件。
- `inkdesk.search(query, scope, limit)`：搜索长期知识。
- `inkdesk.deposit_answer(question, answer, citations, source_context)`：沉淀一段完整回答。
- `inkdesk.deposit_selection(text, reason, source_context)`：沉淀用户选中的片段。
- `inkdesk.open_questions(scope)`：返回相关未解决问题。
- `inkdesk.decision_history(topic)`：返回某个主题的稳定判断和变更记录。

### 外部 Agent 使用协议

外部 Agent 在任务开始时：

1. 判断任务是否依赖长期知识、产品方向、历史决策或仓库约定。
2. 调用 `run_start` 创建或绑定一条 Dev Run。
3. 调用 `context_pack` 获取短上下文。
4. 只把返回结果作为上下文，不自行改写 Inkdesk 长期知识。

外部 Agent 在任务推进中：

1. 阶段性产出技术方案、评审意见、coding 计划、测试结果或修复记录。
2. 调用 `run_event` 把阶段产物写回 Dev Run。
3. 当 Inkdesk 返回 gate / conflict / missing evidence 时，停下来等待 owner 或上层 Harness 裁决。

外部 Agent 在任务结束时：

1. 如果回答中有可长期复用的判断、方案、设计取舍或复盘，提示用户是否沉淀。
2. 用户确认后调用 `deposit_answer` 或 `deposit_selection`。
3. Inkdesk 后台完成提取、证据绑定、topic 路由、冲突检测和质量门控。

### Codex 接入形态

Codex 推荐三层接入：

- MCP server：提供 `context_pack`、`run_start`、`run_event` 和 `deposit` 工具。
- Codex Skill：说明什么时候调用 Inkdesk，什么时候建议用户沉淀。
- 项目 `AGENTS.md`：声明本项目长期知识由 Inkdesk 管理。

项目级提示建议：

```text
When the task depends on product memory, project history, or durable decisions, start or bind an Inkdesk Dev Run and request a context pack before proposing changes.
After producing durable product, architecture, workflow, testing, or debugging decisions, record a run event and ask the user whether to deposit the result into Inkdesk.
Never write long-term knowledge directly to repo docs unless the user explicitly asks; use Inkdesk deposit first.
```

### Claude Code 接入形态

Claude Code 推荐通过 MCP server 和 slash command 接入：

- `/inkdesk-context`：获取当前任务相关上下文。
- `/deposit`：把当前回答或选中内容提交给 Inkdesk 沉淀。

用户体验应该保持简单：

```text
Claude Code 开始任务
调用 /inkdesk-context 绑定 Dev Run 并获取上下文
阶段产物写回 Dev Run
用户输入 /deposit
Inkdesk 后台沉淀并等待审阅
必要时返回轻量冲突裁决卡片
```

## 六、哪些是开发期依赖

以下属于开发期工具，不是线上运行依赖：

- Figma
- GPT-5
- GitHub MCP
- Playwright MCP

这些工具用于：

- 设计探索

- 页面验证

## 七、哪些不进入生产

生产环境不需要把以下内容作为线上服务部署：

- MCP 服务
- Figma
- GitHub 相关辅助工具
- 本地开发辅助脚本

生产环境只需要承载：

- 前端应用
- Python 主后端
- PostgreSQL
- Nginx
- Vault 存储目录

注意：Inkdesk 自己未来提供的 MCP server / CLI bridge 属于产品接入层，不等同于开发期 MCP 工具。开发期 MCP 不进生产；产品化 Inkdesk connector 可以作为独立本地服务或桌面侧桥接器交付。

## 八、工具链原则

- 优先选择能服务 `PRD / 研发任务 -> Dev Run -> Review / Deposit` 主路径的工具
- 不把开发辅助工具强行带进生产环境
- 工具链服务于产品边界，不反过来绑架架构
- 外部 Agent 接入只暴露简单协议：取上下文、记录阶段事件、沉淀结果
- 复杂知识治理保留在 Inkdesk 内部，不泄漏给外部 Agent
