# API 接口规范

## 目标

固定当前 vault-first AI R&D Automation Runtime 的后端接口边界。

当前接口只服务以下主路径：

- Dev Run Console 首页聚合
- `raw` 原始材料导入与读取
- `ingest` 提案审阅
- `wiki` 知识页读取
- `ask` Context Ask 与 deposit / writeback 提案
- `runs` Dev Run 创建、读取、阶段事件、推进与取消
- `deposits` 知识沉淀统一入口
- `compile` 异步编译流水线
- `health` 知识库结构健康检查
- `vault` 知识库初始化与状态

Inkdesk 采用**单实例、默认 workspace、无登录**架构。所有 API 不需要认证 Cookie，直接通过 Next.js 同源 `/api/**` 访问。`workspace_id` 仅作为数据归属字段，不代表多租户。

目标用户主路径是 `PRD / 研发任务 -> Dev Run -> Review / Deposit`。

## 一、健康检查

### `GET /health`

- 目标：基础存活探针

### `GET /actuator/health`

- 目标：暴露服务状态与 retrieval 健康信息

## 二、首页聚合接口

### `GET /api/admin/home`

- 目标：返回 `/app` 首页所需的聚合数据
- 返回重点：
  - summary
  - focusTopic
  - recentSources
  - pendingReviews
  - suggestedQuestions

### `GET /api/runs`

- 目标：读取 Dev Run 列表
- 返回：`DevRunSummaryResponse[]`

### `POST /api/runs`

- 目标：从 PRD / bug / 改造目标创建 Dev Run
- 请求：`CreateDevRunRequest { type, title, goal, repoContext? }`
- 返回：`DevRunResponse` (status 201)

### `GET /api/runs/{run_id}`

- 目标：读取单个 Dev Run 的阶段状态、输出和事件

### `POST /api/runs/{run_id}/events`

- 目标：记录外部 Agent 产生的阶段事件
- 请求：`AddRunEventRequest { stage?, eventType, payload }`

### `POST /api/runs/{run_id}/advance`

- 目标：推进或完成 Dev Run
- 请求：`AdvanceRunRequest { action: "approve" | "complete" }`
- 约束：
  - `approve` 仅在 `stage_status == "awaiting_review"` 时允许
  - `complete` 仅在 `current_stage == "deposit"` 且 `stage_status == "awaiting_review"` 时允许

### `POST /api/runs/{run_id}/cancel`

- 目标：取消 Dev Run

## 三、Raw 接口

### `GET /api/raw`

- 目标：读取 raw vault 中已索引的来源列表

### `POST /api/raw`

- 目标：直接创建一条文本来源

### `POST /api/raw/web`

- 目标：从网页 URL 导入来源

### `POST /api/raw/pdf`

- 目标：上传 PDF 并导入来源

## 四、Ingest 接口

### `GET /api/ingest`

- 目标：读取待审阅提案列表

### `GET /api/ingest/{review_id}`

- 目标：读取单条提案详情

### `POST /api/ingest/{review_id}/accept`

- 目标：接受提案并把变更写入 wiki

### `POST /api/ingest/{review_id}/reject`

- 目标：拒绝提案并保留审阅记录

## 五、Wiki 接口

### `GET /api/wiki`

- 目标：读取知识页列表

### `GET /api/wiki/{topic_id}`

- 目标：读取单个知识页详情
- 返回重点：
  - summary
  - currentUnderstanding
  - keyClaims
  - openQuestions
  - sources
  - researchThread

## 六、Ask 接口

### `POST /api/ask`

- 目标：基于 wiki first、raw second 执行研究问答
- 支持模式：`vault`、`vault_plus_web`

### `GET /api/ask/{ask_turn_id}`

- 目标：读取单轮 Ask 结果

### `GET /api/ask/{ask_turn_id}/thread`

- 目标：读取当前 Ask 所在追问链

### `POST /api/ask/{ask_turn_id}/writeback`

- 目标：把当前 Ask 结果转成一条新的 ingest 提案（委托 `DepositService`）

### `GET /api/ask/briefing`

- 目标：获取 Ask 判断面板摘要

## 七、Deposit 接口

### `POST /api/deposits`

- 目标：知识沉淀统一入口，支持 `answer`、`selection`、`stage_output` 三种来源
- 幂等键：`deposit|{source}|{askTurnId}|{runId}|{understanding}|{selectedText}|{rationale}` 的 content hash
- 请求：`DepositRequest { source, runId?, askTurnId?, stage?, payload }`
- 返回：`DepositResponse { reviewId, status, source, isNew }` (201 新建 / 200 幂等重复)

## 八、编译流水线接口

### `POST /api/raw/{source_id}/compile`

- 目标：对 raw 来源发起异步编译（Insight → Evidence → Router → Conflict → Patch）

### `GET /api/compile/queue`

- 目标：读取编译队列（最近 50 条）

### `GET /api/compile/{task_id}`

- 目标：读取单个编译任务详情（含步骤状态）

### `POST /api/compile/{task_id}/retry`

- 目标：重试失败的编译任务

## 九、健康检查接口

### `GET /api/health`

- 目标：返回知识库结构健康摘要（断链、孤页、缺 frontmatter、缺来源引用）

## 十、Vault 接口

### `GET /api/vault/status`

- 目标：读取知识库初始化状态

### `POST /api/vault/initialize`

- 目标：初始化知识库（code / general_knowledge）

## 十一、MCP Server 入口

### Streamable HTTP `{base}/mcp`

- 提供 4 个工具：`context_pack`、`search`、`deposit`、`health_check`
- 传输安全：仅允许 `localhost` / `127.0.0.1`，启用 DNS rebinding 防护和 Host header 校验
- 使用默认 workspace，无需认证

## 关键边界

- 单实例、默认 workspace（slug: `inkdesk`）、无登录认证
- API 通过 Next.js rewrites 同源代理（`/api/:path*` → 后端）
- AI 不能直接改写 wiki，只能先生成可审阅提案
- 外部 Agent 不能直接写 vault，只能调用 context / deposit 协议
- 未配置 API 基地址时，前端允许回退到本地 fixtures，但后端契约仍以本文件为准
