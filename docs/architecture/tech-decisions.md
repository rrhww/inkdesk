# 技术决策

## 1. 仓库结构

- 继续采用单仓库结构
- `web/` 承担公开输出入口与主系统前端
- `server/` 承担认证、内容、计划与发布接口

## 2. 前端框架

- 继续使用 `Next.js App Router`
- 原因：同一前端可同时承接公开输出入口和主系统，并方便做根路由身份切换

## 3. 主系统路由前缀

- 私有主系统继续固定为 `/app`
- 原因：复用现有工程结构，降低重构成本

## 4. 根路由入口

- `/` 根据登录态决定渲染公开输出入口或跳转主系统
- 原因：对访客保持公开输出入口，对主人保持“一进来就是主系统”的体验

## 5. 登录模型

- 当前阶段使用真实 single-owner auth + `inkvault_owner_session` Cookie
- 登录标识固定为邮箱
- 原因：本地全栈闭环必须覆盖真实登录、登出与会话校验

## 6. 公开输出策略

- 公开输出入口固定为公开文章 + 项目分享
- 不做产品官网式入口
- 不暴露主系统入口

## 7. 主系统首页策略

- `/app` 固定为 Agent 控制台
- 原因：让 Agent 成为进入主系统后的第一屏

## 8. 笔记系统策略

- 笔记继续使用 Markdown 作为底层事实来源
- 原因：保证长期可迁移与可被后续 Agent 消费

## 9. 任务 / 计划策略

- 任务与计划作为一级模块进入主系统
- 原因：Inkvault 已从“存内容”转为“组织工作”

## 10. 发布策略

- 保留发布模块
- 降为次级模块
- 原因：公共表达仍重要，但不能继续主导产品叙事

## 11. 设置持久化策略

- 设置持久化到独立的 `workspace_settings` 表
- `security` 字段由后端派生，只读返回

## 12. 写操作策略

- 写操作统一走后端 REST 接口
- 前端统一通过 Next.js server action 或服务端 helper 调用
- 不在浏览器直接拼接 owner 写请求
