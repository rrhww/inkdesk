# 上线检查清单

## 本轮目标

本轮的放行标准不是"差不多能看"，而是私有 vault-first 本地全栈闭环成立。

Inkdesk 采用**单实例、默认 workspace、无登录**架构。进入 `/app` 无需认证。所有 API 通过 Next.js 同源 `/api/**` 代理到后端。

## 基础设施

- `docker compose` 能拉起 `PostgreSQL + MinIO`
- Python 主后端可连接本地 PostgreSQL（监听 `127.0.0.1:8080`）
- Next.js 通过 rewrites 同源代理到 Python 后端
- 默认启动会自动创建默认 workspace 与示例 research 数据

## 主系统链路

- `/app` 可直接访问（Dev Run Console），无需登录
- `/app` 首屏是 Dev Run 列表和创建入口
- Dev Run 详情页展示 6 段阶段轨道、事件和操作按钮
- `/app/raw`、`/app/ingest`、`/app/wiki`、`/app/ask` 可正常访问
- `/app/ask` 定位为 Context Ask（不再是首页）
- `/app/compile` 编译流水线可正常访问
- `/app/health` 知识库健康页可正常访问
- 不存在的资源返回 404，不重定向到登录页

## 研究闭环链路

- 原始材料可以进入 `raw/`
- ingest 提案不会静默写入 wiki
- 接受提案后才会创建或更新 wiki 页面
- wiki 页面能看到 understanding、claims、questions、sources
- wiki claim 能看到 `supported / partial / unsupported`、`证据 N 条`、`最近验证`
- Ask 返回 wiki/source citation
- Ask 之后可拉取 ask-scoped briefing，并刷新判断面板
- Ask 保持 vault-first / explicit web assist 纪律
- briefing 失败时 `/app` 仍能返回 deterministic 判断，不会整页失效

## Dev Run 链路

- 支持 PRD、bug、改造三种任务类型
- 支持创建、列表、详情和取消
- 六段阶段轨道初始化正确
- 阶段事件追加并更新聚合状态
- `approve` 仅在 `stage_status == "awaiting_review"` 时允许
- `complete` 仅在 `current_stage == "deposit"` 且 `stage_status == "awaiting_review"` 时允许
- 非法状态转换返回 409

## Deposit 链路

- 完整回答、选中片段和阶段输出统一走 Deposit
- Deposit 幂等：相同内容不创建重复提案
- 所有 deposit 进入 ingest 审阅队列
- `writeback` 内部转发到 Deposit 服务

## 自动化验证

- `cd server && python -m pytest` 通过
- `cd web && npm test` 通过
- `cd web && npm run typecheck` 通过
- `cd web && npm run lint` 通过
- `cd web && npm run build` 通过
- `cd web && npm run e2e` 通过
- 如果本地全栈环境已启动，`cd web && npm run e2e:fullstack` 通过

## 阻塞项判定

以下任一项不满足，都不能标记为放行：

- `/app` 无法直接访问或首屏无法使用
- raw / ingest / wiki / ask 任一环节失败
- Dev Run 状态机边界不生效（非法状态转换未被拒绝）
- Ask 后判断面板没有刷新为本轮 Ask 结果
- 提案未经确认直接写入 wiki
- Ask 缺少引用来源
- briefing 失败导致主入口不可用
- Deposit 重复提交产生重复提案
- 测试、类型检查、lint 或构建存在失败
