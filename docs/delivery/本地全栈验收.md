# 本地全栈验收

## 目标

这份文档用于验收 Inkdesk 当前已实现的 Ask / vault-first / Dev Run 本地闭环可用性。

Inkdesk 采用**单实例、默认 workspace、无登录**架构。所有页面无需认证即可直接访问。API 通过 Next.js 同源 `/api/**` 代理到后端。

当前闭环定义为：

- 直接访问 `/app`（Dev Run Console）
- `raw -> ingest -> wiki -> ask` 主路径可用
- Ask 前后都能看到当前知识缺口与下一步动作
- Dev Run 可创建、追踪、推进和完成
- compile / health 页面可访问
- 本地 `PostgreSQL + MinIO + Python 主后端 + Next.js` 协同可用

## 前置条件

1. 已复制环境模板

```powershell
Copy-Item infra/.env.example infra/.env
Copy-Item web/.env.local.example web/.env.local
```

2. 本机已安装 `Docker`、`Python 3.12+`、`Node.js 20+`

## 启动顺序

### 1. 启动基础设施

```powershell
docker compose --env-file infra/.env -f infra/docker-compose.yml up -d
```

### 2. 启动后端

```powershell
cd server
python -m pip install -e .[dev]
python -m uvicorn inkdesk_server.main:app --host 127.0.0.1 --port 8080
```

### 3. 启动前端

```powershell
cd web
npm install
npm run dev
```

## 默认 workspace

数据库为空时，后端会自动创建默认 workspace（slug: `inkdesk`）、legacy note 和示例 research 数据。

## 手动验收步骤

1. 访问 `http://localhost:3000/app`，确认 Dev Run Console 首页可见
2. 在 `/app` 验证 Dev Run 列表、创建入口、suggested questions 可见
3. 创建一条 Dev Run（PRD 类型），确认出现在列表中
4. 打开新建的 Dev Run 详情页，确认阶段轨道（6 段）可见
5. 在 Dev Run 详情页点击"Ask 追问"，跳转 Context Ask 页面
6. 执行一次 Ask，确认返回引用来源、知识缺口和下一步动作
7. 在回答中点击"沉淀到 wiki"，确认跳转审阅队列
8. 打开 `/app/ingest`，确认提案已出现，可接受或拒绝
9. 接受一条提案，确认 wiki 页面已创建/更新
10. 打开 `/app/wiki` 与 `/app/wiki/[id]`，验证 `Current Understanding`、`Key Claims`、`Open Questions`、`Sources` 可见
11. 在 wiki 详情页确认 claim 治理信息可见：`supported / partial / unsupported`、`最近验证`、`证据 N 条`
12. 打开 `/app/compile`，确认编译流水线页面可见
13. 打开 `/app/health`，确认知识库健康页面可见
14. 回到 Dev Run 详情页，推进各阶段（approve），确认状态变更
15. 推进完所有 6 阶段后执行 `complete`，确认 Run 标记为已完成

## 自动化验证

### 后端

```powershell
cd server
python -m pytest
```

### 前端单元 / 集成 / 构建

```powershell
cd web
npm test
npm run typecheck
npm run lint
npm run build
npm run e2e
```

### 真实后端全链路 E2E

```powershell
cd web
npm run e2e:fullstack
```

这个命令会先检查 `.env.local` 中的 API 地址，并等待 Python 后端健康检查就绪，再运行 `tests/e2e/local-fullstack.spec.ts`。

如果当前机器缺少 `Docker`、`PostgreSQL` 未监听，或 Python 后端尚未启动，脚本会输出对应的本地预检提示，帮助定位阻塞点。

如果当前机器没有完整本地全栈环境，这一项应记录为环境阻塞，而不是产品失败。

## 放行标准

只有以下条件同时满足，才能认为本地全栈闭环达成：

- 后端测试通过
- 前端测试通过
- 前端类型检查通过
- 前端 lint 通过
- 前端构建通过
- 默认 E2E 通过
- 全链路 E2E 通过，或明确记录当前机器的环境阻塞原因
- 手动验收没有发现 raw、ingest、wiki、ask、Dev Run 链路断点
- 当前 Ask 首屏与问后判断都能明确展示当前缺口和下一步动作
- claim 治理信息能在 wiki / ingest / Ask 判断面板中被直接识别
- Dev Run 可从创建推进到完成，阶段状态变更正确
- compile 和 health 页面可正常访问
