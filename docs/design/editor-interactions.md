# 编辑器交互说明

## 目标

Inkvault 的编辑器需要同时满足两件事：

- 使用体验接近语雀，降低写作阻力
- 底层始终以 Markdown 作为唯一事实来源

MVP 不追求 Notion 级块编辑，而是优先保证写作稳定、结构清楚、内容可迁移。

## 编辑区结构

笔记编辑页中的编辑器区域应包含：

- 标题输入区
- 正文编辑区
- 顶部工具栏
- 保存状态提示
- 发布操作区

## 支持的基础能力

- 标题
- 段落
- 粗体
- 斜体
- 删除线
- 无序列表
- 有序列表
- 任务列表
- 引用
- 代码块
- 行内代码
- 分割线
- 表格
- 链接
- 图片

## 自动保存策略

- 用户输入后进入“未保存”状态
- 输入停顿后自动触发保存
- 保存中显示明确状态
- 保存成功显示最近更新时间或保存成功提示
- 保存失败时给出可感知提示，但不打断继续输入

## 快照策略

- 自动保存只更新当前正文，不生成历史快照
- 手动点击“保存快照”时才创建修订记录
- 快照用于保留阶段版本，而不是替代自动保存

## 图片插入策略

- 支持上传图片并插入到正文
- 上传成功后立即在编辑区预览
- 图片最终以 Markdown 图片语法落盘
- 公开页渲染时沿用同一资源地址

## 发布策略

- 在编辑页顶部提供发布入口
- 未发布笔记可执行“发布”
- 已发布笔记可执行“更新发布”或“撤回发布”
- 发布状态应在编辑页中可见

## 常见状态

### 正常状态

- 已加载正文
- 可直接输入

### 空状态

- 新建笔记没有正文内容
- 提示开始输入

### 保存状态

- 未保存
- 保存中
- 保存成功
- 保存失败

### 内容状态

- 未发布
- 已发布
- 已撤回

## 异常处理原则

- 网络失败时保留编辑区当前输入
- 图片上传失败时给出重试提示
- 保存失败不能直接清空内容
- 如果未来出现冲突版本，优先保留本地编辑结果并提示用户处理

## 交互边界

- MVP 不做复杂协同光标
- MVP 不做多人同时编辑冲突合并
- MVP 不做复杂 Slash 命令系统
- MVP 不做数据库块级内容存储

## 技术实现原则

- 编辑器呈现可以富文本化
- 最终保存必须是 Markdown
- 后端只存储 Markdown，不存专有富文本结构
- 编辑器选择需优先支持 Markdown round-trip 的稳定性
