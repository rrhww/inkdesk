# 大模型维基与技能工作台产品设计

## 背景

Inkdesk 当前是一套 vault-first 的私有知识系统，核心闭环是：

```text
raw -> ingest -> wiki -> ask
```

文章《AI研发自动化：Wiki知识库+技能包》描述了更偏执行侧的形态：

```text
Sources -> LLM-Wiki -> Schema -> Skills -> Agent Runtime -> Evaluation -> Harness
```

这篇文章最值得吸收的不是照搬 Claude Code 工作流，而是四个长期层次：

- 可检查、可版本化、可外部编辑的文件优先知识库。
- 告诉 Agent 如何维护知识库的 schema 层。
- 把重复工作变成显式流程文件的 skill 层。
- 在提高自治度前做质量检查的 evaluation 和 harness 层。

Inkdesk 应该吸收这套结构，同时保留自己的产品方向：一个 owner 每天可用的私有工作台。

## 产品定位

Inkdesk 应成为个人 LLM-Wiki 与 skill-driven agent workspace 的控制面。

```text
Inkdesk 管理记忆、技能、评测、运行状态和审阅。
Claude Code、Codex、LangGraph 或其他 runtime 负责执行。
Obsidian 或普通编辑器可以检查和编辑 vault。
```

Inkdesk 不替代所有 Agent runtime，而是让记忆层和工作流底座更可靠、可见、可复用。

## 目标

- 让现有 vault 模型更明确兼容 LLM-Wiki 实践。
- 把 skills 当作一等产品对象，而不是藏在应用代码里的 prompt。
- 保持文件系统可迁移，方便高级用户直接使用 Obsidian、Git、Claude Code 或 Codex。
- 通过 Wiki Health 和 Skill Evaluation 增加质量治理。
- AI 生成的正式知识变更必须先经过人工审阅。

## 非目标

- 第一阶段不做完全自治的 Agent 执行系统。
- 不把 Obsidian、Claude Code 或任何单一 runtime 作为核心依赖。
- 不允许 AI 静默改写已接受的 wiki 知识。
- 不把 MVP 扩成多人团队知识库。
- 不恢复 public publishing 或协作发布面。

## 目标形态

Vault 结构：

```text
raw/        原始来源材料
wiki/       已接受的长期知识页
schema/     面向 Agent 的知识维护规则
skills/     可复用工作流说明
evals/      golden tasks、rubrics 和评测运行
runs/       Agent 运行记录、输出与审阅状态
```

应用层：

```text
Raw Library
Ingest Review
Wiki Browser
Ask / Research Chat
Skill Workbench
Wiki Health
Evaluation Harness
Agent Run Console
```

PostgreSQL 继续保存索引、队列和缓存读模型；Vault Markdown 始终是可恢复的长期真相层。

## 核心概念

### 大模型维基

wiki 不是文档堆，而是从 raw sources 编译出来、持续改进的知识层。

知识进入 wiki 的路径应是：

```text
raw source -> ingest proposal -> human review -> accepted wiki patch
```

Ask 也可以生成提案：

```text
ask answer -> save insight -> ingest proposal -> human review -> wiki patch
```

### Schema

Schema 文件是面向 Agent 的操作规则，用来说明 wiki 应如何被维护。

初始 schema 覆盖：

- 目录约定。
- wiki 页面模板。
- 来源引用和 provenance 规则。
- ingest proposal 规则。
- Ask 回答规则。
- Wiki lint 和 health 规则。

这些文件应保存在 vault 中，供 Inkdesk、Codex、Claude Code 或其他 Agent 读取。

### Skills

Skills 是可复用工作流，不只是 prompt。每个 skill 都应包含名称、输入、所需上下文、输出契约、安全规则和验证要求。

初始 knowledge-centered skills：

- Ingest Source。
- Patch Wiki Page。
- Answer From Wiki。
- Create Research Brief。
- Run Wiki Health Check。
- Extract Reusable Insight From Chat。

后续若扩展到研发工作流，再增加：

- Write Technical Plan。
- Review Technical Plan。
- Implement From Plan。
- Prepare Test Plan。
- Diagnose Issue。

### Evaluation Harness

在提高自治度前，Inkdesk 需要轻量评测层。

第一版支持：

- 以 Markdown 或 JSON 保存 golden tasks。
- 用 rubrics 描述期望质量。
- 独立 evaluation run，不修改正式 wiki。
- 保存基础分数与版本对比。

第一版不追求统计严谨，只需要让改进可见，避免 prompt 盲漂。

### Wiki Health

Wiki Health 是 vault 的质量仪表盘。

初始检查：

- 缺少来源引用。
- 内部链接断裂。
- 孤立 wiki 页面。
- 过短疑似 stub 的页面。
- 重复标题或重复别名。
- 缺少必要 frontmatter。
- 已接受 wiki 引用的 raw source 不存在。

后续检查：

- 潜在矛盾。
- 过期页面。
- 可合并的重叠概念。
- 高频出现但未沉淀成页面的概念。
- 输出持续无法通过审阅的 skills。

## 用户体验

### Owner Home

Owner Home 最终应表现为一个活的工作台：

- 最近新增 sources。
- 待审 ingest proposals。
- Wiki Health 分数和主要问题。
- 最近可沉淀的 Ask insights。
- 最近 skill runs。
- 建议维护任务。

### Skill Workbench

Skill Workbench 应支持 owner：

- 浏览 skills。
- 理解每个 skill 的用途。
- 用合适输入启动 skill run。
- 审阅输出。
- 接受、拒绝或修改 wiki 变更提案。
- 必要时导出外部 Agent 兼容的 skill 文件。

UI 默认隐藏 prompt 文件复杂度，但底层文件仍可检查。

### Ask to Wiki

Ask 应成为记忆构建界面，而不仅是问答界面。

一个有价值回答之后，owner 可以选择：

- 只保留为聊天。
- 保存为新的 wiki proposal。
- patch 现有 wiki 页面。
- 创建后续研究任务。

默认安全策略不变：不静默改写正式知识。

## 数据流

### Source Ingest

```text
Owner 添加 raw source
后端保存 raw markdown 和 metadata
Agent 创建 ingest proposal
Owner 审阅 proposal
接受后写入 wiki markdown
数据库索引 wiki page 和 source links
```

### Skill Run

```text
Owner 启动 skill
Inkdesk 解析所需 vault context
Runtime 在受控 run 中执行 skill
输出保存到 runs/
任何 wiki 变更先转成 proposal
Owner 审阅后再接受
```

### Wiki Health

```text
Owner 或定时任务启动 health check
规则扫描 vault files
结果保存为 health run
UI 展示分数、问题和建议修复
修复动作生成 proposal，不静默写入
```

## 架构说明

### Vault Layout

Vault 应保持可迁移。未来结构：

```text
raw/
wiki/
schema/
skills/
evals/
runs/
```

现有部署不需要立即迁移，第一版可以懒创建缺失目录。

### 数据库角色

PostgreSQL 保存：

- 索引记录。
- 审阅队列状态。
- Ask history。
- Skill run metadata。
- Evaluation run metadata。
- Health issue summaries。

数据库不能成为 accepted knowledge 的唯一存储。

### Runtime 角色

第一版可继续使用现有后端 Agent runtime，但设计不能锁死单一 runtime。

可选 runtime：

- 现有 LangGraph 后端。
- 通过导出文件和人工执行接入 Codex 或 Claude Code。
- 未来本地 Agent runner。

Inkdesk 拥有编排状态和审阅；runtime 负责执行。

## 安全规则

- AI 生成的 wiki 变更必须成为可审阅 proposal。
- 已接受 wiki 页面必须保留 raw source 或既有 accepted page 的 provenance。
- Skill run 必须记录输入、resolved context、输出和 acceptance state。
- Evaluation run 不得修改 canonical wiki 文件。
- 外部工具执行必须由 owner 显式启动。

## 建议阶段

### 阶段一：LLM-Wiki 对齐

- 增加 vault-level `schema/` 和 `skills/` 约定。
- 记录初始 schema 文件。
- 增加 Ask-to-Wiki proposal flow。
- 增加基础 Wiki Health checks。

### 阶段二：Skill Workbench

- 增加 skill 列表 UI。
- 增加 skill 详情页。
- 增加 skill run records。
- 将输出转换为可审阅 proposals。
- 支持导出 skill 文件。

### 阶段三：Evaluation Harness

- 增加 golden task 存储。
- 增加 evaluation run records。
- 增加简单 rubric scoring。
- 对比 skill 或 wiki 版本。

### 阶段四：Agent Harness

- 增加多步骤 run orchestration。
- 增加阶段 gates。
- 增加 rollback 和 retry records。
- 只有 review 与 evaluation history 支撑时才提高自治度。

## 本设计方向的验收标准

- 新贡献者能解释 Inkdesk 与普通 RAG app 的区别。
- vault 可在 Inkdesk 外部打开，核心知识不丢失。
- skill 可作为文件检查，也可从 UI 启动。
- Ask 可生成 wiki proposal，但不会静默修改 accepted knowledge。
- Wiki Health 至少能识别断链、孤页、缺来源和缺 frontmatter。
- 架构保持兼容外部 Agent runtime，不锁定单一 provider。

## 待决问题

- `skills/` 是否从一开始采用 Codex / Claude 风格的 `SKILL.md` 目录，还是使用更简单的 Inkdesk-native frontmatter 格式。
- `evals/` 优先 Markdown-first 还是 JSON-first。
- runs 主要保存在 vault、数据库，还是两者都保存。
- 支持多少 Obsidian metadata，但不把 Obsidian 变成依赖。

## 建议

采用混合模型：

```text
Markdown-first files for portability.
Database indexes for product speed and workflow state.
UI-first review for safety.
Runtime-agnostic skill execution for future flexibility.
```

这样既吸收文章最强的思路，即长期知识和可复用 skills 会复利增长，也保留 Inkdesk 的优势：把专家工作流变成连贯的个人工作台，而不是一堆文件和命令行仪式。
