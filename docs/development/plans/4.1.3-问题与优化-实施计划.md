# 4.1.3 问题与优化实施计划

> 对应文章《AI研发自动化：Wiki知识库+技能包》4.1.3“问题&优化”。本计划承接 4.1.2 已形成的 Dev Run、Deposit、基础 Wiki Health、薄 MCP 和异步编译能力，把“能持续生长”的知识库推进为“质量可解释、变化可追踪、评测可准备”的知识库。

> 设计事实以 `docs/development/specs/4.1.1-最初建设-知识库初始化设计.md` 为准；本文件只定义实施顺序、文件边界、测试方法和验收门禁。执行遵守根目录 `AGENTS.md` 的 mentorship-first 结对模式。

## 一、阶段目标

完成以下治理闭环：

```text
Vault 当前状态
  -> 确定性深度扫描
  -> finding + health score + schema gate
  -> 快照与趋势
  -> 人工修复 / review-first 提案
  -> 再次扫描验证
```

同时为后续 Evaluation 建立不污染正式知识的文件边界：

```text
真实使用问题
  -> Golden Task 候选
  -> 评分口径与版本
  -> Evaluation Run manifest
  -> 独立输出目录
  -> 后续 LLM Judge / 对比评测
```

本阶段完成后，用户应能回答：

1. 当前知识库有哪些确定性质量问题？
2. 哪些问题会阻塞后续评测，哪些只是维护建议？
3. 健康度相较上一版本是改善还是退化？
4. 哪些真实任务已经具备成为 Golden Task 的条件？
5. 一次评测如何固定输入版本并与正式 wiki 隔离？

## 二、进入条件

开始本阶段前应满足：

- 4.1.1 Vault 结构、Schema、页面模板、`index.md`、`log.md` 和 `KB-META.md` 可用。
- 4.1.2 基础 Health 能确定性报告断链、孤页、缺 frontmatter、缺来源。
- Deposit 和编译产物仍遵守 review-first，不直接改写正式 wiki。
- `/app/health`、`health_check` MCP 与后端 Health 使用同一应用服务。
- 4.1.2 尚未完成的真实全栈 E2E 必须保留为显式门禁，不得因进入本阶段而删除或伪造通过。

进入条件用于保护基线，不要求为了开始本阶段而一次清零历史 Vault 的所有 warning。

## 三、范围

### 3.1 本阶段包含

1. **3.1 健康规则契约**：页面状态语义、规则编号、严重度、finding 指纹和评分口径。
2. **3.2 深度 Health 与 Schema Gate**：短页、重复标题、stale/outdated、模板完整度和阻塞门禁。
3. **3.3 健康快照与趋势**：保存扫描产物、展示健康分与历史变化。
4. **3.4 多维索引与成熟度治理**：按标签维度组织入口，暴露页面深度和实体覆盖缺口。
5. **3.5 Evaluation 准备**：Golden Task 候选、rubric、版本 manifest 和隔离目录。

### 3.2 本阶段不包含

- 不实现完整 LLM Judge、双 seed、σ₀、显著性判断或模型横向排名。
- 不实现 Skill Workbench 或首批正式 Skill。
- 不让 Health 自动修复正式 wiki；修复必须人工执行或进入 review-first 提案。
- 不做自动语义冲突裁决；本阶段只报告确定性或可解释的信号。
- 不批量生成所有类页、实体页或业务线内容；只建立质量规则、覆盖视图和可审阅补全队列。
- 不引入多用户、多租户、登录、RBAC 或跨 Vault 联邦。
- 不因健康分较高而跳过结构 gate。

## 四、固定实施决策

### 4.1 确定性规则优先

同一份 Vault、同一规则版本、同一评测时间输入必须得到相同结果。Health 扫描不调用 LLM，不依赖网络，也不修改 Vault。

### 4.2 页面状态沿用初始化设计

状态只允许：

```text
stub | draft | stable | outdated
```

- `stub`：明确允许简略，不触发短页问题。
- `draft`：允许未完成，但缺关键段落和超过保鲜期应给 warning。
- `stable`：必须满足模板、来源和结构要求；不满足时可阻塞 schema gate。
- `outdated`：明确退出当前可靠知识集合，仍保留历史和引用关系。

Health 不根据正文长度擅自修改状态。

### 4.3 深度规则的第一版定义

| Rule ID | 信号 | 第一版判定 | 默认严重度 |
|---------|------|------------|------------|
| `KB_SHORT_PAGE` | 短页 | 去掉 frontmatter 后少于 100 个非空白字符，`stub` 豁免 | warning |
| `KB_DUPLICATE_TITLE` | 重复标题 | H1 归一化后完全相同 | error |
| `KB_SIMILAR_TITLE` | 高度相似标题 | 归一化标题长度不少于 4，`1 - Levenshtein距离 / 较长标题长度 >= 0.90` | warning |
| `KB_STALE_PAGE` | 长期未更新 | `draft/stable` 的 `updated` 距扫描基准日超过 180 天 | warning |
| `KB_OUTDATED_REFERENCE` | 来源失效 | `source_file` 指向不存在的 raw 文件，页面未标记 `outdated` | error |
| `KB_EXPLICIT_OUTDATED` | 明确过期 | 页面状态为 `outdated` | info |
| `KB_MISSING_SECTION` | 页面过浅 | `draft/stable` 缺失对应模板的必需标题；`stable` 为 error | warning/error |
| `KB_INVALID_STATUS` | 状态非法 | 缺失或不属于四个合法值 | error |

标题归一化只做 Unicode NFKC、大小写折叠、首尾空白与连续空白合并、常见标点清理。相似标题必须同时返回比较对象和计算依据，避免黑盒判断。

扫描范围必须递归覆盖 `wiki/**` 与 `raw/**`，不能只读取目录首层；路径统一为相对 Vault root 的 `/` 分隔形式。系统页、模板页和 eval 产物通过明确 allowlist/exemption 排除。

### 4.4 Gate 与健康分分离

- `schema_gate=FAILED`：存在任意 `error` finding，阻止进入 Evaluation Run。
- `schema_gate=PASSED`：不存在 `error` finding；warning 和 info 仍展示但不阻塞。
- `health_score`：遵循初始化设计，按“通过的适用检查数 / 全部适用检查数 × 100”计算，四舍五入到一位小数。
- 没有适用检查时不返回虚假的 100 分，返回 `null` 并标记 `not_applicable`。

分数用于趋势，Gate 用于决策；二者不得互相替代。

### 4.5 Finding 必须可稳定识别

每条 finding 至少包含：

```text
ruleId / type / severity / page / detail / evidence / fingerprint
```

`fingerprint` 由规则编号、规范化页面路径和关键证据生成，用于比较新增、持续和已解决问题。响应继续保留 4.1.2 的 `type/severity/page/detail` 字段，避免 Web 与 MCP 客户端突然失效。

### 4.6 文件优先保存治理产物

新增产物使用 4.1.1 已预留的 `evals/`：

```text
evals/
  health/
    rubric.json
    runs/<health-run-id>/manifest.json
  golden/
    candidates.json
    tasks/<task-id>.json
  rubrics/<rubric-id>.json
  runs/<eval-run-id>/manifest.json
  runs/<eval-run-id>/outputs/
```

- 正式 wiki 仍位于 `wiki/`。
- Evaluation 输出只能写入自己的 run 目录。
- JSON 采用 UTF-8、稳定字段顺序和显式 `schemaVersion`，暂不增加 YAML 依赖。
- 写入使用 Vault 的路径防逃逸和原子替换能力。

### 4.7 多维索引复用 tags

不新增与设计冲突的必填 frontmatter 字段。代码型 Vault 可使用 `business-line/<slug>` 标签，通用型 Vault 可使用 `topic/<slug>` 标签；`index.md` 在原有按页面类型视图之外生成按维度视图。无维度标签的页面进入“待分类”，不能静默消失。

### 4.8 实体和深度缺口先报告、后补全

本阶段通过模板完整度、页面类型统计、维度覆盖和实体覆盖报告发现缺口。任何自动补全建议必须进入 ReviewItem；不直接批量创建实体页或深度类页。

## 五、模块与文件边界

### 5.1 后端

| 文件/边界 | 计划职责 |
|-----------|----------|
| `health_service.py` | 扫描编排、兼容现有 Health 响应 |
| 新的 `health_rules.py` | 规则实现、状态语义、标题归一化和 finding 指纹 |
| 新的 `health_history_service.py` | 健康快照、差异和趋势读取 |
| 新的 `evaluation_service.py` | Golden Task/rubric 校验和 Evaluation Run manifest 隔离 |
| `vault.py` | 复用安全路径、原子读写和列举，不承载评分规则 |
| `schemas.py` | Health、Gate、History、Golden/Eval manifest 契约 |
| `main.py` | 路由接线，不实现扫描或评分逻辑 |
| `mcp_services.py` / MCP | 复用 Health 应用服务，保持薄适配 |
| `vault_assets.py` | 仅更新新初始化 Vault 的规则与 evals 基础资产，不覆盖旧 Vault |

优先拆分现有 `health_service.py`，不要继续把所有规则堆进一个类，也不要把 Evaluation 逻辑放入大型 `research.py`。

### 5.2 API 契约

| 方法 | 路径 | 职责 |
|------|------|------|
| `GET` | `/api/health` | 返回当前扫描、健康分和 gate 状态，不持久化 |
| `POST` | `/api/health/runs` | 执行并保存一次健康快照 |
| `GET` | `/api/health/runs` | 返回最近快照和趋势摘要，支持受限 `limit` |
| `GET` | `/api/health/runs/{id}` | 返回单次快照及 findings |
| `GET` | `/api/evals/golden` | 读取并校验 Golden Task 候选/任务 |
| `POST` | `/api/evals/runs` | 仅在 schema gate 通过时创建隔离 manifest，不执行 LLM Judge |
| `GET` | `/api/evals/runs/{id}` | 查看 manifest 和产物边界 |

所有 ID、limit、文件路径和文本字段必须有长度与格式限制。API 不返回 Vault 绝对路径。

### 5.3 前端

| 边界 | 计划职责 |
|------|----------|
| `/app/health` | 当前分数、gate、分类计数、finding 筛选和趋势 |
| Health 摘要组件 | 供 Dev Run Console 复用当前 gate/score |
| `lib/types.ts` | 扩展兼容 Health、History、Gate 类型 |
| `lib/research.ts` | 新增快照与历史 helper |
| `lib/mock/research-fixtures.ts` | 补齐稳定的趋势和深度 finding fixtures |

本阶段不新建完整 `/app/evals` 产品页面；Golden 与 Evaluation Run 的正式工作台属于后续评测阶段。

### 5.4 测试

建议独立新增：

- `server/tests/test_health_deep_rules.py`
- `server/tests/test_health_history_api.py`
- `server/tests/test_evaluation_boundaries.py`
- `web/tests/unit/health-dashboard.test.tsx`
- `web/tests/e2e/health-dashboard.spec.ts`

基础 Health 的已有测试保留，作为兼容回归。

## 六、执行顺序与依赖

```text
3.1 规则契约
  -> 3.2 深度 Health + schema gate
      -> 3.3 快照、差异与趋势
      -> 3.5 Golden/Evaluation 边界

3.1 规则契约
  -> 3.4 多维索引与成熟度治理

3.3 + 3.4 + 3.5
  -> 集成验收与文档收口
```

不得先做趋势 UI 再补快照契约，也不得先创建 Evaluation Run 再补 schema gate。

## 七、实施切片

### 7.1 切片一：健康规则契约（3.1）

#### 目标

固定状态、规则、严重度、finding 和评分语义，为后续扫描和趋势提供稳定输入。

#### RED

- [ ] 非法 `status`、`stub` 短页豁免和 `stable` 模板缺失测试先失败。
- [ ] finding fingerprint 在重复扫描中保持一致的测试先失败。
- [ ] 无适用检查时 score 返回 `null` 的测试先失败。
- [ ] 旧 Health 字段继续存在的兼容测试先失败。

#### GREEN

- [ ] 抽取规则模块和版本常量。
- [ ] 明确 `error/warning/info` 与 gate 映射。
- [ ] Health 响应增加 `ruleVersion`、`healthScore`、`gateStatus`。
- [ ] 更新 `schema/wiki-health-rules.md` 的新初始化模板。
- [ ] 旧 Vault 未含新版规则文件时使用内置同版本默认值，不自动覆盖用户文件。

#### 验收

- [ ] 同一 fixture 重复扫描结果和 fingerprint 完全一致。
- [ ] 四种页面状态都有正反例。
- [ ] 基础四类 finding 无回归。
- [ ] 规则版本出现在响应和快照 manifest 中。

### 7.2 切片二：深度 Health 与 Schema Gate（3.2）

#### 目标

实现短页、重复/相似标题、stale/outdated、来源失效和模板深度检查，并形成阻塞门禁。

#### RED

- [ ] 100 字边界和 `stub` 豁免测试先失败。
- [ ] `wiki/entities/**` 等嵌套页面会被扫描的测试先失败。
- [ ] Unicode、空白、标点差异导致的重复标题测试先失败。
- [ ] 相似标题阈值上下界测试先失败。
- [ ] 180 天边界使用固定扫描日期的测试先失败。
- [ ] raw 来源删除和显式 `outdated` 的差异测试先失败。
- [ ] error 阻塞、warning 不阻塞的 gate 测试先失败。
- [ ] 扫描不修改 Vault 的测试先失败。

#### GREEN

- [ ] 每条规则作为独立纯函数或规则对象实现。
- [ ] 页面收集改为递归且每个 Markdown 文件只读取一次。
- [ ] 扫描时显式传入 `evaluatedAt`，测试不依赖机器当前时间。
- [ ] `HealthService` 聚合基础规则和深度规则。
- [ ] `GET /api/health` 返回 score、gate 和分类统计。
- [ ] MCP `health_check` 复用同一摘要并保持原字段兼容。

#### 验收

- [ ] 深度七类信号均有稳定 fixture。
- [ ] finding 包含可解释 evidence，不只给结论。
- [ ] 路径、模板和系统页豁免行为明确。
- [ ] error 存在时无法创建 Evaluation Run。

### 7.3 切片三：健康快照、差异与趋势（3.3）

#### 目标

把单次扫描升级为可比较的治理记录，并在 `/app/health` 展示趋势。

#### RED

- [ ] 保存快照写入独立 run 目录的测试先失败。
- [ ] 重复 run ID、非法 ID 和路径逃逸测试先失败。
- [ ] 新增、持续、已解决 finding 差异测试先失败。
- [ ] 历史 limit 上限和空历史测试先失败。
- [ ] 健康页 score、gate、趋势和筛选状态测试先失败。

#### GREEN

- [ ] 快照 manifest 固定 Vault 标识、规则版本、扫描时间、score、gate 和 findings。
- [ ] 通过 fingerprint 计算新增、持续、已解决问题。
- [ ] 历史 API 只读取受控目录下的合法 manifest。
- [ ] `/app/health` 展示当前分数、Gate、最近趋势和分类筛选。
- [ ] 趋势图使用可访问的文本/表格降级，不只依赖颜色。

#### 验收

- [ ] 连续两次快照能说明具体改善或退化来源。
- [ ] 删除坏 manifest 不影响当前实时扫描。
- [ ] 页面在无历史、单条历史和多条历史下均可用。
- [ ] 真实浏览器验证筛选、空状态和趋势信息。

### 7.4 切片四：多维索引与成熟度治理（3.4）

#### 目标

解决“只有页面类型、没有业务/主题入口”和“页面数量很多但深度未知”的问题。

#### RED

- [ ] code/general 两类 Vault 标签分组测试先失败。
- [ ] 未分类页面不会从 index 消失的测试先失败。
- [ ] 同一页面多标签去重和排序测试先失败。
- [ ] 各类型页面的 status 分布、缺模板段落和实体覆盖统计测试先失败。
- [ ] index 更新仍遵守 review/accepted 写回时机的测试先失败。

#### GREEN

- [ ] index 生成器保留按页面类型视图并增加按维度视图。
- [ ] 输出 `stub/draft/stable/outdated` 成熟度统计。
- [ ] 输出概念、实体、来源、查询等类型覆盖和待分类列表。
- [ ] code Vault 显示实体覆盖缺口，但不直接扫描并写入代码实体页。
- [ ] 补全建议通过现有 ReviewItem/Deposit 边界进入审阅。

#### 验收

- [ ] 新用户能从维度视图找到入口。
- [ ] 每个页面在类型视图和至少一个维度/待分类视图中可见。
- [ ] 页面浅是因为 `stub`，还是违反 `draft/stable` 规则，可以区分。
- [ ] index 更新不绕过 review-first，也不破坏手工内容保护策略。

### 7.5 切片五：Golden Tasks 与 Evaluation 隔离准备（3.5）

#### 目标

把真实任务候选、评分口径和运行版本文件化，但暂不执行完整模型评测。

#### RED

- [ ] Golden Task 缺输入、期望、来源或 rubric 时校验失败的测试先失败。
- [ ] task/rubric 重复 ID 和 schemaVersion 不兼容测试先失败。
- [ ] schema gate 失败时创建 Eval Run 被拒绝的测试先失败。
- [ ] manifest 固定 task、rubric、Vault 和规则版本的测试先失败。
- [ ] Eval 输出尝试写入 `wiki/` 时被拒绝的测试先失败。

#### GREEN

- [ ] 候选集优先记录真实 Dev Run/Ask 场景，不凭空制造题目。
- [ ] 每个 Task 包含输入、任务类型、期望证据、允许/禁止行为、来源和状态。
- [ ] 每个 rubric 包含维度、分值、通过条件和人工复核说明。
- [ ] 创建 Eval Run 前验证 task、rubric、Vault 版本和 schema gate。
- [ ] manifest 与 outputs 限定在 `evals/runs/<id>/`。
- [ ] 首批准备 5 个候选，覆盖 PRD、bug、改造、Context Ask、知识沉淀；只有审核后的候选才能标记 `active`。

#### 验收

- [ ] Golden Task 和 rubric 可被 Git 版本化和独立校验。
- [ ] Eval Run 能复现“用什么知识、规则和任务版本运行”。
- [ ] 失败或取消的 Eval Run 不修改 wiki、index 或 log。
- [ ] 本阶段没有伪装成 LLM Judge 的占位分数。

## 八、跨切片质量要求

### 8.1 安全

- 所有路径继续经过 Vault root 防逃逸校验。
- API 不暴露绝对路径、密钥、模型参数秘密或内部堆栈。
- `limit`、run ID、task ID、rubric ID 和字符串长度有上限。
- Evaluation 输出目录采用 allowlist，不接受调用方提供任意写入路径。

### 8.2 可观测性

每次持久化扫描至少记录：

- `healthRunId`
- `vaultType`
- `ruleVersion`
- `evaluatedAt`
- `durationMs`
- `healthScore`
- `gateStatus`
- finding 分类计数

不记录 API key、原始敏感正文或主机绝对路径。

### 8.3 兼容性

- 4.1.2 的 Health 字段继续存在。
- 旧 Vault 无新增 eval 文件时仍能实时扫描。
- 初始化只补缺失资产，不覆盖已编辑规则或评测文件。
- MCP `health_check` 不因新增字段破坏现有调用方。
- `index.md` 新视图必须保留原页面类型和统计区。

### 8.4 性能

- 单次扫描只读取每个 Markdown 文件一次，再在内存中执行规则。
- 第一版基准：1000 个 Markdown 页面时，确定性扫描在开发机目标低于 2 秒。
- 性能测试使用生成 fixture，不提交大量临时 Markdown。
- 历史列表默认最多返回 20 条，硬上限 100 条。

## 十、结对分工

### 用户负责核心内容

- 审核并实现页面状态、严重度和 schema gate 的核心判断。
- 审核健康分公式及“分数与门禁分离”的架构取舍。
- 实现或主导 Evaluation Run 隔离边界。
- 解释 fingerprint、时间确定性和 review-first 的设计理由。
- 审核每个切片的最终 diff 与验证证据。

### Codex 负责支持内容

- 维护计划、检查清单、进度和验收报告。
- 准备规则 fixtures、RED 测试建议和边界案例。
- 处理重复性 schemas/types、mock、文档和展示层工作，并交用户审核。
- 审查实现的正确性、性能、安全、兼容性和测试质量。
- 在用户明确授权后直接处理非核心修复和验证工作。

## 十三、预计工期

## 十四、阶段复盘问题
