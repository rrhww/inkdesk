# 4.1.4 Skill 开发准备实施计划

> 4.1.4 是 Inkdesk 在知识库治理完成后、进入 4.2.1 初版 Skill 生产前增加的工程准备阶段。它解决“用什么格式、怎样校验、如何路由、如何测试和怎样安全发布 Skill”的问题，不直接批量开发业务 Skill。

> 本计划以 `docs/development/specs/2026-06-04-大模型维基与技能工作台产品设计.md` 为产品设计依据，并参考本机 `omni-superdev` 的工程组织方式：单一入口路由、内置模式、纪律层与领域能力分离、渐进式加载、资源分层、冲突策略和完成门禁。Inkdesk 不依赖本机绝对路径，也不复制 `omni-superdev` 的具体提示词。

> 执行遵守根目录 `AGENTS.md` 的 mentorship-first 结对模式。用户负责核心契约与路由决策；Codex 负责脚手架、校验、fixtures、文档和验收支持，并交用户审核。

## 一、阶段定位

4.1.3 解决知识库质量和 Evaluation 准备；4.2.1 开始生产首批正式 Skill。4.1.4 位于两者之间：

```text
4.1.3 深度健康与评测准备
  -> 4.1.4 Skill SDK / 生产规范 / 校验门禁
    -> 4.2.1 首批正式 Skill
      -> 4.2.2 真实使用反馈与优化
```

本阶段建立以下生产闭环：

```text
Skill 意图与真实用例
  -> 初始化 Skill package
  -> 编写 SKILL.md + contract.json + 按需资源
  -> structural lint
  -> semantic lint
  -> fixture / forward test
  -> 人工审核
  -> draft -> active
```

## 二、阶段目标

完成一个可被 4.2.1 直接使用的 Inkdesk Skill Development Kit：

1. 固定 portable Skill package 格式。
2. 固定输入、上下文、输出、安全、验证和下游衔接契约。
3. 提供初始化、校验和测试脚手架。
4. 提供 Skill 路由、链路和冲突优先级规则。
5. 提供一个仅用于测试的 reference fixture，证明规范可执行。
6. 为后续 Skill Registry、Workbench、Run 和导出建立稳定读取边界。

## 三、进入条件

- 4.1.1 已创建 `skills/`、`schema/`、`evals/` 和 `runs/`。
- 4.1.2 的 Dev Run、Context Pack、Deposit、Review 和 MCP 应用服务可复用。
- 4.1.3 已固定 schema gate、Golden Task 候选和 Evaluation Run 隔离边界。
- 所有 wiki 写入仍经 proposal/review，Skill 不获得例外权限。
- 产品继续采用单实例、单用户、无登录部署，不为 Skill 引入用户或租户模型。

如果 4.1.3 尚未全部实施，可以先编写 4.1.4 文档契约与纯校验器，但 Skill promotion 与正式生产必须等待 schema/evaluation gate 就绪。

## 四、范围

### 4.1 本阶段包含

1. **4.1.4.1 Skill Package 规范**：目录、命名、生命周期和渐进式加载。
2. **4.1.4.2 Contract Schema 与校验器**：机器可读契约、结构和语义 lint。
3. **4.1.4.3 脚手架与 Registry 读取边界**：初始化、发现、解析和状态报告。
4. **4.1.4.4 路由、链路与冲突策略**：单一入口、Skill 类型、优先级和 Hard Gates。
5. **4.1.4.5 Skill 测试与 Promotion 门禁**：fixtures、契约测试、forward test 记录和激活规则。

### 4.2 本阶段不包含

- 不开发 6 个知识管理 Skill、7 个研发 Skill 或 `skill-router` 正式内容。
- 不建设 `/app/skills` 列表、详情、运行或导出 UI。
- 不实现完整 Skill Runtime、Agent Harness 或自动多步骤执行。
- 不让脚手架生成未经审核即可运行的 active Skill。
- 不引入任意表达式或可执行代码作为 Hard Gate DSL。
- 不建立跨平台完全一致的执行语义；只保证文件可读、核心契约可映射。
- 不把本机 `${CODEX_HOME}/skills/omni-superdev` 作为运行时依赖或提交进仓库。

## 五、参考 `omni-superdev` 后采用的原则

| `omni-superdev` 做法 | Inkdesk 4.1.4 采用方式 |
|----------------------|------------------------|
| 一个主路由拥有端到端流程 | 后续 `skill-router` 是唯一宽入口，领域 Skill 不互相抢占 |
| 内置 direct/planning/execution 等模式 | 宽 Skill 可声明有限 modes；简单 Skill 保持单一职责 |
| discipline 与 domain skill 分层 | TDD、debug、verification 属于纪律；tech-solution 等属于领域能力 |
| `references/prompts/templates/scripts` 分层 | Skill package 按需创建资源目录，保持渐进式加载 |
| `workflow-map` 与 `conflict-policy` | Inkdesk 固定路由优先级、链路选择和冲突处理 |
| implementer/reviewer prompt 分离 | 生成型、检查型 Skill 使用不同自由度和验证策略 |
| plan/spec/code review 双门禁 | Skill 激活前先验契约，再验行为与质量 |
| 完成前必须有验证证据 | Skill promotion 必须引用 lint、fixture 和评测记录 |

不采用的部分：

- 不照搬其全能工程范围到每一个 Inkdesk Skill。
- 不默认使用子 Agent；只有运行环境允许且任务确实可独立时才使用。
- 不复制其本地 planning 脚本；Inkdesk 只实现 Skill SDK 所需的最小脚手架。
- 不允许 Skill 自己提升权限或绕过 Inkdesk 的 Review、Health、Evaluation 门禁。

## 六、固定 Skill Package 规范

### 6.1 目录格式

正式 Skill 采用 Codex/Claude 风格的目录，同时增加 Inkdesk 机器契约：

```text
skills/<skill-name>/
  SKILL.md                 # Agent 可读的核心流程，必需
  contract.json            # Inkdesk 机器契约，必需
  agents/
    openai.yaml            # 外部 Agent/UI 元数据，推荐
  references/              # 按需加载的领域规则，可选
  prompts/                 # 角色或阶段提示模板，可选
  templates/               # 输出文档模板，可选
  scripts/                 # 确定性、可独立测试的辅助脚本，可选
  assets/                  # 输出所需静态资源，可选
```

约束：

- Skill 根目录只允许小写字母、数字和连字符，长度 1–63。
- `SKILL.md` 与 `contract.json` 必须存在。
- Scaffolder 默认同时生成 `agents/openai.yaml`；导入的外部 package 可缺省该文件。
- 资源目录按需创建；空目录、README、CHANGELOG、安装指南不进入 Skill package。
- `SKILL.md` 目标不超过 500 行；复杂细节移入一层 `references/`，禁止深层引用链。
- 所有内部引用使用相对 Skill root 的路径。
- `skills/` 只放正式 Skill package；脚手架模板保存在服务端资源目录，避免被 Registry 误识别为 Skill。

行为用例和 promotion 证据不塞进可分发 package，统一保存在 Vault 级评测目录：

```text
evals/skills/<skill-id>/contract-cases.json
evals/skills/<skill-id>/results/<version>.json
evals/skills/<skill-id>/promotion/<version>.json
```

外部 Agent 使用 Skill package 时不需要加载测试历史。

### 6.2 `SKILL.md` 责任

Frontmatter 仅包含：

```yaml
---
name: skill-name
description: 说明做什么，以及哪些用户意图应触发或不应触发。
---
```

正文只保留 Agent 执行所需的非显然内容：

- 核心目标与边界。
- 必须停止的 Hard Gates。
- 主流程和关键决策点。
- 需要读取哪些 bundled resources。
- 输出后如何验证和衔接下一个 Skill。

触发条件必须主要写在 frontmatter `description`，不能只藏在正文中。

### 6.3 `contract.json` 责任

第一版固定字段：

```json
{
  "schemaVersion": "1.0",
  "id": "tech-solution",
  "version": "0.1.0",
  "status": "draft",
  "category": "engineering",
  "kind": "producer",
  "summary": "从需求和知识上下文生成可评审技术方案",
  "inputs": [],
  "contextRequirements": [],
  "outputs": [],
  "hardGates": [],
  "capabilities": [],
  "writePolicy": {},
  "verification": [],
  "nextSkills": [],
  "supportedRuntimes": []
}
```

字段语义：

| 字段 | 约束 |
|------|------|
| `schemaVersion` | 第一版只接受 `1.0` |
| `id` | 必须与目录名和 SKILL frontmatter `name` 一致 |
| `version` | SemVer，内容或契约变化时显式更新 |
| `status` | `draft / active / deprecated` |
| `category` | `knowledge / engineering / routing / discipline` |
| `kind` | `producer / reviewer / router / diagnostic` |
| `inputs` | 名称、类型、是否必需、长度/格式约束，不承载秘密默认值 |
| `contextRequirements` | Vault、Dev Run、代码仓、来源、阶段等前置上下文 |
| `outputs` | 产物类型、保存位置类别、是否需要 review |
| `hardGates` | 只使用允许的声明式 gate vocabulary |
| `capabilities` | 运行所需能力 allowlist，不等同于权限授予 |
| `writePolicy` | canonical wiki、run artifact、code repo 的写入策略 |
| `verification` | lint、测试、人工复核或 evaluation 要求 |
| `nextSkills` | 可选下游 Skill 与进入条件 |
| `supportedRuntimes` | `inkdesk / codex / claude-code / external-agent` 等兼容声明 |

### 6.4 Skill 类型与自由度

| kind | 典型 Skill | 指令自由度 | 主要要求 |
|------|------------|------------|----------|
| `producer` | tech-solution、coding | 中高 | 原则和输出契约明确，避免穷举所有情况 |
| `reviewer` | tech-review | 低到中 | 保留结构化检查项，结论必须有 evidence |
| `diagnostic` | problem-solve、test-fix | 中 | 必须从真实失败信号开始，禁止猜测式修复 |
| `router` | skill-router | 中 | 只路由和编排，不复制所有领域 Skill 内容 |

## 七、安全与执行契约

### 7.1 Hard Gate vocabulary

第一版只允许以下声明式 gate：

- `required_input`
- `vault_initialized`
- `schema_gate_passed`
- `dev_run_exists`
- `run_stage_is`
- `review_approved`
- `artifact_exists`
- `real_failure_signal_present`
- `human_confirmation`

每个 gate 必须包含稳定 `id`、`kind`、参数、失败码和用户可理解的失败信息。Gate 失败时必须停止或返回 blocked，不允许 Skill 自行降级绕过。

### 7.2 Write Policy

第一版允许值：

```text
canonicalWiki: denied | proposal-only
runArtifacts: denied | allowed
codeRepository: denied | delegated
```

固定规则：

- 不存在 `canonicalWiki: direct`。
- Wiki 变更统一进入 Deposit/ReviewItem。
- Skill 自身只描述所需 capability，实际权限由 Runtime/Harness 决定。
- 外部 Agent 修改代码必须由用户显式启动，并记录到 Dev Run。
- scripts 不得读取未声明的环境变量，不得包含秘密或主机绝对路径。

### 7.3 路径规则

- Vault 与代码仓从 `KB-META.md`、Dev Run 或明确输入解析。
- package 内路径相对 Skill root。
- 不提交用户目录、盘符或团队成员机器路径。
- Validator 扫描 Windows 与 Unix 绝对路径模式。

## 八、路由、链路与冲突策略

### 8.1 单一宽入口

正式阶段由 `skill-router` 承担宽意图入口。其他 Skill 的 description 必须窄到可区分，不能都声明“适用于所有研发任务”。

### 8.2 优先级

```text
1. 用户当前明确指令
2. 项目 AGENTS.md / Schema / 安全约束
3. Dev Run 当前状态与已批准门禁
4. skill-router 的类型与链路选择
5. 当前领域 Skill
6. 默认风格偏好
```

低优先级 Skill 不能覆盖高优先级约束。

### 8.3 两条初始链路

本阶段只固定链路契约，不实现其中 Skill：

```text
知识管理：
ingest-source -> patch-wiki-page
answer-from-wiki -> deposit-answer -> patch-wiki-page
run-wiki-health -> patch-wiki-page

研发自动化：
tech-solution -> tech-review -> coding -> test-prep -> test-fix
                                         -> problem-solve
```

每条连接必须通过 `nextSkills` 声明输入输出可兼容性和进入 gate；不能只在 SKILL.md 正文口头描述。

### 8.4 冲突处理

- 多个领域 Skill 同时匹配时，router 选择范围最窄且满足全部前置条件者。
- reviewer 不接管 producer 的产出职责。
- discipline 类能力只增加门禁，不成为第二个流程控制器。
- 无法唯一选择时返回候选和差异，请用户决定，不随机挑选。
- Skill 链出现环时 validator 失败；显式 retry 边只允许指向当前 Skill，且必须有次数上限。

## 九、模块与文件边界

### 9.1 Vault 默认资产

| 资产 | 职责 |
|------|------|
| `schema/skill-package-spec.md` | 人和 Agent 可读的 package 规范 |
| `schema/skill-contract.schema.json` | `contract.json` 的机器 Schema |
| `schema/skill-routing-rules.md` | 路由优先级、链路和冲突策略 |
| `schema/skill-safety-rules.md` | Hard Gates、写入和路径规则 |

初始化只补缺失资产，不覆盖已有 Vault 文件。

### 9.2 服务端

| 建议边界 | 职责 |
|----------|------|
| 新的 `skill_contracts.py` | Pydantic contract 模型和枚举 |
| 新的 `skill_registry.py` | 递归发现、解析、跨文件一致性和链路校验 |
| 新的 `skill_scaffolder.py` | 从内置模板初始化 draft package |
| 新的 `skill_validation.py` | 结构、语义、安全和引用 lint |
| `vault.py` | 安全读写、路径防逃逸，不解析 Skill 业务语义 |
| `vault_assets.py` | 注册新的 schema 默认资产 |

Registry 只读取并返回 metadata/validation，不执行 Skill。

`skill_contracts.py` 是 contract 机器 Schema 的单一来源，`schema/skill-contract.schema.json` 由它确定性生成并通过 drift test 校验。SKILL frontmatter 与 `agents/openai.yaml` 使用 `PyYAML` 解析；禁止用正则或逐行切割模拟 YAML，`PyYAML` 需加入服务端正式依赖。

### 9.3 开发脚本

提供跨平台入口，业务逻辑复用服务层：

```powershell
cd server
python -m inkdesk_server.skill_cli init <skill-name> --description <text> --category <category> --kind <kind>
python -m inkdesk_server.skill_cli validate [<skill-name>]
python -m inkdesk_server.skill_cli graph
python -m inkdesk_server.skill_cli check <skill-name>
```

- `init` 要求 description、category 和 kind，创建 `draft` package，已存在时拒绝覆盖；资源目录通过显式 `--resources` 创建。
- `validate` 返回稳定退出码和结构化错误。
- `graph` 检查 nextSkills 缺失、输入输出不兼容和循环。
- `check` 汇总 lint、behavioral cases、已记录执行结果、schema/evaluation gate 和人工审核引用，生成 promotion report；它不执行模型，也不自动把 Skill 改为 active。
- Registry 遇到 `status=active` 但缺少同版本 PASSED promotion report 时，将 package 标记为 invalid，不参与路由。

### 9.4 测试 fixtures

测试目录提供：

- 最小合法 producer Skill。
- 最小合法 reviewer Skill。
- 非法 frontmatter、ID 不一致、绝对路径、越权写入、坏引用和循环链路 fixtures。
- fixture 只用于校验器测试，不作为首批业务 Skill 发布。

## 十、校验模型

### 10.1 Structural Lint

- 目录名、必需文件和允许目录。
- SKILL frontmatter 可解析且只有 `name/description`。
- contract JSON 可解析并通过 schemaVersion 1.0。
- SemVer、枚举、字段长度和数组上限。

### 10.2 Semantic Lint

- 目录、frontmatter name、contract id 三者一致。
- description 长度合法且不与 name/summary 简单重复；触发边界质量由 behavioral cases 和人工审核判断。
- required input 在 Hard Gates 中有相应约束。
- output、writePolicy 和 review 要求一致。
- nextSkills 均存在、无非法环，链路输入输出可兼容。
- agents/openai.yaml 存在时与 Skill 名称和描述不冲突。

### 10.3 Safety Lint

- canonical wiki 只能 denied 或 proposal-only。
- 无硬编码绝对路径、秘密样例和未声明 capability。
- scripts 和 templates 的引用不能逃逸 package root。
- Skill 不得声称可以绕过 review、schema gate 或用户确认。

### 10.4 Behavioral Contract Test

每个待激活 Skill 至少提供：

- 2 个应触发用例。
- 2 个不应触发用例。
- 1 个 Hard Gate 失败用例。
- 1 个正常输出契约用例。
- 1 个写入边界用例。

`contract-cases.json` 保存输入和期望，`results/<version>.json` 保存由人工或后续 Runtime/外部 Agent 得到的实际结果与证据。4.1.4 只固定格式并用 reference fixture 验证；正式业务执行结果在 4.2.1 随 Skill 实现后产生。

## 十一、执行顺序与依赖

```text
4.1.4.1 package + contract 决策
  -> 4.1.4.2 parser + validator
      -> 4.1.4.3 scaffolder + registry
      -> 4.1.4.4 routing graph + conflict lint
          -> 4.1.4.5 behavioral fixture + promotion gate
```

Contract Schema 未固定前不实现脚手架；validator 未通过前不实现 Registry API 或 Workbench。

## 十二、实施切片

### 12.1 切片一：Package 与 Contract 规范

#### RED

- [ ] 缺 SKILL.md、缺 contract、非法名称和 ID 不一致测试先失败。
- [ ] contract schemaVersion、SemVer 和枚举错误测试先失败。
- [ ] 额外 frontmatter 字段和深层引用测试先失败。

#### GREEN

- [ ] 添加四份 Skill Schema 默认资产。
- [ ] 添加 contract Pydantic 模型与 JSON Schema。
- [ ] 固定 package、生命周期、类型和资源目录。
- [ ] 固定错误对象：`code/path/message/severity`。

#### 验收

- [ ] 合法 fixture 可解析为稳定 metadata。
- [ ] 非法 fixture 返回可定位错误，不抛内部堆栈。
- [ ] 旧 Vault 初始化时只补缺失 Schema。

### 12.2 切片二：结构、语义与安全 Validator

#### RED

- [ ] writePolicy 越权、绝对路径、坏引用和未声明 capability 测试先失败。
- [ ] description 为空或简单重复、required input 无 gate、output/review 冲突测试先失败。
- [ ] Windows/Unix 路径逃逸模式测试先失败。

#### GREEN

- [ ] 分层实现 structural/semantic/safety lint。
- [ ] 输出稳定 finding code 与最终 PASSED/FAILED。
- [ ] 单个错误不阻止报告其余独立错误。
- [ ] Validator 只读，不修改 package。

#### 验收

- [ ] 所有非法 fixture 被对应规则捕获。
- [ ] 同一 package 重复校验结果顺序一致。
- [ ] 100 个 Skill package 校验目标低于 2 秒。

### 12.3 切片三：Scaffolder、CLI 与 Registry

#### RED

- [ ] init 已存在目录不覆盖测试先失败。
- [ ] init 生成的 draft 通过 structural lint、但不满足 promotion gate 的测试先失败。
- [ ] Registry 忽略非 package 目录并报告 invalid package 的测试先失败。
- [ ] CLI 成功/失败退出码测试先失败。

#### GREEN

- [ ] 内置最小 Skill 模板，不在 Vault `skills/` 中保存模板假包。
- [ ] CLI init/validate/graph/check 复用服务层。
- [ ] Registry 返回 valid、invalid、draft、active、deprecated 摘要。
- [ ] Registry 不执行脚本、不加载 prompt 正文、不调用模型。

#### 验收

- [ ] 新建 draft Skill 不需要手写目录骨架。
- [ ] 生成物没有未替换占位标记、示例秘密或主机绝对路径；用户提供的 description/category/kind 已写入契约。
- [ ] Registry 输出可被后续 `/app/skills` 直接消费。

### 12.4 切片四：Routing Graph 与冲突规则

#### RED

- [ ] nextSkill 缺失、非法环、输入输出不兼容测试先失败。
- [ ] 多宽入口冲突和 router 缺失测试先失败。
- [ ] reviewer/producer 职责冲突测试先失败。
- [ ] retry 自环无上限测试先失败。

#### GREEN

- [ ] 构建只读 Skill graph。
- [ ] 校验单一 router、类别、kind 和链路兼容性。
- [ ] 固定用户/项目/安全/router/domain 优先级。
- [ ] graph CLI 输出节点、边和 findings，不启动运行。

#### 验收

- [ ] 两条初始链路可由 contract 完整表达。
- [ ] 冲突时能解释选择或要求用户决定。
- [ ] Discipline 能加门禁但不能成为第二流程控制器。

### 12.5 切片五：Behavioral Fixture 与 Promotion Gate

#### RED

- [ ] lint 失败无法 active 的测试先失败。
- [ ] 缺触发、非触发、gate 或 write-boundary 用例无法 active 的测试先失败。
- [ ] schema/evaluation gate 未满足无法 active 的测试先失败。

#### GREEN

- [ ] 固定 contract cases、`results/<version>.json` 和 promotion report 格式。
- [ ] reference producer/reviewer fixtures 通过完整准备门禁。
- [ ] promotion report 汇总 lint、contract tests、人工审核和 eval 引用。
- [ ] 激活只修改 contract status/version，不自动执行 Skill。

#### 验收

- [ ] 一条命令可给出 package 是否具备进入 4.2.1 的条件。
- [ ] 失败报告能明确指出缺失证据。
- [ ] reference fixture 不出现在正式 Skill Registry 的 active 列表。

## 十三、Promotion 门禁

`draft -> active` 必须全部满足：

1. Structural、semantic、safety lint 全部 PASSED。
2. Behavioral contract cases 全部通过。
3. Skill description 的触发/非触发边界已人工审核。
4. Hard Gates 和 writePolicy 已人工审核。
5. 存在对应 Golden Task/rubric 或经审核的延期理由。
6. schema gate 通过。
7. version 相较上一 active 版本已更新。

`active -> deprecated` 必须记录替代 Skill 或明确终止原因；deprecated Skill 不参与自动路由，但历史 Run 仍可解析。

## 十五、结对分工

### 用户负责核心内容

- 决定并解释 `SKILL.md + contract.json` 双层契约。
- 审核 Hard Gate vocabulary、writePolicy 和 promotion gate。
- 主导路由优先级、Skill 类型和链路兼容规则。
- 审核 runtime-agnostic 与外部 Agent 兼容边界。
- 审核最终 fixtures、错误语义和 diff。

### Codex 负责支持内容

- 维护计划、Schema 字段表、任务清单和进度。
- 准备 RED fixtures、非法边界和验证矩阵。
- 实现或维护脚手架、重复性 validator 规则、CLI 帮助和文档，交用户审核。
- 对照 `omni-superdev` 检查渐进式加载、冲突策略和完成门禁是否遗漏。
- 组织每个切片的验证报告和学习复盘。
