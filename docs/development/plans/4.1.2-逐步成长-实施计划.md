# 4.1.2 逐步成长实施计划

> 对应文章《AI研发自动化：Wiki知识库+技能包》4.1.2“逐步成长”。本计划承接 4.1.1 的 Vault 初始化结果，把静态知识库推进为可持续摄入、查询、沉淀、巡检和对外提供上下文的研发任务控制面。

> 执行时遵守根目录 `AGENTS.md` 的 mentorship-first 结对模式。行为变更使用 TDD；用户可见的 `web/**` 变更必须在真实浏览器中验收。

## 一、阶段目标

完成下面这条用户主路径：

```text
PRD / bug / 改造任务
  -> Dev Run
  -> Context Pack / Context Ask
  -> 阶段输出与待确认事项
  -> Review / Deposit
  -> Wiki / Run Record
```

同时让一次 raw 摄入从“生成一个提案”成长为可追踪的持续编译过程：

```text
raw
  -> Insight Extractor
  -> Evidence Binder
  -> Topic Router
  -> Conflict Checker
  -> Patch Writer
  -> Review Queue
  -> accepted wiki + index.md + log.md
```

## 二、进入条件

开始本阶段前，4.1.1 必须通过全部门禁：

- 两类 Vault 可以初始化。
- Schema、页面模板、`index.md`、`log.md`、`KB-META.md` 已生成。
- 初始化幂等且不覆盖已有内容。
- Vault 状态接口和空状态初始化流程可用。
- 后端、前端、构建及浏览器验收通过。

如果任一条件未满足，只能修复 4.1.1，不能用 4.1.2 代码绕过初始化缺口。

## 三、范围

### 3.1 本阶段包含

1. **2.1 Dev Run Console**：产品主入口、任务记录、阶段轨道、待确认输出。
2. **2.2 Context Ask + Deposit**：Ask 绑定 Dev Run，完整回答和选中片段进入 ingest 审阅。
3. **2.3 Wiki Health 基础**：断链、孤页、缺 frontmatter、缺来源及基础健康摘要。
4. **2.4 薄 MCP Server**：`context_pack`、`search`、`deposit`、`health_check`。
5. **2.5 异步编译与审阅队列**：多阶段编译、状态可见、接受后同步更新关联文件。

### 3.2 本阶段不包含

- 不实现完整 Skill Workbench 或 13 个正式 Skill。
- 不实现 LLM Judge、Golden Tasks 跑分、σ₀ 或评测晋级。
- 不实现自动调用外部 Agent 的 Harness。
- 不实现长时自治循环、自动发布、自动合并或自动回滚。
- 不实现团队、多用户、多 Vault 联邦。
- 不允许 MCP、Dev Run 或编译流水线直接绕过 review 写正式 wiki。

## 四、固定架构决策

### 4.1 Dev Run 是研发任务聚合根

每个 PRD、bug 或改造任务对应一条 Dev Run。Dev Run 负责记录：

- 任务类型、标题、目标和仓库上下文。
- 当前阶段和整体状态。
- Context Pack 摘要。
- 阶段输出、风险、阻塞点和待确认事项。
- 关联的 Ask、deposit、review 和 wiki 变更。
- 创建、更新、完成和取消时间。

本阶段的 Dev Run 是**可审阅状态机和记录容器**，不是自治执行器。

### 4.2 阶段轨道固定为六段

```text
context -> solution -> review -> coding -> testing -> deposit
```

本阶段允许人工记录阶段输出和推进状态；Skill 与外部 Agent 自动执行留到后续阶段。

### 4.3 事件追加优先于原地覆盖

Dev Run 的关键变化以事件记录：创建、上下文准备、阶段输出、等待确认、放行、阻塞、沉淀、完成和取消。聚合状态可以更新，但审计历史不可丢失。

### 4.4 Deposit 统一写入入口

以下内容共用同一套 deposit 契约：

- Context Ask 完整回答。
- 用户选中的回答片段。
- Dev Run 阶段输出。
- MCP 外部提交。

Deposit 只创建或复用 ingest proposal，不直接修改正式 wiki。旧 `writeback` 接口保留兼容，内部转发到 deposit 服务。

### 4.5 基础 Health 必须确定性执行

2.3 的四项结构检查使用文件和规则扫描，不依赖 LLM：

- 断链。
- 孤页。
- 缺 frontmatter。
- 缺来源。

深度语义冲突、重复标题、stale/outdated 趋势与评分演进属于 4.1.3。

### 4.6 薄 MCP 只复用稳定应用服务

MCP 不直接访问数据库表或 Vault 文件，不复制业务逻辑。四个工具调用与 Web/API 相同的应用服务，并继承 review-first、路径和大小限制。MCP 使用默认 workspace，传输仅允许 localhost/127.0.0.1。

### 4.7 编译流水线输出始终是提案

异步编译可以自动抽取、路由、检查冲突和生成 patch，但最终产物必须进入审阅队列。只有 owner 接受后，才能更新 wiki、`index.md` 和 `log.md`。

## 五、核心状态模型

### 5.1 Dev Run 状态

```text
draft -> active -> awaiting_review -> active
                  -> blocked
active -> completed
draft/active/awaiting_review/blocked -> cancelled
```

- `draft`：任务已创建但尚未开始。
- `active`：正在推进某一阶段。
- `awaiting_review`：存在必须由 owner 裁决的输出。
- `blocked`：缺输入、证据或外部条件。
- `completed`：全部阶段完成且沉淀已处理。
- `cancelled`：owner 主动终止。

### 5.2 阶段状态

```text
pending -> active -> awaiting_review -> completed
                   -> blocked
pending/active/awaiting_review/blocked -> skipped
```

阶段跳过必须记录原因，不能静默跳过。

### 5.3 编译任务状态

```text
queued -> running -> awaiting_review -> completed
                  -> failed
failed -> queued
```

重试必须幂等，同一 source 和内容版本不能生成重复待审提案。

## 六、模块与文件边界

以下是建议边界，不要求把所有逻辑塞进现有 `research.py`：

### 6.1 后端

| 边界 | 主要职责 |
|------|----------|
| `models.py` / schema upgrade | Dev Run、事件、编译任务等持久化模型 |
| `schemas.py` | API 输入输出契约 |
| 新的 run 应用服务 | 创建、读取、状态推进、阶段事件和聚合视图 |
| 现有 research/deposit 服务 | Ask、deposit、review、wiki 写回复用 |
| 新的 health 服务 | 四类确定性结构扫描与摘要 |
| 新的 compiler 服务 | 编译步骤、状态、幂等和提案产出 |
| MCP 入口 | 四个薄工具与应用服务适配 |
| `main.py` | HTTP 路由接线，不承载业务规则 |

### 6.2 前端

| 边界 | 主要职责 |
|------|----------|
| `/app` | Dev Run Console 首页 |
| `/app/ask` | Context Ask，不再承担首页职责 |
| Dev Run 组件 | 任务创建、阶段轨道、运行列表、待确认输出 |
| API helper / types | Run、deposit、health 契约 |
| ingest/wiki/raw 页面 | 复用现有能力并显示 Run 来源上下文 |

### 6.3 测试

建议新增独立测试文件，避免继续膨胀现有大型测试：

- 后端 Dev Run API/服务测试。
- Deposit 契约与兼容测试。
- Wiki Health 规则测试。
- MCP 工具契约测试。
- 编译流水线状态、失败和幂等测试。
- 前端 Dev Run 页面与 API helper 测试。
- Playwright 主流程测试。

## 九、跨切片质量要求

### 9.1 安全

- 采用单实例、默认 workspace、无登录架构。
- Werkzeug/Next.js 只监听 `127.0.0.1`。
- MCP 传输仅允许 `localhost` / `127.0.0.1`，启用 DNS rebinding 防护和 Host header 校验。
- MCP 和 Web 不返回密钥、模型凭据、内部堆栈或不必要绝对路径。
- Deposit、事件和编译输入设置长度与类型限制。
- 批量审阅必须逐项校验，不能部分越权。
- API 通过 Next.js 同源 `/api/**` 代理，避免 CORS。

### 9.2 可观测性

每个关键动作至少记录：

- `run_id`
- `workspace_id`
- `event_type` 或 `compile_step`
- 状态和耗时
- 关联 source/ask/review 标识
- 错误类型，不记录秘密内容

### 9.3 幂等性

- Dev Run 创建支持显式幂等键或稳定去重策略。
- Deposit 对同一来源和内容去重。
- 编译任务按 source 内容版本去重。
- 接受/拒绝重复请求返回稳定结果，不重复写文件。

### 9.4 兼容性

- 4.1.1 生成的 Vault 可直接进入本阶段。
- 旧 AskTurn 无 run 关联时仍可读取。
- 旧 writeback 调用继续工作。
- 已有 raw/ingest/wiki/ask 测试不能回归。

### 后端

```powershell
cd server
python -m pytest
```

必须覆盖：

- Run 状态机和 workspace 隔离。
- Deposit 来源、幂等和 review-first。
- 四类 Health 扫描。
- MCP 契约与写入边界。
- 编译状态、失败、重试和接受/拒绝。

### 前端

```powershell
cd web
npm test
npm run typecheck
npm run lint
npm run build
```

必须覆盖：

- Dev Run 首页四种关键状态。
- Context Ask 和两类 Deposit。
- Health 摘要。
- 编译与审阅状态。

### 浏览器

```powershell
cd web
npm run e2e
```

`npm run e2e` 与 `npm run e2e:fullstack` 串行执行；二者不能并行。

浏览器主路径：

1. 直接进入 `/app`（Dev Run Console），无需登录。
2. 创建 PRD 类型 Dev Run。
3. 查看阶段轨道并进入 Context Ask。
4. 生成回答并沉淀完整回答或选中片段。
5. 在审阅队列接受提案。
6. 查看 wiki、index/log 和 Run 事件同步更新。
7. 查看基础 Health 摘要。
8. 推进全部阶段并完成 Dev Run。

### MCP

四个工具分别验证成功、参数错误、未授权和写入边界。至少有一条端到端路径证明 MCP deposit 会进入 Web 审阅队列。

## 十一、结对分工

默认遵守 mentorship-first：

### 用户负责

- Dev Run 状态模型和核心状态转换。
- Deposit 核心契约与 review-first 边界。
- Health 规则算法的核心实现。
- 编译状态机、幂等和失败恢复。
- 对关键架构取舍给出解释并审核最终 diff。

### Codex 负责

- 维护计划、任务清单和进度。
- 提供 RED 测试设计、边界案例和文件边界建议。
- 审查用户实现、测试输出和安全风险。
- 在用户明确授权时处理非核心重复工作，并交用户审核。
- 整理每个切片的验证报告与学习复盘。

## 十二、提交顺序

每个提交必须是可验证、可回退的能力增量：

1. Dev Run 持久化与服务。
2. Dev Run Console。
3. Context Ask 绑定与统一 Deposit。
4. 完整回答与选中片段交互。
5. Wiki Health 基础扫描。
6. 薄 MCP 四工具。
7. 编译任务与步骤状态。
8. 审阅增强及 index/log 同步。
9. 集成测试、运营文档和发布检查更新。

提交继续遵循 Conventional Commits；未经用户审核不提交。
