## 第一步：最初建设（4.1.1）

### 目标

让 Inkdesk 支持从零初始化一个结构良好的知识库。

### 设计产出

- [x] [知识库初始化设计](../specs/2026-06-05-知识库初始化设计.md) — vault 结构、页面模板、Schema 文件、KB-META.md

### 实现任务

- [ ] Schema 文件模板库：vault-layout / page-template / source-citation / ingest-proposal / ask-answer / wiki-health
- [ ] 页面模板文件：concept / entity / source / index / log
- [ ] 后端 vault 初始化服务（幂等）
- [ ] KB-META.md 自动生成
- [ ] 前端空状态引导页（上传首批材料 → 预览结构 → 确认生成）

### 文章启示

- Schema 先于内容：在第一批 raw 材料进入之前，Schema 就要就位
- index.md + log.md 从初始化就生成，不做事后补救
- KB-META.md 集中管理路径和元信息，避免硬编码

## 第二步：逐步成长（4.1.2）

对应文章 4.1.2。文章在这个阶段没有展开具体过程，核心含义是 KB 在实际使用中持续生长：新文档进入 → 触发摄入 → 更新相关页面 → index 和 log 随之更新。

### 目标

让 Inkdesk 的 `raw → ingest → wiki` 闭环能够持续运行，知识库随使用自然生长。

### 关键能力

- [ ] 每次 Ask 回答提供"沉淀"入口（整条回答 + 选中片段）
- [ ] 后台异步编译流水线：Insight Extractor → Evidence Binder → Topic Router → Conflict Checker → Patch Writer
- [ ] 审阅队列：diff 预览、接受/拒绝、批量操作
- [ ] 每次摄入自动更新 index.md 和 log.md
- [ ] 摄入时自动检查并更新受影响的相关页面

### 文章启示

- 一次摄入触发多个页面更新（不只是新建一页，而是更新所有被影响的实体页/概念页/index）
- 摄入不是"建完就完"，而是"持续编译"的过程

---

## 第三步：问题 & 优化（4.1.3）

对应文章 4.1.3。文章总结了 5 类 KB 成长中的真实问题及解决方案。

### 目标

建立 Inkdesk 的 Wiki Health 检查和评测驱动改进机制。

### 关键能力

| 文章问题 | Inkdesk 对应 |
|---------|------------|
| 业务线视角缺失 | index.md 支持多维度分类视图 |
| 页面太浅不能指导决策 | 深度模板 + status 字段追踪成熟度 |
| 实体层空白 | 混合识别策略 + 分批补全 |
| 路径硬编码 | KB-META.md 集中管理（初始化就做） |
| KB 质量不可量化 | 评测系统（结构层 → 实用层 → 信号层 → 治理层） |

- [ ] Wiki Health 自动检查：断链、孤页、缺 frontmatter、缺来源、短页、重复标题
- [ ] 健康分仪表盘：问题列表 + 分数趋势
- [ ] Golden tasks 定义（首批 5-10 条）
- [ ] LLM Judge 评分
- [ ] σ₀ 噪声基线判断真假改进
- [ ] 结构层 gate：lint + schema_gate 不过阻塞

### 文章启示

- 评测不是"跑个分"，而是四层闭环：结构层保证格式正确、实用层保证能完成任务、信号层排除噪声、治理层保证可追溯
- 页面 status（stub/draft/stable/outdated）是区分"故意简略"和"应该补充"的关键

## 第四步：初版 Skill 生产（4.2.1）

对应文章 4.2.1。以 kb-tech-solution 为例展示了从 brainstorm 到初版 Skill 的过程。

### 目标

设计 Inkdesk 的首批 Skill。

### 首批 Skill

| Skill | 来源 | 说明 |
|-------|------|------|
| ingest-source | 现有能力 | 摄入新来源材料，生成审阅提案 |
| patch-wiki-page | 现有能力 | 修改已有 wiki 页面 |
| answer-from-wiki | 现有能力 | 基于 wiki 回答，带来源引用 |
| deposit-answer | 当前重点 | 将回答沉淀为 wiki 提案 |
| run-wiki-health | 当前重点 | 执行 wiki 健康检查 |
| extract-insight | 当前重点 | 从对话中提取可复用的知识片段 |

### 文章启示

- 初版 Skill 先用 brainstorm 设计 demo，不必追求完整
- 每个 Skill 需要配套文档（安全规则、异常处理、示例）
- 不要试图一次性穷举所有 Skill，从最常用的开始

## 第五步：Skill 问题优化（4.2.2）

对应文章 4.2.2。在实战中拿着真实需求反复使用 Skill，总结问题并迭代优化。

### 目标

建立 Skill 的实战反馈 → 迭代优化闭环。

### 关键做法（来自文章）

- 用真实任务驱动 Skill 优化，不凭空设计
- 每次发现问题后记录、分析、修改 Skill，再验证
- 优先解决高频痛点（如"找不到默认代码仓"、"输出太多实现代码"）

### Inkdesk 落地

- [ ] Skill Run 记录：每次运行记录输入、输出、错误、审阅状态
- [ ] Skill 问题追踪：从 Run 记录中提取高频问题
- [ ] Skill 版本对比：优化前后跑分比较

---

## 第六步：大道至简（4.2.3）

对应文章 4.2.3。核心观点：好 Skill 不是靠堆砌补丁和情况枚举，而是要抽取"经久耐用、万变不离其宗"的钢筋结构。

### 目标

定期回顾已有 Skill，提炼核心原则，精简冗余。

### 关键原则（来自文章 kb-tech-solution 的演化）

- 从 500+ 行 + 5 个配套文档 → 270 行 + 1 个模板
- 把复杂的规则表变成 Hard Gates（5 条核心规则）
- 把详细的流程描述变成 Quick Reference 表
- 只保留核心原则，给模型留发挥空间

### Inkdesk 落地

- [ ] Skill 定期回顾机制（每积累 50+ Run 记录后触发）
- [ ] Skill 行数 / 配套文档数作为复杂度指标
- [ ] 精简后的 Skill 必须通过已有评测集验证

## 当前执行顺序

```text
第一步（最初建设）── 当前阶段
  │  Schema 模板 + 页面模板 + 初始化服务 + 空状态 UI
  │
第二步（逐步成长）── 第一步完成后
  │  Ask 沉淀入口 + 异步编译流水线 + 审阅队列
  │
第三步（问题&优化）── 闭环跑通后
  │  Wiki Health + 评测基础设施 + 健康仪表盘
  │
第四步（初版 Skill）── 评测就位后
  │  首批 6 个 Skill 设计与开发
  │
第五步（Skill 优化）── 实战反馈后
  │  问题追踪 + 迭代优化 + 版本对比
  │
第六步（大道至简）── 积累足够后
      Skill 提炼重构 + 精简验证
```
