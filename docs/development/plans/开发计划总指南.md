# 开发计划总指南

> 按文章《AI研发自动化：Wiki知识库+技能包》第 4-8 节的结构，结合 Inkdesk 产品现状制定的完整开发计划。整合了阶段门禁、验收标准与执行原则。

> 方法论来源归档见 [项目映射](../../references/AI研发自动化：Wiki知识库+技能包-项目映射.md) 与 [原文](../../references/AI研发自动化：Wiki知识库+技能包.md)。

## 执行原则

- 先建设持久层，再提升自治度。
- AI 产生的知识变更默认进入审阅，除非未来有评测支撑的门禁允许自动通过。
- 已接受知识和可复用工作流必须以文件形式保存。
- Claude Code、Codex、Cursor、Obsidian、LangGraph 都是执行器或编辑器，不是长期真相来源。
- 只有 Wiki Health 与 Evaluation Run 证明流程可靠后，才提升 Harness 自治度。

## 第一步：最初建设（4.1.1）

> **状态：设计已完成，当前进入实施。** 本阶段是后续知识摄入、健康检查、Skill 和 Evaluation 的文件基础。

### 目标

让 Inkdesk 支持从零初始化一个结构良好的知识库。

### 设计产出

- [x] [知识库初始化设计](../specs/4.1.1-最初建设-知识库初始化设计.md) — vault 结构、页面模板、Schema 文件、KB-META.md

### 适配原则

- 吸收原文的 Schema-first、持续编译、交叉引用和巡检思想，不照搬 mkt-link 的固定目录、业务概念与企业工具。
- Inkdesk 已有的 `raw -> ingest -> wiki -> ask` 能力继续复用，本阶段不重复建设已有闭环。
- qmd、Obsidian 和代码静态分析是可替换工具或扩展能力，不作为通用知识库初始化的硬依赖。
- 初始化只负责建立可持续生长的文件基础；持续摄入、关联更新、健康治理分别由后续阶段承接。

| 原文任务 | Inkdesk 适配结论 | 计划归属 |

| T6 qmd 搜索 | 由 Inkdesk 现有检索底座承接，qmd 不作为硬依赖 | 已替代 |
| T7 固定核心概念 | 改为从用户材料中识别种子概念，不内置 mkt-link 业务词 | 第二步 2.5 |
| T8 首批文档摘要 | 由 ingest 提案和审阅流程承接，不直接写入正式 wiki | 已有闭环 + 第二步 2.5 |

| T12 核心类详情 | 按项目需要生成，不纳入通用初始化门禁 | 后续扩展 |
| T13 Obsidian 验证 | 保留 Markdown 兼容性验收，Obsidian 仅作可选编辑器 | 第一步 |
| T14 lint | 先做基础结构检查，再进入深度健康治理 | 第二步 2.3 + 第三步 |
| T15 query 实测 | 改为 Inkdesk 查询与知识更新闭环的集成验收 | 第二步完成后 |
| T16 运营文档 | 输出初始化、摄入、页面新增与更新的使用规范 | 第一步 |

### 实现任务

- [ ] 统一代码项目型与通用知识型的初始化范围、页面类型和元信息规范。
- [ ] 建立 Schema、页面模板、`index.md`、`log.md` 与 `KB-META.md` 的默认资产。
- [ ] 扩展现有 Vault 初始化能力，支持首次初始化、重复执行和增量补全。
- [ ] 保证已有 Vault 内容不被初始化流程静默覆盖。
- [ ] 提供空知识库引导，让用户能够预览结构、确认初始化并导入首批材料。
- [ ] 编写面向使用者和维护者的 Vault 运营规范。

### 验收标准

- 新知识库初始化后具备完整的核心目录、Schema、页面模板、索引、日志与元信息。
- 代码项目型和通用知识型均能获得与场景匹配的知识库骨架，不包含硬编码业务概念。
- 重复初始化结果保持一致；已有内容保持不变，仅补充缺失的基础资产。
- Schema 与模板可在任意 Markdown 编辑器中阅读，Obsidian 可打开但不是运行依赖。
- 空状态引导可以完成“预览结构 → 确认初始化 → 导入首批材料”的主流程。
- 自动化验证覆盖幂等性、增量保护和基础资产完整性。
- 运营文档能够说明初始化、摄入、页面新增和知识更新的基本规则。

### 文章启示

- Schema 先于内容：在第一批 raw 材料进入之前，Schema 就要就位
- index.md + log.md 从初始化就生成，不做事后补救
- KB-META.md 集中管理路径和元信息，避免硬编码

## 第二步：产品主入口与研发任务台（4.1.2 前半）

### 2.1 Dev Run Console

**目标：** 把 `/app` 从 Ask 问答框改为研发任务控制台，这是第二步的首要产品修正项。

**详细文档：** [4.1.2 逐步成长实施计划](4.1.2-逐步成长-实施计划.md)

- [ ] `/app` 改为 Dev Run Console，Ask 降级为 Context Ask
- [ ] PRD / bug / 改造任务入口
- [ ] 阶段轨道（Context Pack → 方案 → 评审 → coding → 测试 → 沉淀）
- [ ] 待确认阶段输出面板
- [ ] 可用 Skills 摘要
- [ ] 知识健康摘要（wiki health 简要信号）

**验收标准：**
- `/app` 首屏不再是 Ask 大输入框，而是研发任务入口和运行中的任务状态。
- 用户可以从 PRD、bug 或改造目标创建 Dev Run。

### 2.2 Context Ask + Deposit

**目标：** 让 Ask 退居为 Dev Run 内的上下文查询能力，回答能整条或选中片段进入 ingest 审阅。

**详细文档：** [Context Ask 沉淀流程实施计划](2026-06-04-问答沉淀主流程实施计划.md)

- [ ] 每次 Ask 回答提供"沉淀"入口（整条回答 + 选中片段）
- [ ] 新增显式 `/api/ask/{id}/deposit`，保留 `/api/ask/{id}/writeback` 兼容
- [ ] 所有 Context Ask / Dev Run 阶段沉淀都进入 ingest 审阅
- [ ] 沉淀记录关联到 Dev Run
- [ ] Context Ask 页面不再是 `/app` 首页

**验收标准：**
- 从 Context Ask 沉淀的回答进入审阅队列，而非静默写入 wiki。
- 整条回答和选中片段两类沉淀入口均可用。
- 沉淀元数据可追溯到来源 Dev Run。

### 2.3 Wiki Health（基础）

**目标：** 先补最低限度的结构问题扫描，让知识库质量可见。

- [ ] 断链检测
- [ ] 孤页检测
- [ ] 缺 frontmatter 检测
- [ ] 缺来源引用检测
- [ ] 基础健康分展示

**验收标准：** Wiki Health 至少能报告上述四类结构问题，并在 `/app` 中以简要信号展示。

### 2.4 薄 MCP Server

**目标：** 把当前稳定能力（取上下文、搜索、沉淀、健康检查）暴露为 MCP 工具，让 Claude Code 等外部 Agent 可用。

**约束：** 所有写入继续走 review-first，不直接写 accepted wiki，不绕过 ingest。

- [ ] `context_pack`：基于 task / repo / files / run_id 生成短上下文包
- [ ] `search`：搜索 wiki 和 raw
- [ ] `deposit`：提交沉淀请求，进入 ingest 审阅队列
- [ ] `health_check`：返回当前知识库健康摘要

**验收标准：**
- 外部 Agent 可通过 MCP 获取上下文包和提交沉淀。
- MCP 写入不会绕过 ingest / review。

### 2.5 异步编译流水线与审阅队列

**目标：** raw 材料进入后，后台自动完成 Insight 提取、证据绑定、Topic 路由、冲突检查和 Patch 生成，结果进入审阅队列。

- [ ] Insight Extractor → Evidence Binder → Topic Router → Conflict Checker → Patch Writer
- [ ] 审阅队列：diff 预览、接受/拒绝、批量操作
- [ ] 每次摄入自动更新 index.md 和 log.md
- [ ] 摄入时自动检查并更新受影响的相关页面

**验收标准：**
- 一次 raw 导入触发完整流水线，结果出现在审阅队列。
- 接受提案后 wiki 文件和索引文件同步更新。

### 第二步文章启示

- 一次摄入触发多个页面更新（不只是新建一页，而是更新所有被影响的实体页/概念页/index）
- 摄入不是"建完就完"，而是"持续编译"的过程
- 多平台接入不必等到最终 Harness；可以先通过一个受控 MCP 服务把当前知识底座接出去
- MCP 的写入必须继续走 review-first，不能绕过 ingest
- Ask-first 只适合作为早期技术验证；面向 AI 研发自动化时，首页必须表达 Dev Run，而 Ask 应降级为 Context Ask

## 第三步：深度健康与评测准备（4.1.3）

**详细文档：** [4.1.3 问题与优化实施计划](4.1.3-问题与优化-实施计划.md)

### 目标

在基础 Wiki Health（第二步 2.3）之上，建立深度治理能力，并为后续系统评测准备稳定的数据与版本边界。

> 本步骤以第二步 2.3 完成为前提，聚焦深度健康信号、结构门禁和评测准备，不在此阶段建设完整的 LLM Judge 与横向对比体系。

### 关键能力

| 文章问题 | Inkdesk 对应 |
|---------|------------|
| 业务线视角缺失 | index.md 支持多维度分类视图 |
| 页面太浅不能指导决策 | 深度模板 + status 字段追踪成熟度 |
| 实体层空白 | 混合识别策略 + 分批补全 |
| 路径硬编码 | KB-META.md 集中管理（初始化就做） |
| KB 质量不可量化 | 评测系统（结构层 → 实用层 → 信号层 → 治理层） |

- [ ] 深度 Wiki Health：短页、重复标题、stale/outdated 标记
- [ ] 健康分仪表盘：问题列表 + 分数趋势
- [ ] Golden Tasks 候选集与评分口径，优先来自真实使用场景
- [ ] Evaluation Run 的版本、隔离与产物边界
- [ ] 结构层 gate：lint + schema_gate 不过阻塞

### 验收标准

- Wiki Health 能在基础结构检查之上报告短页、重复标题和 stale/outdated 等深度问题。
- 健康仪表盘可展示问题列表和分数趋势。
- Golden Tasks 候选和评分口径可以文件化、版本化。
- Evaluation Run 不污染正式 wiki，Schema 不合规则不能进入后续评测。

### 文章启示

- 评测不是"跑个分"，而是四层闭环：结构层保证格式正确、实用层保证能完成任务、信号层排除噪声、治理层保证可追溯
- 页面 status（stub/draft/stable/outdated）是区分"故意简略"和"应该补充"的关键

## 第四步：Skill 开发准备（4.1.4）

**详细文档：** [4.1.4 Skill 开发准备实施计划](4.1.4-Skill开发准备-实施计划.md)

### 目标

在批量开发正式 Skill 前，固定 portable Skill package、机器契约、校验器、脚手架、路由冲突规则和 promotion 门禁。

### 关键能力

- `SKILL.md + contract.json` 双层契约。
- `references/prompts/templates/scripts/assets` 渐进式资源分层。
- Structural、semantic、safety 三层 lint。
- Skill scaffolder、Registry 读取边界和 routing graph。
- Hard Gates、writePolicy、能力 allowlist 和 promotion report。
- producer、reviewer、diagnostic、router 四种 Skill 类型及不同自由度。

### 本阶段不做

- 不开发首批 13 个正式 Skill。
- 不建设完整 Skill Workbench 或 Runtime。
- 不允许 Skill 直接写 accepted wiki。
- 不把本机 `omni-superdev` 作为运行时依赖。

### 验收标准

- 合法 package 可被初始化、发现和校验。
- 非法路径、越权写入、坏引用、链路环和职责冲突可稳定报告。
- Skill 只有在 lint、behavioral fixture、schema/evaluation gate 和人工审核通过后才能 active。
- 4.2.1 可直接使用脚手架生产正式 Skill，不再临时决定格式和门禁。

### `omni-superdev` 参考原则

- 单一宽入口负责路由，领域 Skill 保持窄职责。
- 流程、纪律和领域能力分层，避免多个控制器竞争。
- SKILL.md 保持精炼，详细规则与确定性脚本按需加载。
- 完成声明必须绑定验证证据。

## 第五步：初版 Skill 生产（4.2.1）

### 目标

设计 Inkdesk 的首批 Skill，每个 Skill 遵循统一的设计模式。

### 从文章第 5 节提取的 Skill 设计模式

文章 8 个 Skill 虽然领域不同，但共享完全相同的结构骨架：

```text
每个 Skill =
  要素 & 边界（需要什么输入，做什么不做什么）
  + Hard Gates（不满足条件就停止）
  + 步骤流程（理解 → 调研 → 执行 → 输出 → 下游衔接）
  + 并发子 Agent（知识库匹配 + 代码分析 + 安全调查）
  + 下游衔接（明确告知下一步是什么 Skill）
```

这个模式直接适用于 Inkdesk 的 Skill 设计。Inkdesk 不需要照搬文章的 Java 研发技能，但必须采用相同的结构约定。

### Inkdesk 首批 Skill（知识管理领域）

| Skill | 要素 | Hard Gates | 说明 |
|-------|------|-----------|------|
| ingest-source | raw 材料 + KB | raw 材料非空才能继续 | 摄入新来源，生成审阅提案 |
| answer-from-wiki | 问题 + KB | KB 已初始化才能继续 | 基于 wiki 回答，带来源引用 |
| deposit-answer | 回答内容 + KB | 回答非空才能继续 | 将回答沉淀为 wiki 提案 |
| patch-wiki-page | 目标页面 + 变更内容 + KB | 目标页面存在才能继续 | 修改已有 wiki 页面 |
| run-wiki-health | KB | KB 已初始化才能继续 | 执行 wiki 健康检查 |
| extract-insight | 对话内容 + KB | 对话非空才能继续 | 从对话中提取可复用知识片段 |

### Inkdesk 首批 Skill（研发自动化领域）

| Skill | 要素 | Hard Gates | 说明 |
|-------|------|-----------|------|
| tech-solution | PRD + KB + 代码仓上下文 | PRD 和 KB 不可缺失 | 生成可评审技术方案 |
| tech-review | 技术方案 + PRD + 相关上下文 | 技术方案不可缺失 | 检查方案边界、风险、依赖、测试范围 |
| coding | PRD + 技术方案 + 代码仓 | 方案未确认前不能进入 | 执行受控代码实现 |
| test-prep | PRD + 技术方案 + 变更范围 | 变更范围不可缺失 | 生成测试范围、测试数据和验收点 |
| test-fix | 测试失败日志 + 代码上下文 | 必须有真实失败信号 | 修复自动化测试或实现缺陷 |
| problem-solve | 报错 / 线上现象 + 近期变更 + KB | 现象描述不可为空 | 做问题排查和复盘沉淀 |
| skill-router | 用户意图 + 当前 run 状态 | 意图不可为空 | 路由到正确研发 Skill |

### 文章第 5 节对 Inkdesk 的额外启示

**5.1-5.5 的 Skill 链模式：**

```text
tech-solution → tech-review → just-coding → test-pre → test-fix
```

每个 Skill 的输出是下一个的输入，下游衔接明确。Inkdesk 需要两条 Skill 链。

知识管理链：

```text
ingest-source → patch-wiki-page
answer-from-wiki → deposit-answer → patch-wiki-page
run-wiki-health → patch-wiki-page（修复健康问题）
```

研发自动化链：

```text
PRD intake
  → tech-solution
  → tech-review
  → coding
  → test-prep
  → test-fix
  → problem-solve / deposit
```

**5.6 全能管家的路由模式：**

文章把 kb-just-ask 设计为"所有技能的总管家，直接提问，可协助路由到其他 skill"。Inkdesk 也需要一个路由 Skill，让用户不需要记住每个 Skill 的名字，只需描述意图，由路由 Skill 分发。

**5.7-5.8 的答疑和排查模式：**

专业答疑和问题排查是最高频的使用场景。Inkdesk 的 answer-from-wiki 需要吸收这个模式——不只是回答问题，而是给出有来源引用、有证据链、有置信度标注的回答。

### 验收标准

- skill 既能作为文件检查，也能在 UI 中看到。
- 用户不读应用代码也能理解一个 skill 的用途。
- skill 不能直接改写 wiki；输出必须成为 proposal 或 run record。

### 文章启示

- 初版 Skill 先用 brainstorm 设计 demo，不必追求完整
- 每个 Skill 需要配套文档（安全规则、异常处理、示例）
- 不要试图一次性穷举所有 Skill，从最常用的开始
- 要素不全就停止——这是 Skill 质量的生命线

## 第六步：Skill 问题优化（4.2.2）

### 目标

建立 Skill 的实战反馈 → 迭代优化闭环。

### 文章踩过的坑（对应 Inkdesk）

| 文章问题 | 优化方案 | Inkdesk 启示 |
|---------|---------|------------|
| 找不到默认代码仓/知识库 | 新增"框架代码原则"章节 | Skill 启动时必须校验 KB 存在性 |
| 输出太多详细代码 | "Interaction Simplification Principle" 表 | 定义 Skill 输出格式约束 |
| 交互过多打扰用户 | "非必要询问跳过"判定矩阵 | 只在关键决策点确认 |
| 外部经验未借鉴 | 把"现有系统调研/技术调研"并入流程 | Skill 内置调研阶段 |
| 压力场景下不严谨 | 连续 5 次 Edit 修补 Guard Rails | 用评测集覆盖边界场景 |
| 硬编码路径 | 改成相对路径 + KB-META | 从初始化就避免 |

### Inkdesk 落地

- [ ] Skill Run 记录：每次运行记录输入、输出、错误、审阅状态
- [ ] Skill 问题追踪：从 Run 记录中提取高频问题
- [ ] Skill 版本对比：优化前后跑分比较

## 第七步：大道至简（4.2.3）

### 核心洞察

文章 kb-tech-solution 的演化是整篇文章最有价值的方法论：

```text
500+ 行 + 5 个配套文档 → 270 行 + 1 个模板
```

精简方法：
- 把复杂的规则表变成 Hard Gates（5 条核心规则）
- 把详细的流程描述变成 Quick Reference 表
- 把交互规则简化成"减少非必要沟通，只在关键决策点确认"
- 只保留核心原则和"目的"描述，给模型留发挥空间

### 来自文章 6.3.3 的关键数据

精简后的新 Skill vs 旧 Skill 的评测结果：

```text
技术方案：新 Skill +8.7（更好的架构设计、风险预案）
技术评审：新 Skill -0.7（模板化检查在覆盖面上更稳定）
综合平均：新 Skill +4.0
```

启示：**精简对执行型 Skill（写方案、写代码）有显著提升，但对检查型 Skill（评审）需要保留更多结构化清单。** Inkdesk 应该按 Skill 类型分别对待。

### Inkdesk 落地

- [ ] Skill 定期回顾机制（每积累 50+ Run 记录后触发）
- [ ] 按 Skill 类型制定精简策略：执行型 Skill 偏原则驱动，检查型 Skill 保留结构化清单
- [ ] 精简后必须通过已有评测集验证

## 第八步：系统评测（文章第 6 节）

### 目标

建立 Inkdesk 的三层评测体系。

### 评测架构（来自文章 6.1）

```text
评测类型              评测对象             Inkdesk 对应
─────────────────────────────────────────────────────
6.1 KB 性能评测       KB 质量              Wiki Health + Golden Tasks
6.2 横向对比评测      KB+Skills vs 裸 AI   Inkdesk 完整方案 vs 普通 RAG
6.3 迭代评测          Skill v1 vs v2       Skill 版本对比
```

### 评测设计细节

**评分维度（来自文章 6.1.1）：**

| 维度 | 权重 | 评分方 | Inkdesk 对应 |
|------|------|--------|------------|
| Task 可用性 | 50% | LLM Judge | Skill 执行结果评分的核心维度 |
| Source→Wiki 编译质量 | 30% | LLM Judge | 摄入质量评估 |
| Wiki 结构健康 | 20% | 规则扫描 | Wiki Health 自动化检查 |
| Schema 合规 | gate | 规则 | 不过直接红灯 |

**评测流程（来自文章 6.1.2）：**

```text
Golden Tasks 准备（预先标注评分标准）
  → 并行执行（独立 sandbox，防止污染 KB）
    → LLM Judge 双 seed 打分（监控 Judge 稳定性）
      → σ₀ 噪声基线判断（区分真改进 vs 噪声）
        → 人工复核（双 seed 差 ≥ 1 分的任务）
```

**σ₀ 噪声基线（来自文章 6.1.2）：**

> 随机 5 条任务在 KB 不变下跑 3 轮，Judge 分的标准差 σ 再取中位数。|Δ| ≥ 3σ₀（实测 0.71）才算真改进，99.7% 置信。

**盲评机制（来自文章 6.2.5）：**

> - 生成和评分使用不同模型（Opus 4.6 生成，Opus 4.7 评分）
> - 生成和评分在不同平台（Claude Code 生成，Aone-Copilot 评分）
> - 产物统一脱敏（删除 KB 引用、Skill 标识），随机 Submission ID

### Inkdesk 落地

- [ ] 在第三步候选集基础上扩充系统评测任务，覆盖知识管理与研发自动化主流程
- [ ] LLM Judge 双 seed 评分
- [ ] σ₀ 噪声基线计算
- [ ] 评测运行沙箱隔离（不污染正式 wiki）
- [ ] 盲评机制：生成与评分用不同模型 / 不同上下文
- [ ] 评测结果仪表盘：分数趋势、版本对比、改进信号

### 验收标准

- Evaluation run 可见、可重复。
- 失败评测不能修改 accepted wiki。
- skill 或 schema 变更可以在 promotion 前通过评测比较。

## 第九步：实战优化（文章第 7 节）

### 7.1 Git 管理团队 KB

文章用 crontab 定时任务 + kb-sync 技能管理 KB 的版本和同步。Inkdesk 当前是单人使用，但 vault 天然就是 Git 仓库。后续的 git 集成方向：

- [ ] Vault 变更为自动生成 commit message
- [ ] kb-sync 风格的拉取/推送 Skill

### 7.2 MCP 服务收拢

文章踩的最大的坑：单个 Skill 连接多个 MCP 服务，光连 MCP 就要折腾 20 分钟。最终解法——**所有工具收拢到一个 MCP 服务**。

这对 Inkdesk 的启示直接且关键：**Inkdesk 的 MCP Server 应该是一个入口，对外暴露 context_pack / deposit / search / health_check，内部由统一的流水线引擎处理**。不要让用户分别连接"搜索 MCP"、"沉淀 MCP"、"健康检查 MCP"。

另外文章的多平台兼容（Claude Code / Aone-Copilot / OpenClaw）验证了 Inkdesk 的 MCP Server 方向是正确的——Skill 文件通用，执行环境可选。

### 7.3 测试数据准备

文章用 MCP 工具直连 odps / holo / oceanbase / mysql / igraph 捞真实测试数据，"不可编造测试数据，必须从真实数据库查询"。这是面向企业内部的场景。

对 Inkdesk 的启示：评测数据（Golden Tasks）应该是真实场景的，不是人工编造的。从用户的真实 Ask 记录中筛选和匿名化后纳入 Golden Tasks。

### 7.4 问题排查 Skill

文章的诊断链路：拿到报错 → 查发布记录（a1-CLI 查近一周发布）→ 判断是否发布导致 → 若非发布导致，结合 KB 和代码仓库分析。

对 Inkdesk 的启示：Wiki Health 的问题修复 Skill 也应该有类似的诊断链路——先判断问题类型（结构/内容/过期）→ 查 log.md 追溯变更 → 查相关页面 → 给出修复提案。

## 第十步：长期规划（文章第 8 节）

### 8.1 KB 维护 & 质量监管

文章强调三个重点：新文档进入需要严格把关、持续关注 lint 健康分、过期/错误知识及时清理。

Inkdesk 对应：审阅流程的 Hard Gates + Wiki Health 持续监控 + outdated 状态自动标记。

### 8.2 Skill 自动优化

文章的构想是基于"评测系统打分"与"用户使用反馈"，LLM 定期给出每个技能的优化建议，并通过迭代评测验证是否为正向优化。

Inkdesk 对应：Skill 版本对比评测（6.3 模式）→ 自动生成优化建议 → 人工确认 → 跑分验证。

### 8.3 跨多知识库发展

文章提到四个 KB 需要互联：mkt-link-kb、tmc-datacube-kb、mkt-gemini-kb、mkt-zelda-kb。

Inkdesk 对应：KB-META.md 中的 `related_kbs` 字段已经预留了跨 KB 关联的能力。但当前阶段先做好单 KB，跨 KB 是单 KB 稳定后的自然扩展。

### 8.4 全自动化（Harness）

文章的终局描述与 Inkdesk 的目标完全一致：

```text
LLM 是执行者，Harness 是导演 ——
不替 LLM 思考，但负责编排、门禁、护栏、回滚。
```

Inkdesk 当前和文章团队处在同样的位置：**已经具备每步骤的原子能力，缺的是上层的 harness 规则体系**。

这不是当前阶段的重点（当前重点是第一步：最初建设），但它验证了 Inkdesk 的终局方向是正确的。

### 验收标准

- Harness 记录每个阶段的决策。
- Owner 可以暂停、拒绝、重试或回滚一个 run。
- 高自治度必须 opt-in，并且有 evaluation history 支撑。

## 阶段门禁

每个阶段必须在上一阶段的验收标准全部通过后，才能开始。允许的并行例外仅限明确标注的范围。

```text
第一步（知识库初始化）── 当前进行中
  │  建立 Schema、模板、索引、日志、元信息和增量初始化能力
  │
  │  第一步通过后，进入第二步
  ↓
第二步 2.1（Dev Run Console）
  │  /app 改为研发任务台
  │
  ├── 2.2（Context Ask + Deposit）── 紧跟 2.1
  ├── 2.3（Wiki Health 基础）────── 与 2.2 可并行
  ├── 2.4（薄 MCP Server）───────── 与 2.2/2.3 可并行
  │    薄 MCP 必须满足：
  │      - 不直接写 accepted wiki
  │      - 不绕过 ingest / review
  │      - 不把自己当作完整 Harness
  │
  └── 2.5（异步编译流水线 + 审阅队列）── 2.2 稳定后
  │
  │  完成 2.1–2.5 后，进入第三步
  ↓
第三步（深度健康 + 评测准备）
  │  Golden Tasks 候选、结构门禁和评测隔离边界可与深度 Wiki Health 并行建设
  │  完成后进入第四步
  ↓
第四步（Skill 开发准备）
  │  固定 package、contract、validator、scaffolder、routing 和 promotion gate
  │  不在此阶段前建设完整 Skill Workbench
  ↓
第五步（首批 Skill）
  │  使用已验证脚手架生产正式 Skill，并接入 Workbench
  ↓
第六步（Skill 优化）→ 第七步（Skill 精简）
  │  积累足够 Run 记录后自然推进
  ↓
第八步（系统评测）→ 第九步（实战优化）→ 第十步（长期规划）
  │  完整评测就位后逐层叠加自治度
  │  只有具备通过评测历史的 workflow 才允许更高自治度
  ↓
Harness 终局
```

核心约束：

- **第一步必须先完成。** 新知识库在接受首批材料前应具备 Schema、模板、索引、日志和元信息。
- **第二步 2.1–2.5 不全部完成，不开始 Skill Workbench 实现。** 允许 2.4 薄 MCP 并行建设，但能力范围只限 `context_pack / search / deposit / health_check`。
- **第四步未通过，不批量生产正式 Skill。** 4.2.1 必须复用统一 package、contract、validator 和 promotion gate。
- **Evaluation 未就位前，不做自治多步骤运行。**
- **不允许静默写 wiki。** 所有正式 wiki 修改必须经过 ingest 审阅。
- **不做团队或多用户知识库**（当前是单人私有系统）。
- **不把 Obsidian 作为硬依赖。**
